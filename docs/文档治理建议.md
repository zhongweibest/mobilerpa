# MobileRPA 文档治理建议

## 1. 文档目的

本文档用于梳理当前 MobileRPA 项目中的文档结构，明确哪些文档应作为长期维护的主文档，哪些文档内容重复，哪些文档建议合并或归档。

后续新增文档时，应优先参考本文档的分类和归属，避免同一类内容散落在多份文档中反复维护。

## 2. 总体结论

当前文档主要分为 6 类：

1. 总体方案类
2. 计划排期类
3. 看板执行类
4. 验收清单类
5. 子仓库架构与规范类
6. 工具使用类

目前重复最明显的是“计划排期类”。`开发计划.md`、`快速上线计划.md`、`快速上线角色分工排期.md`、`开发阶段计划.md`、`项目看板.md` 都在描述开发阶段、开发周期、任务拆分、角色分工和执行顺序，只是粒度不同。

建议后续采用以下原则：

- 长期方案只维护在 `整体规划方案.md`
- 开发阶段和开发周期定义只维护在 `开发阶段计划.md`
- 当前执行进度、任务状态、周期状态和阶段状态只维护在 `项目看板.md`
- `acceptance-checklists/` 只维护验收入口索引，具体可执行验收步骤放到对应模块的 `docs/` 目录
- 子仓库内部只维护本仓相关架构、规范和工具说明

### 2.1 新增功能文档同步约定

后续新增功能、补充能力或调整已有功能边界时，默认先同步文档，再进入实现。除非用户明确要求跳过文档阶段，否则应遵守以下顺序：

1. 先判断该功能是否影响项目总纲、阶段计划、执行看板或模块验收方式。
2. 如果影响项目级方案、配置、协议、运行策略或数据模型，先更新 `docs/整体规划方案.md`。
3. 如果影响开发周期归属、任务拆分、验收门禁或推进顺序，先更新 `docs/开发阶段计划.md`。
4. 如果影响当前执行入口、任务状态、下一步建议或验收入口，先更新 `docs/项目看板.md`。
5. 如果功能已经实现且可以人工验收，再补对应模块自己的手工验收文档；未实现功能不预写完整验收步骤，只在计划和看板中写清验收目标。
6. 只有在用户审阅并同意文档方案后，才进入代码实现阶段，除非用户明确要求直接开发。

这条约定适用于：

- 新接口
- 新配置项
- 新消息类型
- 新运行策略
- 新增开发周期或增强周期
- 会影响验收方式的功能增强

### 2.2 协作确认硬约束

后续整理文档、修改规范、调整计划或处理命名时，如果出现以下情况，我必须先停下来，给出建议、影响和可选方案，等待用户确认后再执行：

1. 我对用户意图有疑问，或者发现需求存在多种合理解释。
2. 我认为当前要求可能不合理、可能带来隐藏风险，或者可能影响后续使用体验。
3. 修改涉及工程惯例、工具默认行为或平台识别规则，例如 `README.md`、配置文件名、入口文件名、目录结构等。
4. 修改会改变已经约定的开发阶段、开发周期、任务顺序或验收门禁。
5. 修改会造成大量文件重命名、迁移、归档或引用调整。

确认前只允许做不改变结果的检查、阅读、梳理和建议，不擅自替用户做最终决策。

## 3. 建议保留为主文档

### 3.1 `docs/README.md`

定位：项目文档入口索引。

保留原因：

- 快速说明每类文档应该看哪一份
- 避免新成员或后续开发时在多份计划文档之间反复确认

维护建议：

- 保留
- 新增、归档或调整文档职责时优先更新这里
- 不在这里展开详细方案和任务清单

### 3.2 `docs/整体规划方案.md`

定位：项目整体方案主文档。

保留原因：

- 覆盖技术选型、总体架构、通信方案、设备模型、存储、部署和风险边界
- 适合作为项目总纲

维护建议：

- 保留
- 后续架构级调整优先更新这里
- 不在这里维护每周任务状态

### 3.3 `docs/项目看板.md`

定位：项目执行主看板。

保留原因：

- 可勾选、可标记状态
- 每个任务有编号、角色和验收方式
- 最适合日常推进

维护建议：

- 保留为执行唯一入口
- 后续每完成一个功能项，就更新这里
- 不再用其他计划文档同步维护具体任务状态

### 3.4 `docs/开发阶段计划.md`

定位：开发阶段计划文档。

保留原因：

- 定义开发阶段、开发周期和任务拆分关系
- 定义任务、周期、阶段必须由用户明确验收后才能标记完成的门禁规则

维护建议：

- 保留
- 只在开发阶段、开发周期、验收门禁或任务归属发生变化时更新
- 不在这里维护实际状态，状态统一维护在 `项目看板.md`

### 3.5 `docs/acceptance-checklists/README.md`

定位：项目级验收入口索引。

保留原因：

- 避免根目录验收文档与模块验收文档重复维护
- 快速说明当前功能应该打开哪个模块验收文档
- 明确未实现功能不使用预写草稿作为验收依据

维护建议：

- 保留 `README.md` 作为索引
- 不在该目录继续新增具体功能验收清单
- 当前可执行验收文档必须放到对应模块内，例如 `repos/mobilerpa-center/docs/server/中心服务手工验收.md`
- 未实现功能只在看板中维护验收目标，等功能实现后再生成模块内验收文档
- 早期预写验收清单已归档到 `docs/archive/acceptance-checklists-drafts/`

## 4. 建议合并或降级为参考资料

### 4.1 `docs/archive/开发计划.md`

当前问题：

- 与 `开发阶段计划.md`、`快速上线计划.md`、`项目看板.md` 都有阶段和开发顺序重叠
- 现在更像早期规划草案

建议：

- 不再作为日常维护文档
- 将核心“串行/并行关系”吸收到 `开发阶段计划.md`，将状态维护吸收到 `项目看板.md`
- 后续可归档到 `docs/archive/`

### 4.2 `docs/快速上线计划.md`

当前问题：

- 与 `项目看板.md` 的第 1 到第 4 阶段内容高度重叠
- 与 `快速上线角色分工排期.md` 也重复描述 4 周上线节奏

建议：

- 保留为“快速上线策略说明”
- 不再维护任务级清单
- 任务级内容以 `项目看板.md` 为准
- 可在文档开头增加说明：“执行状态以看板为准”

### 4.3 `docs/archive/快速上线角色分工排期.md`

当前问题：

- 与 `项目看板.md` 的角色任务划分重复
- 未来如果两个文档同时维护，容易出现状态不一致

建议：

- 将角色分工原则并入 `项目看板.md`
- 该文档后续归档
- 不再单独维护每周角色任务

### 4.4 `docs/工作区规划.md`

当前问题：

- 与 `整体规划方案.md` 中的仓库规划和文档语言约定有重叠

建议：

- 保留为轻量工作区入口说明
- 不再扩展为详细规划文档
- 详细仓库策略以 `整体规划方案.md` 为准

## 7. 已执行整理顺序

整理顺序如下：

1. 建立 `docs/README.md`，作为项目文档入口索引。
2. 先确认 `项目看板.md` 作为执行状态唯一入口。
3. 在 `快速上线计划.md`、`快速上线角色分工排期.md`、`开发计划.md` 顶部增加说明，标记执行状态以看板为准。
4. 将 `快速上线角色分工排期.md` 的角色分工和协作原则补充到 `开发阶段计划.md`。
5. 将旧位置的前端快速上线清单内容合并到 `docs/web/前端结构说明.md`，并归档到 `docs/web/archive/前端快速上线清单.md`。
6. 将旧位置的 Agent 快速上线检查项合并到 `手机端运行时最小契约.md`，并归档到 `docs/archive/手机端四周快速上线实施清单.md`。
7. 创建归档目录，把不再维护的计划草案移入归档。
8. 将 `docs/acceptance-checklists/` 收敛为入口索引，把未实现功能的预写验收清单归档到 `docs/archive/acceptance-checklists-drafts/`。

## 8. 当前维护建议

不建议删除归档文档。

后续维护方式：

1. `项目看板.md` 继续作为后续开发唯一执行入口。
2. 归档文档只用于追溯历史讨论，不再维护状态和任务清单。
3. 如果归档文档中发现遗漏细节，先合并到对应主文档，再更新本治理文档。
4. 具体功能验收文档只在功能实现后生成，并放到对应模块自己的 `docs/` 目录。

这样既能减少后续维护混乱，也不会因为过早清理文档导致历史决策丢失。
