# MobileRPA 整体规划方案

## 1. 文档目的

本文档用于统一 MobileRPA 项目的整体建设方案，覆盖系统目标、仓库规划、技术选型、架构分层、通信协议、数据存储、开发调试、测试部署和基础设施约束。

本文档的定位不是单点设计说明，而是各子系统开发的总纲。新增的数据库设计、接口协议、模块设计、测试方案，都应与本文档保持一致。

## 2. 项目目标

本项目要建设的是一套可落地的真机自动化执行平台，而不是单机脚本集合。整体目标聚焦以下能力：

1. 批量真机统一接入、注册、绑定和管理
2. 中心服务与手机端建立实时双向通信
3. 业务脚本支持版本化、分发、切换和回滚
4. 每台设备的执行进度、状态和错误可独立追踪
5. 执行日志、截图、dump 和结果能够集中归档与查看
6. 形成从开发、调试、联调、灰度到生产发布的闭环

## 3. 建设思路

整个体系按两条线并行建设：

### 3.1 平台线

平台线负责形成生产运行能力，主要包含：

- 设备管理
- 设备发现
- 计划任务调度
- 脚本管理
- 工作流定义
- 手机侧 Agent 管理与调度
- 文件与日志归档

### 3.2 调试线

调试线负责形成开发验证能力，主要包含：

- 真机快速调试
- 脚本调试入口
- ADB 辅助工具
- 多设备批量推送辅助
- 局域网联调
- 灰度验证
- 问题回放

### 3.3 两条线的关系

两条线共享同一套业务脚本核心代码，但运行入口不同：

- 调试线强调快速验证、快速迭代、快速定位问题
- 平台线强调稳定调度、状态可追踪、结果可归档

这意味着业务脚本不能直接耦合平台通信逻辑，平台适配需要由 Agent 层承担。

## 4. 技术选型

### 4.1 中心端

- 后端语言：Go
- 通信方式：HTTP + WebSocket
- 数据库：SQLite
- 缓存：Go 进程内内存
- 文件存储：Linux 本地磁盘
- 部署方式：单机 Linux

### 4.2 管理后台

- 前端框架：Vue
- 形态：前后端分离
- 作为中心端的 Web 管理入口，与中心服务后端同仓维护

### 4.3 手机端

- 运行载体：AutoJs6
- 结构：Agent 与业务脚本解耦，分别独立工程化维护
- 调试工具： Go CLI mobilerpa-toolkit

### 4.4 选型原则

- 优先选择易落地、易维护的方案
- 优先保证单机可运行、可调试、可部署
- 为迁移 PostgreSQL、Redis、多节点部署预留抽象，但不提前引入复杂基础设施

## 5. 仓库与工作区规划

### 5.1 工作区结构

工作区采用“一个根文档仓 + 多个独立业务仓”的组织方式。根目录用于维护总纲、验收入口、归档说明和工作区级配置模板；`repos/` 下各子仓库分别维护中心端、手机端 Agent、业务脚本和工具链。

正式主线仓库如下：

1. `repos/mobilerpa-center`
2. `repos/mobilerpa-agent`
3. `repos/mobilerpa-script`
4. `repos/mobilerpa-toolkit`

根仓库默认不直接承载子仓库业务代码，子仓库由各自 Git 历史独立管理，避免嵌套仓库状态互相污染。

### 5.2 mobilerpa-center

该仓库包含：

- Go 中心服务
- Vue 管理后台
- API 协议文档
- WebSocket 协议文档
- 数据表设计
- 部署与运维文档

中心服务后端与管理后台前端采用同仓维护，原因如下：

- 前后端联动频繁
- 设备模型、计划任务模型、脚本版本模型由同一套中心端协议驱动
- 协议变更需要前后端一起迭代
- 管理后台是中心端的 Web 管理入口，不是独立于中心服务之外的另一套业务中心

### 5.3 mobilerpa-agent

该仓库包含：

- AutoJs Agent 运行时
- Agent 的 TypeScript 工程、构建与发布产物
- Agent 真机调试与验收文档
- JS 回滚基线

单独拆仓的原因：

- Agent 和脚本生命周期与中心服务不同
- 手机端调试链路要独立于平台
- Agent 运行时能力需要独立维护

Agent 仓库按双目录隔离：

- `js-agent/`：JS 回滚基线
- `ts-agent/`：正式维护主线

正式发布目录以 `ts-agent/release/` 为准。

### 5.4 mobilerpa-script

该仓库包含：

- 各业务脚本的 TypeScript 源码工程
- 共享脚本运行时 helper
- 统一发布工具
- 各脚本版本的发布产物目录

脚本仓库统一采用“TypeScript 开发 + JavaScript 发布”的工程模式：

- `packages/<script-package>/`：源码工程
- `publish/<script_name>/<version>/`：发布产物目录

手机端和 Agent 始终只运行构建后的 JavaScript，不直接运行 TypeScript 源码。

### 5.5 mobilerpa-toolkit

该仓库包含：

- ADB 设备发现与配对辅助
- Agent 推送与启动命令封装
- 中心服务网页下发 Agent 时复用的命令行能力

之所以独立拆仓，是为了让“设备运维工具链”和“中心服务主业务逻辑”解耦维护。

### 5.6 前端继续与中心端同仓维护

中心服务后端与管理后台前端同仓维护，不单独拆出第三个前端仓库。原因如下：

- 管理后台围绕中心服务的设备、计划任务、脚本版本和工作流定义模型展开
- 前后端共享同一套接口协议、发布节奏和验收口径
- 单独拆仓会增加协议同步、联调和发布协调成本

### 5.7 Git 版本管理边界

工作区采用“四个子仓库 + 一个根文档仓”的版本管理边界：

1. 根仓库：维护 `docs/`、根目录 `README.md`、根目录 `.gitignore` 和安全的配置模板。
2. `repos/mobilerpa-center`：维护 Go 中心服务、Vue 管理后台、中心端模块文档和验收文档。
3. `repos/mobilerpa-agent`：维护手机端 Agent、运行架构、运行契约和验收文档。
4. `repos/mobilerpa-script`：维护业务脚本 TypeScript 源码工程、共享运行时能力和发布产物目录。
5. `repos/mobilerpa-toolkit`：维护设备发现、Agent 推送、启动与停止等跨平台工具。

根仓库默认忽略本机运行产物、历史源材料、真实配置和子仓库目录。典型忽略范围包括：

```text
.tmp/
projects/
scripts/
config.ini
choose_device_push.bat

.vscode/
!.vscode/*.example.json

repos/mobilerpa-center/
repos/mobilerpa-agent/
repos/mobilerpa-script/
repos/mobilerpa-toolkit/
```

### 5.8 Legacy 代码处理原则

`projects/`、`scripts/` 等历史目录只作为 legacy 源材料保留，不作为正式主线代码目录。正式 Agent 维护在 `repos/mobilerpa-agent`，正式业务脚本维护在 `repos/mobilerpa-script`。

新功能进入对应正式仓库，不再向 legacy 目录叠加。需要复用历史逻辑时，应迁移到正式仓库的工程结构、构建流程和验收文档中。

### 5.9 文档与注释语言

项目文档默认使用中文编写。新增的架构、开发、协议、部署、运维、验收文档，以及必要的代码注释，默认也使用中文。

涉及工程惯例、工具默认识别或平台展示的命名可以保留英文；正文说明以中文为主。

## 6. 系统总体架构

系统拆成 4 个核心部分：

1. 中心端
2. 手机 Agent
3. 业务脚本仓库
4. 调试与测试体系

这里的“中心端”由 Go 中心服务和 Vue 管理后台共同组成，两者一起构成平台管理与调度能力；管理后台不是独立于中心服务之外的另一套业务中心，而是中心端的 Web 管理入口。

### 6.1 中心端职责

- 设备注册与绑定
- 设备发现与 Agent 下发
- WebSocket 长连接管理
- 计划任务调度与运行控制
- 脚本版本与清单管理
- 工作流定义维护
- 手机侧 Agent 管理与调度
- 日志与文件接收

### 6.2 管理后台职责

管理后台是中心端的 Web 管理入口，负责把中心端能力以可视化方式暴露给人操作，不单独承担独立于中心服务之外的业务职责。

- 设备管理
- 设备发现
- 计划任务管理
- 脚本版本管理
- 工作流定义管理
- Agent 下发与设备运维动作入口
- 待绑定设备管理

### 6.3 手机 Agent 职责

- 首次注册
- 基于 `ANDROID_ID` 生成稳定 `agent_uuid`
- 保存本地标识与运行时配置
- 建立 WebSocket 长连接
- 保持心跳
- 在连接异常或中心短暂不可用时按退避策略自动重连
- 接收任务
- 检查脚本版本
- 下载完整脚本版本目录
- 启动脚本
- 上报进度、日志、结果、截图和错误

### 6.4 业务脚本职责

业务脚本只负责自动化执行逻辑，不负责平台通信。职责包括：

- UI 查询
- 页面识别
- 点击与输入
- OCR
- 弹窗处理
- 状态机推进

## 7. 中心与手机通信方案

通信采用双通道设计：

### 7.1 HTTP

负责：

- 注册
- 鉴权
- 脚本清单查询
- 脚本下载
- Agent 版本清单查询
- Agent 包下载
- 文件上传
- 管理后台调用的管理接口

定义一下接口：

- `POST /api/v1/device/register`
- `GET /api/v1/script/manifest?script_name=<name>&script_version=<version>`
- `GET /api/v1/script/download?script_name=<name>&script_version=<version>&relative_path=<path>`
- `GET /api/v1/scripts`
- `GET /api/v1/scripts/{script_name}/versions/{script_version}`
- `POST /api/v1/scripts/upload`
- `POST /api/v1/scripts/deploy`
- `POST /api/v1/scripts/deploy-all`
- `GET /api/v1/script-names`

脚本分发对 Agent 侧已经按“元数据查询 + 文件下载”两段式实现：

1. `GET /api/v1/script/manifest?script_name=<name>&script_version=<version>`
   返回脚本版本元数据，不直接返回文件内容。
2. `GET /api/v1/script/download?script_name=<name>&script_version=<version>&relative_path=<path>`
   返回脚本版本目录内指定文件内容。

Agent 先请求 `manifest`，再按 `files` 清单逐个调用 `download`，同步完整脚本版本目录到手机端本地缓存。

脚本分发采用“脚本上传入中心脚本库 + 中心统一解压存储 + 按版本目录文件清单分发”的方案。核心约束是：

1. 中心服务不依赖 `mobilerpa-agent` 仓库路径或本地源码目录读取脚本。
2. 脚本上传入口正式支持 `zip` 上传；中心落地后统一保存为“已解压目录”。
3. `manifest` 返回 `entry_file` 和完整 `files` 清单。
4. `download` 支持通过 `relative_path` 下载脚本版本目录内的任意文件。
5. Agent 按 `files` 清单同步完整目录，而不是只下载 `index.js`。
6. 任务创建时只引用 `script_name` 和 `script_version`，不直接携带脚本源码。
7. 管理后台提供脚本版本列表、版本详情、上传入口和独立下发入口。
8. 同版本允许显式强制更新，但中心会切换到新的物理存储目录，不依赖先删掉旧目录才能完成更新。

中心脚本库存储结构按下面模式组织：

```text
/data/mobilerpa/scripts/<script_name>/<storage_dir>/
```

说明：

1. 逻辑版本以 `script_name + script_version` 识别。
2. 中心内部物理存储目录使用唯一化目录名。
3. 路径以数据库记录的 `file_path` 为准，而不是由 `<script_version>` 目录名硬编码推断。

上传处理规则如下：

1. 如果上传的是 `zip`，中心在校验通过后解压到目标版本目录。
2. 中心落地完成后统一生成或校验 `manifest.json`。
3. 版本根目录必须能找到 `index.js` 作为入口文件。
4. 同一个 `script_name + script_version` 默认视为不可覆盖版本；如需替换，必须走显式重传确认（`force`）流程。
5. 如果解压后的版本根目录下找不到 `index.js`，上传必须失败，并明确提示“不能在 zip 或目录下找到 index.js”。

`manifest` 接口响应结构至少包含以下字段：

```json
{
  "status": "ok",
  "data": {
    "script_name": "shoppe_sync",
    "script_version": "v0.1.2",
    "entry_file": "index.js",
    "checksum_sha256": "abc123",
    "download_url": "/api/v1/script/download?script_name=shoppe_sync&script_version=v0.1.2&relative_path=index.js",
    "source_type": "zip",
    "storage_type": "directory",
    "files": [
      {
        "relative_path": "index.js",
        "checksum_sha256": "abc123"
      },
      {
        "relative_path": "manifest.json",
        "checksum_sha256": "def456"
      }
    ]
  }
}
```

需要单独说明的是：

1. 管理后台脚本库维护接口和 Agent 脚本拉取接口是两套接口。
2. `/api/v1/scripts*` 与 `/api/v1/script-names` 面向管理后台和人工维护。
3. `/api/v1/script/manifest` 与 `/api/v1/script/download` 面向 Agent 运行时拉取。

可继续优化的方向包括：

1. 补充更明确的 `Content-Type`、`ETag`、`Content-Length` 等下载响应头约定。
2. 明确 `manifest` 中入口文件校验值与每个文件校验值的职责边界。
3. 视需要补充目录上传或整包下载能力，但不改变 Agent 已依赖的两段式拉取协议。

`download` 接口行为如下：

1. 请求参数使用 `script_name`、`script_version` 和 `relative_path`。
2. 成功时返回指定文件内容。
3. `relative_path` 用于下载版本目录内具体文件；如果未传，默认返回 `index.js`。
4. 响应头包含 `Content-Type`、`X-Script-Name`、`X-Script-Version`、`X-Script-Checksum-SHA256`、`X-Script-Relative-Path`。
5. 如果脚本或版本不存在，返回明确错误。

可补充 `Content-Length`、`ETag` 等更标准的缓存与校验响应头，并继续把兼容性判断优先收口在 `manifest` 阶段。

### 7.2 WebSocket

负责：

- 建立设备长连接
- 心跳保活
- 断线自动重连
- 心跳超时离线判定
- 任务下发
- 实时进度回传
- 日志流上报
- 任务结果回传
- 配置刷新和取消指令
- 后台订阅消息推送
- 批量 Agent 下发过程中的在线状态确认与结果回执

设备侧正式消息类型至少包括：

- `hello`
- `heartbeat`
- `assign_task`
- `sync_script`
- `script_sync_ack`
- `script_sync_result`
- `start_workflow_session`
- `stop_workflow_session`
- `workflow_session_ack`
- `workflow_session_event`
- `workflow_session_result`
- `script_execution_ack`
- `script_execution_event`
- `script_execution_result`

平台侧后台订阅消息与设备侧协议分层维护，分别对应“设备侧实时协议”和“后台订阅协议”，但消息结构和事件命名风格保持一致。

### 7.3 Agent 侧重连策略

为避免中心服务重启、网络抖动或短时连接重置后需要人工重启 Agent，手机端 WebSocket 需要具备自动重连能力。

Agent 侧按以下策略实现：

- 第 1 次重连等待 `3s`
- 第 2 次重连等待 `6s`
- 第 3 次重连等待 `12s`
- 第 4 次重连等待 `24s`
- 第 5 次重连等待 `48s`
- 第 6 次及以后等待 `60s`

补充约束：

1. 达到 `60s` 后，所有重连都维持 `60s` 间隔，不再继续增大。
2. 总重试次数不限，持续尝试直到中心服务恢复或 Agent 被人工停止。
3. 每次重连成功后，必须重新发送 `hello`，在收到 `hello ack` 后才恢复心跳。
4. 任意时刻只允许存在一个 WebSocket 连接实例、一个心跳调度器和一个重连调度器。
5. 主动关闭 Agent 时不进入自动重连。

### 7.4 统一消息结构

消息统一使用 JSON，最少包含以下字段：

- `type`
- `request_id`
- `device_id`
- `timestamp`
- `payload`

### 7.5 后台实时订阅约束

最终系统需要覆盖以下四类实时消息，并统一采用订阅机制：

- 设备消息：设备上线、离线、绑定变化、设备任务变化
- 任务消息：任务创建、分配、开始、进度、完成、失败、取消
- 日志消息：任务运行日志、关键步骤日志、人工排障关注日志
- 异常消息：设备异常、任务异常、脚本异常、上传异常、系统异常

同时，后台还需要建设一套“消息列表”能力，用来统一查看最近发生的重要事件。该消息列表本质上不是单独一套协议，而是对上述实时消息做统一投递、筛选和展示。

这里必须先约束几个原则：

1. 首屏查询与历史回看走 HTTP，全量读取；实时更新走 WebSocket 增量推送。
2. 不让前端页面分别维护各自独立的轮询体系，统一走“HTTP 全量 + WebSocket 增量”模式。
3. 后台订阅协议需要支持按主题订阅，而不是只有一个无差别广播流。
4. 后台消息列表只消费“已标准化的事件”，不直接消费原始设备侧日志流。
5. 设备侧与后台侧都可以使用 WebSocket，但后台订阅连接应视为独立角色，避免把后台页面直接伪装成设备连接。

后台订阅主题至少包括：

- `devices`
- `plan_runs`
- `logs`
- `alerts`
- `feed`

其中：

- `devices` 用于设备状态变化推送
- `plan_runs` 用于计划任务实例状态和进度变化推送
- `logs` 用于需要实时显示的任务日志摘要
- `alerts` 用于异常、告警和需人工关注事件
- `feed` 用于消息列表统一消费的重要事件流

### 7.6 后台实时消息统一事件模型

后台订阅消息统一采用“事件信封”结构，避免不同页面各自定义不同格式。

至少包含以下字段：

- `event_id`：事件唯一标识
- `topic`：所属主题，例如 `devices`、`plan_runs`、`logs`、`alerts`、`feed`
- `event_type`：事件类型，例如 `device_online`、`plan_run_failed`
- `entity_type`：事件主体类型，例如 `device`、`plan_run`
- `entity_id`：事件主体标识，例如 `device_id`、`plan_run_id`
- `device_id`：如适用，保留设备维度
- `plan_run_id`：如适用，保留计划任务实例维度
- `timestamp`：事件发生时间
- `summary`：用于消息列表和异常中心的简要说明
- `payload`：结构化详细数据

约束如下：

1. 页面实时刷新只依赖结构化字段，不依赖解析自由文本。
2. 同一类事件要能同时服务于“页面局部更新”和“消息列表入流”。
3. `summary` 字段必须可直接展示给人工值守人员。
4. `payload` 用于详情页、排障面板或跳转，不要求在消息列表中全部展开。

### 7.7 后台实时能力要求

后台实时能力采用“HTTP 全量 + WebSocket 增量”的统一模式，并满足以下要求：

1. 设备页、计划任务页、实例页和消息列表都不各自维护独立轮询体系。
2. 设备状态变化、任务状态与结果、日志摘要、异常与消息流都通过标准化事件推送。
3. 订阅必须支持按主题过滤，不做无差别全量广播。
4. 订阅连接需要支持重连、消息补偿、消息去重和消息保留策略。
5. 后台事件协议在页面消费前应保持统一，不以页面私有格式二次定义。

### 7.8 在线状态与心跳超时策略

设备在线状态不能只依赖 WebSocket 正常断开事件，还需要支持“连接未显式断开，但连续多个心跳周期未上报”的超时离线判定。

在线状态与心跳超时采用以下默认策略：

- 心跳间隔：`30s`
- 离线超时：`90s`
- 扫描间隔：`15s`

该策略的目标是：

- 避免把单次网络抖动误判为离线
- 在 Agent 异常卡死、网络半开连接或服务端未及时收到断开事件时，仍能自动回收在线状态
- 为多设备稳定性治理保留统一配置入口

这三个值不应写死在代码中，而应作为中心服务启动参数参与配置加载，支持：

- `.env` 默认值
- 环境变量覆盖
- 由 Linux 服务启动配置统一管理

配置项命名如下：

- `CENTER_HEARTBEAT_INTERVAL`
- `CENTER_DEVICE_OFFLINE_TIMEOUT`
- `CENTER_DEVICE_OFFLINE_SCAN_INTERVAL`

默认值为：

- `CENTER_HEARTBEAT_INTERVAL=30s`
- `CENTER_DEVICE_OFFLINE_TIMEOUT=90s`
- `CENTER_DEVICE_OFFLINE_SCAN_INTERVAL=15s`

## 8. 设备标识与位置模型

### 8.1 双标识设计

设备采用双标识：

- `device_id`：中心生成，永久不变
- `agent_uuid`：手机本地生成，辅助识别

`device_id` 为中心服务分配的字符串型主键，整体规划文档不强制限定展示格式；设备对外展示格式以后端实现和数据库生成策略为准。

### 8.2 device_id 设计原则

`device_id` 不能依赖以下信息：

- IP
- Android ID
- 物理位置

原因是这些字段都可能变化，而平台设备主键必须长期稳定。

### 8.3 首次接入流程

首次接入流程按以下顺序：

1. 手机安装 Agent
2. Agent 首次启动，生成本地 `agent_uuid`
3. Agent 上报基础信息
4. 中心创建待绑定设备记录
5. 中心分配 `device_id`
6. 手机保存 `device_id`
7. 运维在后台完成绑定、分组和物理位置录入

### 8.4 物理位置字段

使用 `physical_slot` 记录设备上架位置，例如：

- `Z01-R03-S12`

含义：

- `Z01`：区域
- `R03`：机架或排
- `S12`：槽位

原则：

- `device_id` 永久不变
- `physical_slot` 可变
- 两者必须解耦

## 10. 数据存储设计

### 10.1 数据库存储原则

使用 SQLite，遵循以下原则：

- SQLite 只保存事实态、索引态、汇总态
- 大量原始日志不直接落库
- 大文件、截图、dump 走文件系统

### 10.2 核心表

主链路至少包含以下表， 各表功能如下：

1. `devices`
   保存设备基础信息、在线状态、绑定状态、环境检查结果和当前位置投影。
2. `location_nodes`
   保存区域、排、槽位的树形结构以及槽位与设备的绑定关系。
3. `script_names`
   保存脚本名称主表，用于维护脚本唯一名称。
4. `script_versions`
   保存脚本版本元数据、入口文件、存储路径和版本状态。
5. `software_packages`
   保存软件包元数据、文件路径和版本化上传结果。
6. `workflow_defs`
   保存工作流定义主表。
7. `workflow_nodes`
   保存工作流节点定义。
8. `workflow_edges`
   保存工作流节点之间的连线关系。
9. `plan_defs`
   保存计划任务定义，包括目标类型、调度类型、启用状态和重试策略。
10. `plan_definition_rows`
    保存计划任务默认绑定的分区-排集合。
11. `plan_runs`
    保存计划任务实例。
12. `plan_device_runs`
    保存计划任务实例下每台设备的独立运行记录。
13. `plan_run_events`
    保存计划任务实例及设备运行事件。
14. `system_settings`
    保存系统级配置项，例如发现配置、重试配置等。
15. `daily_reports`
    保存日报或按日聚合统计结果。

### 10.3 内存缓存

不引入 Redis，进程内维护以下短期状态：

- `device_id -> websocket session`
- `device_id -> last heartbeat`
- `message_id -> pending ack`
- `device_id -> current runtime state`

原则：

- 内存只存短期态
- SQLite 存长期事实态
- 服务重启后依赖设备自动重连恢复

其中与在线状态相关的短期态至少应支持：

- `device_id -> websocket session`
- `device_id -> last heartbeat timestamp`
- 后台扫描基于 `last heartbeat timestamp` 识别超时设备

在线状态回收策略要求：

- 正常断开连接时立即标记离线
- 未正常断开但超过 `offline timeout` 时，由后台扫描补充标记离线
- 标记离线时保留最近一次 `last_heartbeat_at`，用于人工排障

## 12. 手机端与脚本结构

### 12.1 手机端目录模型

```text
/sdcard/mobilerpa/
  agent/
    agent.js
    config.json
    runtime/
  scripts/
    shoppe/
      2026.06.08.001/
  logs/
  screenshots/
  dumps/
  temp/
```

### 12.2 业务脚本目录模型

```text
repos/mobilerpa-script/
  packages/
    <script-package>/
      src/
      package.json
      script.config.json
      build.js
  <script_name>/
    <version>/
      index.js
      index_debug.js
      manifest.json
      vX.Y.Z.zip
```

正式口径：

1. 源码以 `repos/mobilerpa-script/packages/` 为准。
2. 发布目录 `publish/<script_name>/<version>/` 视为构建产物。
3. 调试入口使用 `index_debug.js`。
4. 真正给 Agent 执行的入口使用 `index.js`。

### 12.3 脚本版本包

脚本以版本包形式交付，并由中心统一解压成目录后管理。上传来源可以是 `zip` 或目录，但中心内部不依赖原始上传形态。每个版本包至少包含：

- `index.js`
- `manifest.json`
- 业务脚本依赖的相对路径文件

其中 `manifest.json` 至少包含：

- `script_name`
- `version`
- `entry_file`
- `compatible_agent`
- `checksum`
- `description`

其中：

1. `script_name`
   必须与中心侧 `script_names.script_name`、`script_versions.script_name`、计划任务或工作流节点引用的脚本名称一致。
2. `version`
   必须与中心侧 `script_versions.version`、计划任务或工作流节点引用的脚本版本一致。
3. `entry_file`
   明确 Agent 最终执行的入口文件，例如 `index.js`。
4. `compatible_agent`
   用于判断 Agent 是否可以执行该脚本版本。

### 12.4 版本流转

脚本发布流转固定为：

- `dev`
- `beta`
- `prod`

流程为：

1. 本地调试形成 `dev`
2. 测试设备验证
3. 平台发布为 `beta`
4. 灰度验证通过后升为 `prod`

### 12.5 Agent 下载与执行脚本链路

手机端不应该在收到 `assign_task` 后直接盲目执行，而应走“检查 -> 拉取清单 -> 下载目录文件 -> 落盘 -> 校验 -> 执行”的固定链路。

流程如下：

1. 中心通过 `assign_task` 下发：
   `plan_run_id`、`plan_device_run_id`、`device_id`、`script_name`、`script_version`、`params`。
2. Agent 收到计划任务实例设备运行指令后，先检查本地是否已有 `/sdcard/mobilerpa/scripts/<script_name>/<script_version>/manifest.json`。
3. 如果本地不存在该版本，或本地 `manifest.json` 与运行指令要求不一致，则先请求中心脚本元数据接口。
4. Agent 调用脚本元数据接口，获取：
   `script_name`、`script_version`、`entry_file`、`checksum_sha256`、`download_url`、`files`。
5. Agent 根据 `manifest.files` 逐个调用下载接口，把脚本版本目录文件同步到本地。
6. Agent 校验入口文件、清单信息和必要摘要值。
7. 校验通过后，确保本地目录为：
   `/sdcard/mobilerpa/scripts/<script_name>/<script_version>/`
8. Agent 再次读取该目录下的 `manifest.json`，确认 `script_name`、`version`、`entry_file` 一致。
9. Agent 组装执行上下文：
   `plan_run_id`、`plan_device_run_id`、`device_id`、`script_name`、`script_version`、`params`、`trace_id`、`timeouts`。
10. Agent 通过 `entry_file` 指向的入口脚本加载并执行 `run(context)`。
11. 脚本执行过程中，Agent 继续负责步骤事件上报、日志上报和最终执行结果 / `workflow_session_result` 回传。
12. 工作流状态机的真源固定在 Agent 本地；中心保存的是计划任务实例与设备运行记录上的投影状态，不再承担逐节点同步推进。

这一链路的核心原则是：

1. 中心负责决定“执行哪个脚本版本”。
2. Agent 负责保证“本地拿到正确版本的脚本包并按入口执行”。
3. 业务脚本只负责自动化逻辑，不负责下载、注册、长连接和平台通信。

进一步约束：

1. `assign_task` 不直接携带完整脚本源码，只携带 `script_name`、`script_version` 和执行参数。
2. Agent 只信任中心返回的 `manifest` 元数据和校验信息，不通过任务参数拼接任意本地脚本路径。
3. 执行入口必须来自脚本包内 `manifest.json.entry_file`，并且该值要与中心 `manifest` 接口返回一致。
4. 如果本地已有同版本缓存且校验一致，Agent 应直接复用本地缓存，不重复下载。

### 12.6 脚本管理页面扩展原则

脚本管理不只包括“上传”和“下发”，还包括“查看”和“编辑”。但为了避免已发布版本被直接改写，必须遵守以下规则：

1. 网页查看脚本内容属于近中期能力，可在脚本管理页查看版本目录、文件清单和文件内容。
2. 网页编辑脚本时，编辑后默认应保存为新版本，而不是直接覆盖已发布版本。
3. 已发布版本原则上保持只读，除非用户明确执行高风险覆盖操作。
4. 如支持在线编辑，应同步支持版本对比、草稿保存和发布说明。

## 13. 任务执行模型

任务采用状态机模型，不写成大段顺序脚本。

示例步骤：

- `INIT_APP`
- `CHECK_HOME`
- `CHECK_READY`
- `CHECK_DELAY_TEXT`
- `CLICK_START`
- `WAIT_RESULT`
- `HANDLE_POPUP`
- `VERIFY_LOCKED`
- `COMPLETE`

每推进一步就上报执行步骤事件，这样平台才能准确知道：

- 执行到哪一步
- 成功还是失败
- 卡在什么地方
- 是否需要人工介入

## 14. 调试体系

调试体系必须独立于平台，但可与平台联调。

### 14.1 本地快速调试模式

- 不依赖平台
- 通过 ADB/推送工具直接在真机运行调试入口
- 快速查看日志、截图和页面状态

### 14.2 局域网联调模式

- 手机连接中心服务
- 仅对测试设备下发调试任务
- 验证版本切换、Agent 协议和上报链路

### 14.3 平台生产模式

- 正式设备池
- 统一调度
- 统一日志、结果和日报

### 14.4 调试工具形态

调试工具采用 Go CLI，例如：

```text
mobilerpa-toolkit push-center --device <adb-device> --center-base-url <url> --agent-root <path>
mobilerpa-toolkit push-manual --all --center-base-url <url> --agent-root <path>
mobilerpa-toolkit start-agent --device <adb-device>
mobilerpa-toolkit stop-agent --device <adb-device>
```

补充约束：

1. 调试工具里的“批量推送”仅针对本机通过 ADB 连接并授权的真机。
2. “面向中心服务在线设备的批量 Agent 下发”属于平台能力，应由中心服务、后台和设备侧共同闭环，不以本地 ADB 脚本直接替代。

## 15. 管理后台范围

管理后台覆盖以下页面或入口：

1. 主页看板
2. 设备列表
3. 设备发现
4. 设备绑定
5. 计划任务
6. 计划任务实例与设备运行明细
7. 脚本管理
8. 工作流编排
9. 软件管理
10. 系统配置

后台重点不是复杂权限系统，而是服务平台主链路的可视化与可操作性。

后台还需要补一类“设备运维动作”入口，用于：

- 批量下发 Agent 新版本
- 批量查看设备 Agent 版本
- 批量触发配置刷新或重连

但该能力属于平台化运维增强，不和本地 ADB 调试脚本混用。

### 15.1 消息列表定位

消息列表不是简单的“日志页复制品”，而是面向值守和排障的统一事件流入口。

消息列表优先展示：

- 设备上线/离线
- 任务分配/开始/完成/失败
- 关键异常与人工关注事件
- 日志中的关键摘要事件

消息列表不承担完整原始日志存档职责，原始日志仍以日志页或日志详情链路为准。

### 15.2 消息列表与异常事件的关系

- 消息列表强调“最近发生了什么”
- 异常事件强调“有哪些异常需要集中处理”

两者应共享同一套后台事件模型，但展示目标不同：

- 消息列表按时间流和主题筛选为主
- 异常事件按严重级别、状态、设备、任务聚合为主

- 每日备份 SQLite 和脚本目录
- 每日归档日志和截图

与设备在线状态相关的运行配置统一进入中心服务启动参数，例如：

- `CENTER_HTTP_ADDR`
- `CENTER_DB_PATH`
- `CENTER_HEARTBEAT_INTERVAL`
- `CENTER_DEVICE_OFFLINE_TIMEOUT`
- `CENTER_DEVICE_OFFLINE_SCAN_INTERVAL`

### 17.3 手机侧基础设施

重点不在 USB，而在：

- 稳定供电
- 稳定 Wi-Fi
- 散热
- 编号体系

设备编排参考：

- 每区 30 到 60 台设备
- 企业级 AP
- 统一支架或机架
- 手机编号与 `physical_slot` 对应

ADB 只保留用于：

- 初次安装 Agent
- 本地调试
- 故障排查

不作为生产主链路。

## 18. 风险与边界

方案采用轻量可落地版，需要明确边界。

### 18.1 适用规模

- 10 到 100 台：非常合适
- 几百台：可继续运行
- 1000 台：可验证业务，但需谨慎控制风险

### 18.2 主要风险

- SQLite 写入竞争
- 单机内存状态易失
- 单点故障
- 本地磁盘增长
- WebSocket 连接管理复杂度上升
- 后台实时事件模型过晚统一会导致前端多页面返工
- 消息列表如果直接消费原始日志，容易导致噪音过大和筛选困难

### 18.3 应对策略

- 任务事件批量落库
- 设备自动重连和状态补报
- 日志归档与截图清理
- SQLite 定期备份
- 在代码边界上为迁移 PostgreSQL 和 Redis 预留抽象
- 提前统一后台订阅主题和事件信封结构
- 消息列表只消费标准化事件，不直接消费未经归类的原始日志
- 将工作流编排层建立在已验收的单设备脚本执行主链路之上，避免把执行问题和编排问题混在一个周期里处理

## 19. 实施原则总结

所有设计与开发遵循以下原则：

1. 保证单机可运行、可调试、可部署
2. 优先保证主链路完整
3. 保证平台与脚本解耦
4. 新功能优先进入新仓库，不再向 legacy 目录叠加
5. 形成稳定规范并持续收口历史脚本和工具
6. 新增会影响架构、计划、看板或验收方式的功能时，同步更新项目级文档

这份文档作为项目总纲，如需调整，优先在本文档更新，再同步修改子系统文档。
