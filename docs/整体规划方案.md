# MobileRPA 整体规划方案

## 1. 文档目的
本文档用于统一 MobileRPA 项目的整体建设方案，覆盖系统目标、仓库规划、技术选型、架构分层、通信协议、数据存储、开发调试、测试部署和基础设施约束。

本文档的定位不是单点设计说明，而是后续各子系统开发的总纲。后续新增的数据库设计、接口协议、模块设计、测试方案，都应与本文档保持一致。

## 2. 项目目标
本项目要建设的是一套可落地的真机自动化执行平台，而不是单机脚本集合。第一阶段目标聚焦以下能力：

1. 批量真机统一接入、注册、绑定和管理
2. 中心服务与手机端建立实时双向通信
3. 业务脚本支持版本化、分发、切换和回滚
4. 每台设备的执行进度、状态和错误可独立追踪
5. 执行日志、截图、dump 和结果能够集中归档与查看
6. 形成从开发、调试、联调、灰度到生产发布的闭环

## 3. 建设思路
整个体系按两条线并行建设：

### 3.1 平台线
平台线负责形成生产运行能力，主要包含：

- 中心服务
- 设备管理
- 任务调度
- 脚本版本管理
- Agent 版本与批量下发管理
- 文件与日志归档
- 报表与异常中心
- 管理后台

当前阶段的优先级调整为：先把“脚本管理”能力做完整，再继续改造任务中心和工作流编排。原因是脚本管理同时服务于调试链和执行链，只有先稳定脚本版本、脚本包下发和脚本选择，任务中心才能从“手工输入脚本名和版本”升级为“关联已发布脚本版本”。

### 3.2 调试线
调试线负责形成开发验证能力，主要包含：

- 真机快速调试
- 脚本调试入口
- ADB 辅助工具
- 多设备批量推送辅助
- 局域网联调
- 灰度验证
- 问题回放

### 3.3 两条线的关系
两条线共享同一套业务脚本核心代码，但运行入口不同：

- 调试线强调快速验证、快速迭代、快速定位问题
- 平台线强调稳定调度、状态可追踪、结果可归档

这意味着业务脚本不能直接耦合平台通信逻辑，平台适配需要由 Agent 层承担。

## 4. 技术选型

### 4.1 中心端
- 后端语言：Go
- 通信方式：HTTP + WebSocket
- 数据库：SQLite
- 缓存：Go 进程内内存
- 文件存储：Linux 本地磁盘
- 部署方式：单机 Linux

### 4.2 管理后台
- 前端框架：Vue
- 形态：前后端分离
- 与中心服务同仓管理，先采用 monorepo

### 4.3 手机端
- 运行载体：AutoJs6
- 结构：Agent + 业务脚本
- 调试工具：前期保留 ADB/批处理链路，后期演进为 Go CLI

### 4.4 选型原则
- 优先选择当前阶段最容易落地、最容易维护的方案
- 优先保证单机可运行、可调试、可部署
- 为未来迁移 PostgreSQL、Redis、多节点部署预留抽象，但第一阶段不提前引入复杂基础设施

## 5. 仓库与工作区规划

### 5.1 工作区结构
当前工作区保留为一个总工作区，但内部按两个独立仓库推进：

1. `repos/mobilerpa-center`
2. `repos/mobilerpa-agent`

### 5.2 mobilerpa-center
该仓库包含：

- Go 中心服务
- Vue 管理后台
- API 协议文档
- WebSocket 协议文档
- 数据表设计
- 部署与运维文档

采用单仓的原因：

- 当前前后端联动频繁
- 设备模型、任务模型、脚本版本模型还在快速变化
- 协议变更需要前后端一起迭代

### 5.3 mobilerpa-agent
该仓库包含：

- AutoJs Agent
- 业务脚本
- 调试入口
- 真机调试工具
- 脚本版本清单

单独拆仓的原因：

- Agent 和脚本生命周期与中心服务不同
- 手机端调试链路要独立于平台
- 业务脚本版本需要独立演进

### 5.4 暂不拆第三个前端仓库
当前不建议将后台前端再拆成第三个仓库。原因如下：

- 前端团队还不是独立运作形态
- API 和协议尚未稳定
- 额外拆仓只会增加同步与发布复杂度

### 5.5 Legacy 目录处理原则
以下目录暂不删除：

- `projects/`
- `scripts/`
- `choose_device_push.bat`

它们当前作为 legacy 参考源保留，后续按模块逐步迁移到新仓库中，不在旧目录上继续叠加新功能。

## 6. 系统总体架构
系统拆成 5 个核心部分：

1. 中心服务
2. 管理后台
3. 手机 Agent
4. 业务脚本仓库
5. 调试与测试体系

### 6.1 中心服务职责
- 设备注册与绑定
- WebSocket 长连接管理
- 任务下发与状态追踪
- 脚本版本与清单管理
- 日志与文件接收
- 日报、统计、异常聚合

### 6.2 管理后台职责
- 设备管理
- 任务管理
- 脚本版本管理
- 脚本独立下发
- 脚本调试入口说明
- 异常中心
- 日报中心
- 待绑定设备管理

### 6.3 手机 Agent 职责
- 首次注册
- 保存本地标识
- 建立 WebSocket 长连接
- 保持心跳
- 在连接异常或中心短暂不可用时按退避策略自动重连
- 接收任务
- 检查脚本版本
- 下载完整脚本版本目录
- 启动脚本
- 上报进度、日志、结果、截图和错误

### 6.4 业务脚本职责
业务脚本只负责自动化执行逻辑，不负责平台通信。职责包括：

- UI 查询
- 页面识别
- 点击与输入
- OCR
- 弹窗处理
- 状态机推进

## 7. 中心与手机通信方案
通信采用双通道设计：

### 7.1 HTTP
负责：

- 注册
- 鉴权
- 脚本清单查询
- 脚本下载
- Agent 版本清单查询
- Agent 包下载
- 文件上传
- 后台管理接口

第一阶段建议保留的接口类型：

- `POST /api/v1/device/register`
- `POST /api/v1/device/upload`
- `GET /api/v1/script/manifest/...`
- `GET /api/v1/script/download/...`

脚本分发相关接口建议尽早冻结为“元数据查询”和“包下载”两段式协议：

1. `GET /api/v1/script/manifest?script_name=<name>&version=<version>`
   返回脚本元数据，不直接返回脚本正文。
2. `GET /api/v1/script/download?script_name=<name>&version=<version>`
   返回脚本包下载流，或返回一次性下载地址。

这样可以把“先确认版本信息”和“真正下载文件”拆开，便于 Agent 做缓存判断和校验。

当前阶段采用“脚本上传入中心脚本库 + 中心统一解压存储 + 版本目录文件清单分发”的方案，先满足多文件脚本包下发和真机调试需求。该方案的核心约束是：

1. 中心服务不再依赖 `mobilerpa-agent` 仓库路径或本地源码目录读取脚本。
2. 脚本上传入口支持 `zip` 或目录两种来源，但中心落地后统一保存为“已解压目录”。
3. `manifest` 返回 `entry_file` 和完整 `files` 清单。
4. `download` 支持通过 `relative_path` 下载脚本版本目录内的任意文件。
5. Agent 按 `files` 清单同步完整目录，而不是只下载 `index.js`。
6. 任务创建时只引用 `script_name` 和 `script_version`，不直接携带脚本源码。
7. 管理后台提供脚本版本列表、版本详情、上传入口和独立下发入口。

推荐的中心脚本库存储目录如下：

```text
/data/mobilerpa/scripts/<script_name>/<script_version>/
```

推荐的上传处理规则如下：

1. 如果上传的是 `zip`，中心在校验通过后解压到目标版本目录。
2. 如果上传的是目录，中心先完成目录内容接收，再整理到目标版本目录。
3. 中心落地完成后统一生成或校验 `manifest.json`。
4. 同一个 `script_name + script_version` 默认视为不可覆盖版本，后续如需替换，应走“新版本发布”或显式重传确认流程。
5. 无论上传来源是 `zip` 还是目录，脚本版本根目录下都必须能找到 `index.js`。
6. 如果 `zip` 解压后或目录根目录下找不到 `index.js`，上传必须失败，并明确提示“不能在 zip 或目录下找到 index.js”。

建议 `manifest` 接口最小响应结构如下：

```json
{
  "status": "ok",
  "data": {
    "script_name": "shoppe",
    "version": "2026.06.08.001",
    "entry_file": "main.js",
    "package_type": "zip",
    "checksum_type": "sha256",
    "checksum": "abc123",
    "download_url": "/api/v1/script/download?script_name=shoppe&version=2026.06.08.001",
    "compatible_agent": ">=0.1.0",
    "package_file_id": "file_000001",
    "updated_at": "2026-06-12T10:00:00Z"
  }
}
```

建议 `download` 接口行为如下：

1. 请求参数仍使用 `script_name` 和 `version`。
2. 成功时返回指定文件内容。
3. 请求参数增加 `relative_path`，用于下载版本目录内具体文件。
4. 响应头中建议带上 `Content-Type`、`Content-Length`、`ETag` 或等价校验信息。
5. 如果脚本不存在，返回明确错误，例如 `404 script_version_not_found`。
6. 如果版本存在但 Agent 不兼容，优先在 `manifest` 阶段暴露，不建议等下载完成后再失败。

### 7.2 WebSocket
负责：

- 建立设备长连接
- 心跳保活
- 断线自动重连
- 心跳超时离线判定
- 任务下发
- 实时进度回传
- 日志流上报
- 任务结果回传
- 配置刷新和取消指令
- 后台订阅消息推送
- 后续批量 Agent 下发过程中的在线状态确认与结果回执

建议首批消息类型：

- `hello`
- `heartbeat`
- `assign_task`
- `task_ack`
- `task_progress`
- `task_log`
- `task_result`
- `cancel_task`
- `refresh_config`

平台侧后台订阅消息不建议与设备侧协议完全混写，后续应按“设备侧实时协议”和“后台订阅协议”分层维护，但消息结构和事件命名风格应保持一致。

### 7.2A Agent 侧重连策略
为避免中心服务重启、网络抖动或短时连接重置后需要人工重启 Agent，手机端 WebSocket 需要具备自动重连能力。

第一版建议固定为以下策略：

- 第 1 次重连等待 `3s`
- 第 2 次重连等待 `6s`
- 第 3 次重连等待 `12s`
- 第 4 次重连等待 `24s`
- 第 5 次重连等待 `48s`
- 第 6 次及以后等待 `60s`

补充约束：

1. 达到 `60s` 后，后续所有重连都维持 `60s` 间隔，不再继续增大。
2. 总重试次数不限，持续尝试直到中心服务恢复或 Agent 被人工停止。
3. 每次重连成功后，必须重新发送 `hello`，在收到 `hello ack` 后才恢复心跳。
4. 任意时刻只允许存在一个 WebSocket 连接实例、一个心跳调度器和一个重连调度器。
5. 主动关闭 Agent 时不进入自动重连。

### 7.3 统一消息结构
消息统一使用 JSON，最少包含以下字段：

- `type`
- `request_id`
- `device_id`
- `timestamp`
- `payload`

### 7.4 后台实时订阅约束
最终系统需要覆盖以下四类实时消息，并统一采用订阅机制：

- 设备消息：设备上线、离线、绑定变化、当前任务变化
- 任务消息：任务创建、分配、开始、进度、完成、失败、取消
- 日志消息：任务运行日志、关键步骤日志、人工排障关注日志
- 异常消息：设备异常、任务异常、脚本异常、上传异常、系统异常

同时，后台还需要建设一套“消息列表”能力，用来统一查看最近发生的重要事件。该消息列表本质上不是单独一套协议，而是对上述实时消息做统一投递、筛选和展示。

这里必须先约束几个原则：

1. 首屏查询与历史回看走 HTTP，全量读取；实时更新走 WebSocket 增量推送。
2. 不让前端页面分别维护各自独立的轮询体系，统一走“HTTP 全量 + WebSocket 增量”模式。
3. 后台订阅协议需要支持按主题订阅，而不是只有一个无差别广播流。
4. 后台消息列表只消费“已标准化的事件”，不直接消费原始设备侧日志流。
5. 设备侧与后台侧都可以使用 WebSocket，但后台订阅连接应视为独立角色，避免把后台页面直接伪装成设备连接。

推荐的后台订阅主题至少包括：

- `devices`
- `tasks`
- `logs`
- `alerts`
- `feed`

其中：

- `devices` 用于设备状态变化推送
- `tasks` 用于任务状态和进度变化推送
- `logs` 用于需要实时显示的任务日志摘要
- `alerts` 用于异常、告警和需人工关注事件
- `feed` 用于消息列表统一消费的重要事件流

### 7.5 后台实时消息统一事件模型
后台订阅消息建议统一采用“事件信封”结构，避免不同页面各自定义不同格式。

建议至少包含以下字段：

- `event_id`：事件唯一标识
- `topic`：所属主题，例如 `devices`、`tasks`、`logs`、`alerts`、`feed`
- `event_type`：事件类型，例如 `device_online`、`task_failed`
- `entity_type`：事件主体类型，例如 `device`、`task`
- `entity_id`：事件主体标识，例如 `device_id`、`task_id`
- `device_id`：如适用，保留设备维度
- `task_id`：如适用，保留任务维度
- `timestamp`：事件发生时间
- `summary`：用于消息列表和异常中心的简要说明
- `payload`：结构化详细数据

约束如下：

1. 页面实时刷新只依赖结构化字段，不依赖解析自由文本。
2. 同一类事件要能同时服务于“页面局部更新”和“消息列表入流”。
3. `summary` 字段必须可直接展示给人工值守人员。
4. `payload` 用于详情页、排障面板或后续跳转，不要求在消息列表中全部展开。

### 7.6 后台实时能力的落地顺序
为了避免在后期把多个页面从轮询强行改造成订阅，后台实时能力需要提前规划、分阶段落地。

推荐顺序如下：

1. 第一阶段只完成设备页的 HTTP 全量展示，不强制引入实时推送。
2. 第二阶段先补后台订阅基础设施，只落设备上下线与设备状态变化事件。
3. 第三阶段在任务主链路稳定后，再补任务状态、进度和结果订阅。
4. 第四阶段在日志和异常链路稳定后，再补日志摘要、异常告警和消息列表。
5. 第五阶段在早期生产试运行阶段，再完善订阅重连、消息补偿、消息去重和消息保留策略。

这样做的目标是：

- 避免当前已验收功能返工
- 避免前端页面先写各自轮询逻辑再统一改造
- 避免在任务、日志、异常模型未稳定前过早冻结后台事件协议

### 7.7 在线状态与心跳超时策略
设备在线状态不能只依赖 WebSocket 正常断开事件，还需要支持“连接未显式断开，但连续多个心跳周期未上报”的超时离线判定。

第一阶段建议采用以下默认策略：

- 心跳间隔：`30s`
- 离线超时：`90s`
- 扫描间隔：`15s`

该策略的目标是：

- 避免把单次网络抖动误判为离线
- 在 Agent 异常卡死、网络半开连接或服务端未及时收到断开事件时，仍能自动回收在线状态
- 为后续多设备稳定性治理保留统一配置入口

这三个值不应写死在代码中，而应作为中心服务启动参数参与配置加载，支持：

- `.env` 默认值
- 环境变量覆盖
- 后续部署到 Linux 服务时由启动配置统一管理

第一阶段建议配置项命名如下：

- `CENTER_HEARTBEAT_INTERVAL`
- `CENTER_DEVICE_OFFLINE_TIMEOUT`
- `CENTER_DEVICE_OFFLINE_SCAN_INTERVAL`

默认值建议：

- `CENTER_HEARTBEAT_INTERVAL=30s`
- `CENTER_DEVICE_OFFLINE_TIMEOUT=90s`
- `CENTER_DEVICE_OFFLINE_SCAN_INTERVAL=15s`

## 8. 设备标识与位置模型

### 8.1 双标识设计
设备采用双标识：

- `device_id`：中心生成，永久不变
- `agent_uuid`：手机本地生成，辅助识别

建议 `device_id` 形式：

- `dev_000001`
- `dev_000823`

### 8.2 device_id 设计原则
`device_id` 不能依赖以下信息：

- IP
- Android ID
- ADB 序列号
- 物理位置

原因是这些字段都可能变化，而平台设备主键必须长期稳定。

### 8.3 首次接入流程
首次接入建议按以下顺序：

1. 手机安装 Agent
2. Agent 首次启动，生成本地 `agent_uuid`
3. Agent 上报基础信息
4. 中心创建待绑定设备记录
5. 中心分配 `device_id`
6. 手机保存 `device_id`
7. 运维在后台完成绑定、分组和物理位置录入

### 8.4 物理位置字段
建议使用 `physical_slot` 记录设备上架位置，例如：

- `Z01-R03-S12`

含义：

- `Z01`：区域
- `R03`：机架或排
- `S12`：槽位

原则：

- `device_id` 永久不变
- `physical_slot` 可变
- 两者必须解耦

## 9. 中心服务模块规划
`mobilerpa-center` 中心服务建议拆成以下模块：

- `api`
- `ws`
- `scheduler`
- `device`
- `task`
- `script`
- `filemgr`
- `report`
- `logger`

推荐目录边界：

- `cmd/`：启动入口
- `internal/app`：应用装配
- `internal/config`：配置
- `internal/api`：HTTP 路由和处理器
- `internal/ws`：WebSocket 生命周期与消息分发
- `internal/device`：设备注册、绑定、状态处理
- `internal/task`：任务创建、流转、结果落库
- `internal/script`：脚本版本和清单
- `internal/scheduler`：调度循环、超时、重试、取消
- `internal/filemgr`：文件路径和上传落盘
- `internal/report`：日报与汇总
- `internal/logger`：日志初始化和设备日志分流
- `pkg/protocol`：共享协议模型

## 10. 数据存储设计

### 10.1 数据库存储原则
第一阶段使用 SQLite，遵循以下原则：

- SQLite 只保存事实态、索引态、汇总态
- 大量原始日志不直接落库
- 大文件、截图、dump 走文件系统

### 10.2 核心表
第一阶段至少包含以下 6 张表：

1. `devices`
2. `tasks`
3. `task_events`
4. `script_versions`
5. `uploaded_files`
6. `daily_reports`

其中与“任务名称、任务版本、脚本文件关联”直接相关的建议如下：

1. `tasks`
   保存一次实际执行任务的业务信息和调度信息。
2. `script_versions`
   保存脚本版本元数据，不直接把脚本源码正文存入 `tasks`。
3. `uploaded_files`
   保存脚本包、截图、dump 等文件与业务记录的关联关系。

建议字段职责如下：

1. `tasks.task_id`
   任务唯一标识。
2. `tasks.device_id`
   本次任务执行设备。
3. `tasks.script_name`
   任务要执行的脚本名，例如 `shoppe`。
4. `tasks.script_version`
   任务要执行的脚本版本，例如 `2026.06.08.001`、`beta` 或 `prod`。
5. `tasks.params_json`
   本次任务参数。
6. `tasks.workflow_run_id`
   如任务来自工作流编排，则关联所属工作流运行实例。
7. `tasks.workflow_node_id`
   如任务来自工作流编排，则关联所属工作流节点。

`script_versions` 建议至少包含：

1. `script_name`
2. `version`
3. `entry_file`
4. `manifest_json`
5. `package_file_id`
6. `checksum`
7. `status`
8. `created_at`
9. `storage_type`
10. `storage_path`
11. `source_type`

字段补充说明：

1. `storage_type`
   当前建议固定为 `directory`，表示中心内部统一以已解压目录保存。
2. `storage_path`
   指向中心脚本库中的版本目录路径。
3. `source_type`
   记录该版本最初是通过 `zip` 上传还是目录上传进入中心。

这里的 `package_file_id` 指向 `uploaded_files.file_id`，用于把“原始上传包或上传任务记录”和“脚本版本记录”关联起来；实际下发时优先使用中心内部已解压目录，而不是直接把原始上传包发给 Agent。

### 10.3 内存缓存
不引入 Redis，进程内维护以下短期状态：

- `device_id -> websocket session`
- `device_id -> last heartbeat`
- `message_id -> pending ack`
- `device_id -> current runtime state`

原则：

- 内存只存短期态
- SQLite 存长期事实态
- 服务重启后依赖设备自动重连恢复

其中与在线状态相关的短期态至少应支持：

- `device_id -> websocket session`
- `device_id -> last heartbeat timestamp`
- 后台扫描基于 `last heartbeat timestamp` 识别超时设备

在线状态回收策略要求：

- 正常断开连接时立即标记离线
- 未正常断开但超过 `offline timeout` 时，由后台扫描补充标记离线
- 标记离线时保留最近一次 `last_heartbeat_at`，用于人工排障

```text
      error.log
    devices/
      dev_0001/
        2026-06-08.log
```

日志分流原则：

- 服务日志单独存放
- 设备日志单独存放
- 不混写

### 11.1 脚本版本文件与任务信息的关联方式

任务记录不直接保存脚本源码，而是保存“脚本名 + 脚本版本”这两个业务定位字段。

关联链路建议固定为：

1. `tasks.script_name` + `tasks.script_version`
   定位本次任务要执行哪个脚本版本。
2. `script_versions(script_name, version)`
   定位该版本的元数据记录。
3. `script_versions.package_file_id`
   指向脚本包文件记录。
4. `uploaded_files.file_id`
   定位脚本包实际落盘路径、校验值和上传时间。

也就是说：

`task` -> `script_versions` -> `uploaded_files` -> 磁盘脚本包文件

这样设计的好处是：

1. 同一个脚本版本可以被很多任务复用。
2. 任务表不存大字段，不把脚本源码塞进任务记录。
3. 后续回查某个任务使用了哪个脚本文件、哪个版本包会很直接。
4. 便于后续做脚本版本回滚、灰度和审计。

### 12.3 脚本版本包
脚本以版本包形式交付，并由中心统一解压成目录后管理。上传来源可以是 `zip` 或目录，但中心内部不依赖原始上传形态。每个版本包至少包含：

- `main.js`
- `manifest.json`
- `flows/`
- `pages/`
- `utils/`

其中 `manifest.json` 建议至少包含：

- `script_name`
- `version`
- `entry_file`
- `compatible_agent`
- `checksum`
- `description`

其中：

1. `script_name`
   必须与中心侧 `tasks.script_name`、`script_versions.script_name` 一致。
2. `version`
   必须与中心侧 `tasks.script_version`、`script_versions.version` 一致。
3. `entry_file`
   明确 Agent 最终执行的入口文件，例如 `main.js`。
4. `compatible_agent`
   用于判断当前 Agent 是否可以执行该脚本版本。

### 12.4 版本流转
脚本发布流转固定为：

- `dev`
- `beta`
- `prod`

流程为：

1. 本地调试形成 `dev`
2. 测试设备验证
3. 平台发布为 `beta`
4. 灰度验证通过后升为 `prod`

### 12.5 Agent 下载与执行脚本的推荐链路

手机端不应该在收到 `assign_task` 后直接盲目执行，而应走“检查 -> 拉取清单 -> 下载目录文件 -> 落盘 -> 校验 -> 执行”的固定链路。

推荐流程如下：

1. 中心通过 `assign_task` 下发：
   `task_id`、`device_id`、`script_name`、`script_version`、`params`。
2. Agent 收到任务后，先检查本地是否已有 `/sdcard/mobilerpa/scripts/<script_name>/<script_version>/manifest.json`。
3. 如果本地不存在该版本，或本地 `manifest.json` 与任务要求不一致，则先请求中心脚本元数据接口。
4. Agent 调用脚本元数据接口，获取：
   `script_name`、`version`、`entry_file`、`checksum`、`download_url`。
5. Agent 根据 `manifest.files` 逐个调用下载接口，把脚本版本目录文件同步到本地。
6. Agent 校验入口文件、清单信息和必要摘要值。
7. 校验通过后，确保本地目录为：
   `/sdcard/mobilerpa/scripts/<script_name>/<script_version>/`
8. Agent 再次读取该目录下的 `manifest.json`，确认 `script_name`、`version`、`entry_file` 一致。
9. Agent 组装执行上下文：
   `task_id`、`device_id`、`script_name`、`script_version`、`params`、`trace_id`、`timeouts`。
10. Agent 通过 `entry_file` 指向的入口脚本加载并执行 `run(context)`。
11. 脚本执行过程中，Agent 继续负责进度上报、日志上报、截图收集和最终 `task_result` 回传。

这一链路的核心原则是：

1. 中心负责决定“执行哪个脚本版本”。
2. Agent 负责保证“本地拿到正确版本的脚本包并按入口执行”。
3. 业务脚本只负责自动化逻辑，不负责下载、注册、长连接和平台通信。

进一步约束：

1. `assign_task` 不直接携带完整脚本源码，只携带 `script_name`、`script_version` 和执行参数。
2. Agent 只信任中心返回的 `manifest` 元数据和校验信息，不通过任务参数拼接任意本地脚本路径。
3. 执行入口必须来自脚本包内 `manifest.json.entry_file`，并且该值要与中心 `manifest` 接口返回一致。
4. 如果本地已有同版本缓存且校验一致，Agent 应直接复用本地缓存，不重复下载。

### 12.6 脚本管理页面的远期演进

脚本管理的远期目标不只包括“上传”和“下发”，还包括“查看”和“编辑”。但为了避免已发布版本被直接改写，必须遵守以下演进规则：

1. 网页查看脚本内容属于近中期能力，可在脚本管理页查看版本目录、文件清单和文件内容。
2. 网页编辑脚本属于后续增强能力，编辑后默认应保存为新版本，而不是直接覆盖已发布版本。
3. 已发布版本原则上保持只读，除非用户明确执行高风险覆盖操作。
4. 后续如支持在线编辑，应同步支持版本对比、草稿保存和发布说明。

## 13. 任务执行模型
任务必须采用状态机模型，不建议写成大段顺序脚本。

示例步骤：

- `INIT_APP`
- `CHECK_HOME`
- `CHECK_READY`
- `CHECK_DELAY_TEXT`
- `CLICK_START`
- `WAIT_RESULT`
- `HANDLE_POPUP`
- `VERIFY_LOCKED`
- `COMPLETE`

每推进一步就上报 `task_progress`，这样平台才能准确知道：

- 当前执行到哪一步
- 成功还是失败
- 卡在什么地方
- 是否需要人工介入

## 14. 计划任务调度层规划

当前 `task` 主链路只解决“单次原子执行任务如何下发、执行和回传”的问题，`workflow_def` 主链路只解决“多脚本能力之间如何描述关系模板”的问题。对于开始时间、截止时间、按自然天循环、次日自动启动、多设备统一调度、运行中动态增删设备、统一设备占用裁决等生产调度场景，需要在 `task` 和 `workflow_def` 之上增加一层独立的“计划任务调度层”。

### 14.1 计划任务调度层的核心定义

计划任务调度层后续应建立在以下共识之上：

1. 脚本版本是能力单元，能力以 `script_name + script_version` 为唯一执行版本标识。
2. 工作流定义是能力模板，描述多个脚本能力之间的关系与推进规则。
3. 单任务是能力实例，是某个脚本能力在某台设备上的一次原子执行。
4. 计划任务是调度主体，决定“什么能力在什么时间、以什么节奏、在哪些设备上执行”。
5. 计划任务运行实例是某次调度真正落地后的运行批次，实例下再挂多台设备运行记录。
6. 设备占用最终应归口到计划任务运行域，而不是继续由手工任务域和工作流实例域各自分散维护。

### 14.2 与现有任务系统和工作流系统的关系

这一层的核心约束如下：

1. `task` 仍然保留为单台设备原子执行载体，但不再承担跨天、循环、批次调度职责。
2. `workflow_def` 继续保留为能力模板，不再承载开始时间、截止时间、次日自动启动等调度属性。
3. `plan_def` 用于描述一条正式调度规则，可指向“单脚本+版本”或“工作流定义”。
4. `plan_run` 用于描述计划任务的一次真实启动批次。
5. `plan_device_run` 用于描述某台设备在该批次中的独立运行状态。
6. 后续运行态统一从“计划任务实例”入口查看，旧的“工作流实例层”不再作为主入口长期保留。
7. 手工单任务仍可作为调试/临时执行入口存在，但设备占用的正式裁决权最终收敛到计划任务运行域。

### 14.3 第一版必须支持的能力边界

第一版计划任务调度能力建议支持：

1. 基于单脚本版本创建计划任务。
2. 基于工作流定义创建计划任务。
3. 计划任务定义时选择目标设备集合。
4. 支持一次性执行。
5. 支持按自然天循环执行。
6. 支持每日开始时间。
7. 支持每日截止时间。
8. 支持手动启用和停用计划任务。
9. 支持手动停止当前运行实例。
10. 支持运行中添加设备。
11. 支持运行中删除设备。
12. 支持单设备停止和整体停止。
13. 支持设备占用冲突返回明确错误。
14. 支持工作流目标在运行期间继续复用自身的节点、循环和停止规则。

第一版时间规则收口如下：

1. `daily_start_time` 和 `daily_deadline_time` 统一定义在计划任务层，不下沉到工作流定义层。
2. “次日自动启动”不再依赖前一天是不是命中截止时间，而是依赖计划任务自身是否处于启用状态且配置了按天循环。
3. 只要某台设备在前一天的该计划任务运行实例中被真正启动过，那么第二天到达新的 `daily_start_time` 时，都要重新加入新实例，从头开始执行。
4. 不论设备前一天是因为命中截止时间停止、人工删除设备、执行完成、工作流自然结束，还是其他受控停止原因，只要它属于前一天这条计划任务的已启动设备集合，次日都应按新实例重新启动。
5. 如果计划任务被手动停用，则后续不再自动拉起新实例。
6. 第一版不实现复杂日历、周计划、节假日规则、多时段窗口和节点级时间策略。

第一版暂不强制支持：

1. 图形化拖拽计划日历。
2. 任意复杂条件表达式引擎。
3. 多计划任务并行抢占同一设备。
4. 跨设备共享变量。
5. 在线热替换已运行中的脚本或工作流目标。

### 14.4 计划任务目标模型

计划任务中的目标对象必须明确绑定：

1. `target_type`
2. `target_id`
3. 如为脚本目标，则还需要 `script_name + script_version`
4. 如为工作流目标，则需要 `workflow_def_id`

第一版建议只支持两种目标类型：

1. `script`
   直接调度某个 `script_name + script_version`。
2. `workflow`
   调度某个 `workflow_def_id`。

这意味着：

1. 工作流仍然存在，但它是被计划任务调度的能力模板，而不是最顶层调度实体。
2. 单脚本和工作流在调度入口上统一，不再分别维护两套跨天、循环和设备批量启动逻辑。

### 14.5 运行实例与上下文隔离

计划任务定义与运行实例必须分离：

1. `plan_def`
   共享调度规则，描述“应该在什么时候、以什么方式调度什么目标”。
2. `plan_run`
   一次真实启动批次，描述“这条计划任务在某一天某个时刻启动了一次”。
3. `plan_device_run`
   某台设备在该批次中的独立运行记录。

计划任务运行时上下文必须归属于 `plan_run` 或 `plan_device_run`，而不是归属于模板。第一版建议至少保存：

1. `started_devices_snapshot`
2. `added_devices`
3. `removed_devices`
4. `stop_reason`
5. `last_started_date`
6. `last_completed_date`

若目标类型是工作流，则工作流内部上下文继续保留在工作流执行域；若目标类型是脚本，则可只维护计划任务设备运行上下文。

### 14.6 设备选择、占用与运行中动态调整

计划任务启动时必须明确选择目标设备。定义请求至少应包含：

1. `target_type`
2. `target_id`
3. `device_ids[]`

同一个计划任务一次启动多台设备时，本质上是：

1. 校验所有设备
2. 创建一个 `plan_run`
3. 在该 `plan_run` 下为每台可运行设备创建一条 `plan_device_run`

必须坚持统一设备占用约束：

1. 同一设备同一时刻只允许属于一个运行中的计划任务设备运行记录。
2. 手工任务下发前也必须检查该设备是否已被运行中的计划任务占用。
3. 若设备已被占用，则接口必须返回明确错误，并指出占用它的是哪条计划任务或哪次计划任务运行实例。

计划任务启动后，运行设备集合不应被视为冻结。第一版必须允许两类运行中调整操作：

1. 向当前计划任务实例添加新设备。
2. 从当前计划任务实例移除某台设备。

其中要明确：

1. 添加设备后，应立即创建新的 `plan_device_run`，并立即拉起目标执行。
2. 删除设备后，应立即停止该设备当前执行，不影响同一实例下其他设备继续运行。
3. 添加设备时仍然必须复用与启动时相同的占用校验规则。
4. 已被手动从实例中移除的设备，不再属于该实例的活跃设备集合。

### 14.7 停用、停止与次日是否继续的规则

这部分必须严格区分“停止今天”和“取消未来调度”：

1. 停止某个 `plan_device_run`
   只影响当前实例中的该台设备，不影响该计划任务后续日期是否继续启动。
2. 停止某个 `plan_run`
   只停止今天这次运行实例，不直接改变该计划任务未来是否继续自动启动。
3. 停用某个 `plan_def`
   表示从此不再自动生成新的运行实例。

因此，用于“截至第二天不再启动整个工作流或单脚本计划”的正式方法，应是：

1. 手动停用计划任务。

而不是依赖“昨天手动停止了一次运行实例”去隐式推导未来都不再启动。

### 14.8 数据模型建议

为了避免把调度规则、能力模板和原子执行混在一起，建议至少规划以下表：

1. `plan_defs`
2. `plan_targets`
3. `plan_devices`
4. `plan_runs`
5. `plan_device_runs`
6. `plan_events`
7. `plan_contexts`

其中建议含义如下：

1. `plan_defs`
   维护计划任务定义，例如目标类型、循环方式、开始时间、截止时间、启停状态。
2. `plan_targets`
   维护计划任务目标对象，统一映射到脚本或工作流。
3. `plan_devices`
   维护计划任务绑定的设备集合。
4. `plan_runs`
   维护一次真实启动批次，例如某个计划任务在今天 09:00 被调度启动了一次。
5. `plan_device_runs`
   维护单台设备在该批次下的独立执行状态，例如当前状态、是否手动移除、停止原因。
6. `plan_events`
   维护计划任务域的标准化事件，供前端实例页、消息列表和问题排查使用。
7. `plan_contexts`
   维护批次级或设备级上下文，例如昨日已启动设备集合、手动增删设备记录、最近自动启动日期。

工作流定义层后续只保留能力模板相关字段，不再新增时间策略字段。开始时间和截止时间统一移除到计划任务层。

### 14.9 执行链路与中心/Agent 职责边界

在 10 台以内的早期环境里，中心服务直接启动工作流或直接下发单任务都能工作；但在未来 300 到 500 台设备规模下，必须提前约束中心与 Agent 的职责边界。

推荐采用以下执行链路：

1. 中心服务负责：
   - 管理脚本版本、工作流定义和计划任务定义。
   - 定时扫描计划任务，生成或停止 `plan_run`。
   - 对设备占用冲突、运行状态、停止控制做统一裁决。
   - 聚合计划任务实例事件、单设备运行事件和异常事件。
2. Agent 负责：
   - 接收单脚本执行指令或工作流执行指令。
   - 在本地执行目标脚本，或推进工作流内部节点。
   - 在关键步骤完成时回传标准化事件。
   - 在需要脚本时按脚本名和版本号拉取或校验本地脚本包。

后续推荐的稳态方向不是“页面直接启动工作流实例”，而是：

1. 页面创建或启用计划任务。
2. 中心按计划任务规则生成 `plan_run`。
3. `plan_run` 再决定拉起脚本目标还是工作流目标。
4. Agent 仅回传关键步骤事件、结果事件、停止事件和异常事件。
5. 中心基于这些关键事件更新 `plan_device_runs`、工作流执行域和消息事件域。

### 14.10 计划任务与工作流的典型映射

典型场景应映射为：

1. 工作流定义层负责描述 `A -> B -> CDE 循环 -> FG 循环` 这类能力关系。
2. 计划任务层负责描述“每天 09:00 启动该工作流，23:59 截止，在一组设备上运行”。
3. 单脚本场景则映射为“每天 10:00 调度 `open_wechat@v0.1.0` 到一组设备执行”。

其共同原则是：

1. 能力模板和调度规则必须分离。
2. 单脚本和工作流共享同一套调度机制。
3. 多台设备共享同一条计划任务定义，但运行记录完全隔离。

### 14.11 500 台设备规模下的压力评估与约束

按 `heartbeat = 30s` 估算，500 台设备在线时，平均心跳流量约为：

1. `500 / 30 ≈ 16.7 条心跳/秒`

仅从心跳角度看，这个量级对 Go 中心服务是可控的。真正的压力点不在“在线设备数量本身”，而在以下几类场景：

1. 每天大量计划任务同时到达开始时间，需要瞬时拉起多台设备执行。
2. 每台设备每个步骤都强依赖中心逐条下发，导致双向命令频率过高。
3. 后台实时订阅把所有设备、所有计划任务事件无差别广播给所有页面。
4. 查询接口继续把计划任务实例、设备运行明细和工作流明细混在同一张大列表里。

因此在 300 到 500 台设备目标下，必须遵守以下约束：

1. 心跳、计划任务事件、任务结果、工作流关键步骤事件全部走标准化消息体，但按主题拆分消费。
2. WebSocket 写入按设备连接串行化，避免并发写同一连接。
3. 计划任务实例页、设备运行页和工作流事件页必须分层查询，不做一张大列表全承载。
4. 前端订阅必须按主题、页面、条件做过滤，不做全量广播直出。
5. 计划任务层只保留关键事件，不默认上报超细粒度实时流日志。

### 14.12 第一版验收目标

第一版人工验收建议至少覆盖：

1. 创建一条“单脚本+版本”计划任务。
2. 创建一条“工作流定义”计划任务。
3. 计划任务可绑定多台设备。
4. 到达每日开始时间后，会创建一个新的 `plan_run`。
5. 同一 `plan_run` 下可看到多台设备运行记录。
6. 运行中添加设备后，新设备立即开始执行。
7. 运行中删除设备后，该设备立即停止执行。
8. 到达每日截止时间后，当前仍在运行的设备会被停止。
9. 第二天到达新的开始时间后，会为昨日启动过的设备重新创建新的运行批次。
10. 手动停用计划任务后，次日不再自动启动。
11. 若设备已被其他计划任务占用，则返回明确冲突错误。
12. 运行态统一从计划任务实例页查看，而不是继续从旧工作流实例页查看。

### 14.13 当前迁移目标与边界

截至 2026-06-20，当前正式迁移目标如下：

1. 原有单任务链路继续保留，用作调试和兼容入口。
2. 原有工作流定义继续保留，用作能力模板。
3. 新增计划任务层后，新的开始时间、截止时间、按天循环、次日自动启动全部迁移到计划任务层。
4. 后续运行态主入口统一迁移到计划任务实例页。
5. 原“工作流实例层”页面和接口不作为长期主入口保留，迁移完成后应从主菜单中移除。

当前同时已经明确保留的边界如下：

1. 任务中心与手工任务正式入口已移除；`internal/task` 仅保留为脚本型计划任务执行底座，不再承担独立运行入口语义。
2. 工作流定义页当前仍保留，但只负责维护能力模板，不再负责跨天调度。
3. 该兼容期约束已结束：工作流实例层已进入移除落地阶段，正式运行主线统一按计划任务实例执行。
4. 更细粒度的实时流日志、运行变量编辑、在线流程图编辑仍放在后续阶段，不在当前最小闭环范围内。

### 14.14 统一设备占用判定方案

`14.6` 的统一设备占用目标已在当前实现中落地：设备占用已经收口到 `plan_device_runs.status in (pending, running)` 单域，不再补查 `workflow_runs`。孤立工作流实例的根因修复也已转化为“直接移除工作流实例层”方案的一部分：运行态主链路统一为 `plan_def -> plan_run -> plan_device_run`，不再通过 `POST /api/v1/workflows/{id}/start` 创建任何独立实例。当前权威口径以 `repos/mobilerpa-center/docs/server/中心服务架构.md` 与 `工作流实例层移除迁移说明.md` 为准。

### 14.15 工作流实例层移除与执行域合并（已确认）

经确认，将“工作流实例”概念从项目中移除，工作流型计划任务的运行态直接由计划任务实例承载，消除 `plan_device_runs` 与 `workflow_runs` 的重复记账，使执行域收敛为 `plan_def -> plan_run -> plan_device_run` 单条链路。本节为架构级决策记录，具体边界、接口与 schema 变更以 `repos/mobilerpa-center/docs/server/中心服务架构.md` 的 `工作流实例层移除（已确认方案）` 一节和 `工作流实例层移除迁移说明.md` 为准。

已确认的设计要点：

1. `plan_runs.target_ref_id` 对工作流型指向 `workflow_def_id`（能力模板），不再指向任何实例。
2. 每台设备的工作流执行投影直接落在 `plan_device_runs`（新增 `current_node_id`、最近事件摘要/序号/时间等列）；脚本型这些列留空。
3. 删除“同一时刻只允许一个主工作流运行态”的全局约束，设备占用统一只靠 per-device 占用裁决（见 `14.6` 与统一占用判定）。
4. 删除旧的“中心逐节点 task 推进”链路（`startRunFromNode`、`advanceRunAfterNode` 以及 `tasks` 表上的 `workflow_instance_id`、`workflow_run_id` 列），工作流执行统一走 `workflow_session_*` 会话模型；`workflow_node_id` 仅保留为 session 下派生脚本任务的节点归属字段。
5. 不做数据迁移（当前为开发阶段、无历史包袱）：保留 `workflow_defs`/`workflow_nodes`/`workflow_edges` 定义表，直接删除 `workflow_instances`/`workflow_runs`/`workflow_contexts`/`workflow_events`；schema 变更写入迁移说明。
6. 不保留任何短期兼容：会话载荷按 `plan_run_id` + `plan_device_run_id` 定位，Agent checkpoint 同步改键，前端删除“工作流实例”页，工作流节点事件并入 `plan_events`、在计划任务实例页查看。
7. 手工任务能力已完成正式入口清理：公开 `POST /api/v1/tasks`、`/api/v1/tasks/{id}/terminate`、`/api/v1/devices/{id}/manual-tasks/{id}` 以及相关前端入口均已移除。当前仅保留脚本型计划任务执行底座（`internal/task` 服务、`tasks`/`task_events` 表、`assign_task`/`task_ack`/`task_progress`/`task_result` 协议、`dispatch.AssignTask`）。

影响范围：后端 `plan`/`workflow`/`dispatch`/`ws`/`api`/`storage`/`task`、Agent `workflow_session_runner` 的 keying 与 checkpoint、前端 `WorkflowsPage`/计划任务实例页、相关验收文档。迁移以新开发周期 `C2-7` 跟踪。`14.14` 中“脚本型补查 `workflow_runs`”的临时改动在本节落地后随之移除（合并后占用判定回归 `plan_device_runs` 单域）。

## 15. 调试体系
调试体系必须独立于平台，但可与平台联调。

### 15.1 本地快速调试模式
- 不依赖平台
- 通过 ADB/推送工具直接在真机运行调试入口
- 快速查看日志、截图和页面状态

### 15.2 局域网联调模式
- 手机连接中心服务
- 仅对测试设备下发调试任务
- 验证版本切换、Agent 协议和上报链路

### 15.3 平台生产模式
- 正式设备池
- 统一调度
- 统一日志、结果和日报

### 15.4 调试工具演进
前期保留现有批处理和 ADB 方式，后续逐步演进为 Go CLI，例如：

```text
mobidev push --device 7199294c --script shoppe --version dev
mobidev run --device 7199294c --entry debug_popup.js
mobidev tail --device 7199294c
mobidev pull-screenshot --device 7199294c
```

补充约束：

1. 当前调试工具里的“批量推送”仅针对本机当前通过 ADB 连接并授权的真机。
2. 后续“面向中心服务在线设备的批量 Agent 下发”属于平台能力，应由中心服务、后台和设备侧共同闭环，不以本地 ADB 脚本直接替代。

## 16. 管理后台范围
后台优先建设以下页面：

1. 设备总览
2. 任务控制台
3. 脚本版本管理
4. 异常中心
5. 日报中心
6. 待绑定设备页
7. 消息列表

当前实现顺序调整为：

1. 先建设脚本管理页，完成脚本版本展示、版本文件清单、脚本独立下发和调试入口说明。
2. 再改造任务控制台，让创建任务时从已管理脚本版本中选择，而不是手工输入脚本名和版本。
3. 最后再继续推进工作流编排，让工作流节点引用脚本版本。

第一阶段后台重点不是复杂权限系统，而是服务平台主链路的可视化与可操作性。

后续扩展中，后台还需要补一类“设备运维动作”入口，用于：

- 批量下发 Agent 新版本
- 批量查看设备当前 Agent 版本
- 批量触发配置刷新或重连

但该能力属于平台化运维增强，不和当前本地 ADB 调试脚本混用。

### 16.1 消息列表定位
消息列表不是简单的“日志页复制品”，而是面向值守和排障的统一事件流入口。

消息列表优先展示：

- 设备上线/离线
- 任务分配/开始/完成/失败
- 关键异常与人工关注事件
- 日志中的关键摘要事件

消息列表不承担完整原始日志存档职责，原始日志仍以日志页或日志详情链路为准。

### 16.2 消息列表与异常中心的关系

- 消息列表强调“最近发生了什么”
- 异常中心强调“有哪些异常需要集中处理”

两者应共享同一套后台事件模型，但展示目标不同：

- 消息列表按时间流和主题筛选为主
- 异常中心按严重级别、状态、设备、任务聚合为主

## 18. 部署与基础设施

### 18.1 第一阶段部署
采用单机 Linux 部署。

建议配置：

- MVP 到 100 台：8 核 CPU，16GB 到 32GB 内存，512GB SSD
- 几百台规模：16 核 CPU，32GB 到 64GB 内存，1TB NVMe，UPS，定时备份

部署内容：

- Go 服务二进制
- SQLite 数据文件
- scripts 目录
- uploads 目录
- logs 目录
- 可选 Nginx 反向代理

### 18.2 服务运行方式
- 使用 `systemd` 管理服务
- 每日备份 SQLite 和脚本目录
- 每日归档日志和截图

与设备在线状态相关的运行配置，建议统一进入中心服务启动参数，例如：

- `CENTER_HTTP_ADDR`
- `CENTER_DB_PATH`
- `CENTER_HEARTBEAT_INTERVAL`
- `CENTER_DEVICE_OFFLINE_TIMEOUT`
- `CENTER_DEVICE_OFFLINE_SCAN_INTERVAL`

### 18.3 手机侧基础设施
重点不在 USB，而在：

- 稳定供电
- 稳定 Wi-Fi
- 散热
- 编号体系

建议：

- 每区 30 到 60 台设备
- 企业级 AP
- 统一支架或机架
- 手机编号与 `physical_slot` 对应

ADB 只保留用于：

- 初次安装 Agent
- 本地调试
- 故障排查

不作为生产主链路。

## 19. 风险与边界
第一阶段方案是轻量可落地版，需要明确边界。

### 19.1 适用规模
- 10 到 100 台：非常合适
- 几百台：可继续运行
- 1000 台：可验证业务，但需谨慎控制风险

### 19.2 主要风险
- SQLite 写入竞争
- 单机内存状态易失
- 单点故障
- 本地磁盘增长
- WebSocket 连接管理复杂度上升
- 后台实时事件模型过晚统一会导致前端多页面返工
- 消息列表如果直接消费原始日志，容易导致噪音过大和后续筛选困难

### 19.3 应对策略
- 任务事件批量落库
- 设备自动重连和状态补报
- 日志归档与截图清理
- SQLite 定期备份
- 在代码边界上为未来迁移 PostgreSQL 和 Redis 预留抽象
- 提前统一后台订阅主题和事件信封结构
- 消息列表只消费标准化事件，不直接消费未经归类的原始日志
- 将工作流编排层建立在已验收的单任务执行主链路之上，避免把执行问题和编排问题混在一个周期里处理

## 20. 实施原则总结
后续所有设计与开发遵循以下原则：

1. 先完成单机可运行、可调试、可部署
2. 先打通主链路，再补外围能力
3. 先保证平台与脚本解耦，再追求开发速度
4. 新功能优先进入新仓库，不再向 legacy 目录叠加
5. 先形成稳定规范，再逐步迁移历史脚本和工具
6. 新增会影响架构、计划、看板或验收方式的功能时，先同步更新项目级文档，再进入实现
