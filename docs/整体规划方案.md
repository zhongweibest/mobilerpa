# MobileRPA 整体规划方案

## 1. 文档目的
本文档用于统一 MobileRPA 项目的整体建设方案，覆盖系统目标、仓库规划、技术选型、架构分层、通信协议、数据存储、开发调试、测试部署和基础设施约束。

本文档的定位不是单点设计说明，而是后续各子系统开发的总纲。后续新增的数据库设计、接口协议、模块设计、测试方案，都应与本文档保持一致。

## 2. 项目目标
本项目要建设的是一套可落地的真机自动化执行平台，而不是单机脚本集合。第一阶段目标聚焦以下能力：

1. 批量真机统一接入、注册、绑定和管理
2. 中心服务与手机端建立实时双向通信
3. 业务脚本支持版本化、分发、切换和回滚
4. 每台设备的执行进度、状态和错误可独立追踪
5. 执行日志、截图、dump 和结果能够集中归档与查看
6. 形成从开发、调试、联调、灰度到生产发布的闭环

## 3. 建设思路
整个体系按两条线并行建设：

### 3.1 平台线
平台线负责形成生产运行能力，主要包含：

- 中心服务
- 设备管理
- 任务调度
- 脚本版本管理
- 文件与日志归档
- 报表与异常中心
- 管理后台

### 3.2 调试线
调试线负责形成开发验证能力，主要包含：

- 真机快速调试
- 脚本调试入口
- ADB 辅助工具
- 局域网联调
- 灰度验证
- 问题回放

### 3.3 两条线的关系
两条线共享同一套业务脚本核心代码，但运行入口不同：

- 调试线强调快速验证、快速迭代、快速定位问题
- 平台线强调稳定调度、状态可追踪、结果可归档

这意味着业务脚本不能直接耦合平台通信逻辑，平台适配需要由 Agent 层承担。

## 4. 技术选型

### 4.1 中心端
- 后端语言：Go
- 通信方式：HTTP + WebSocket
- 数据库：SQLite
- 缓存：Go 进程内内存
- 文件存储：Linux 本地磁盘
- 部署方式：单机 Linux

### 4.2 管理后台
- 前端框架：Vue
- 形态：前后端分离
- 与中心服务同仓管理，先采用 monorepo

### 4.3 手机端
- 运行载体：AutoJs6
- 结构：Agent + 业务脚本
- 调试工具：前期保留 ADB/批处理链路，后期演进为 Go CLI

### 4.4 选型原则
- 优先选择当前阶段最容易落地、最容易维护的方案
- 优先保证单机可运行、可调试、可部署
- 为未来迁移 PostgreSQL、Redis、多节点部署预留抽象，但第一阶段不提前引入复杂基础设施

## 5. 仓库与工作区规划

## 5.1 工作区结构
当前工作区保留为一个总工作区，但内部按两个独立仓库推进：

1. `repos/mobilerpa-center`
2. `repos/mobilerpa-agent`

### 5.2 mobilerpa-center
该仓库包含：

- Go 中心服务
- Vue 管理后台
- API 协议文档
- WebSocket 协议文档
- 数据表设计
- 部署与运维文档

采用单仓的原因：

- 当前前后端联动频繁
- 设备模型、任务模型、脚本版本模型还在快速变化
- 协议变更需要前后端一起迭代

### 5.3 mobilerpa-agent
该仓库包含：

- AutoJs Agent
- 业务脚本
- 调试入口
- 真机调试工具
- 脚本版本清单

单独拆仓的原因：

- Agent 和脚本生命周期与中心服务不同
- 手机端调试链路要独立于平台
- 业务脚本版本需要独立演进

### 5.4 暂不拆第三个前端仓库
当前不建议将后台前端再拆成第三个仓库。原因如下：

- 前端团队还不是独立运作形态
- API 和协议尚未稳定
- 额外拆仓只会增加同步与发布复杂度

### 5.5 Legacy 目录处理原则
以下目录暂不删除：

- `projects/`
- `scripts/`
- `choose_device_push.bat`

它们当前作为 legacy 参考源保留，后续按模块逐步迁移到新仓库中，不在旧目录上继续叠加新功能。

## 6. 系统总体架构
系统拆成 5 个核心部分：

1. 中心服务
2. 管理后台
3. 手机 Agent
4. 业务脚本仓库
5. 调试与测试体系

### 6.1 中心服务职责
- 设备注册与绑定
- WebSocket 长连接管理
- 任务下发与状态追踪
- 脚本版本与清单管理
- 日志与文件接收
- 日报、统计、异常聚合

### 6.2 管理后台职责
- 设备管理
- 任务管理
- 脚本版本管理
- 异常中心
- 日报中心
- 待绑定设备管理

### 6.3 手机 Agent 职责
- 首次注册
- 保存本地标识
- 建立 WebSocket 长连接
- 保持心跳
- 接收任务
- 检查脚本版本
- 下载脚本
- 启动脚本
- 上报进度、日志、结果、截图和错误

### 6.4 业务脚本职责
业务脚本只负责自动化执行逻辑，不负责平台通信。职责包括：

- UI 查询
- 页面识别
- 点击与输入
- OCR
- 弹窗处理
- 状态机推进

## 7. 中心与手机通信方案
通信采用双通道设计：

### 7.1 HTTP
负责：

- 注册
- 鉴权
- 脚本清单查询
- 脚本下载
- 文件上传
- 后台管理接口

第一阶段建议保留的接口类型：

- `POST /api/v1/device/register`
- `POST /api/v1/device/upload`
- `GET /api/v1/script/manifest/...`
- `GET /api/v1/script/download/...`

### 7.2 WebSocket
负责：

- 建立设备长连接
- 心跳保活
- 任务下发
- 实时进度回传
- 日志流上报
- 任务结果回传
- 配置刷新和取消指令

建议首批消息类型：

- `hello`
- `heartbeat`
- `assign_task`
- `task_ack`
- `task_progress`
- `task_log`
- `task_result`
- `cancel_task`
- `refresh_config`

### 7.3 统一消息结构
消息统一使用 JSON，最少包含以下字段：

- `type`
- `request_id`
- `device_id`
- `timestamp`
- `payload`

## 8. 设备标识与位置模型

### 8.1 双标识设计
设备采用双标识：

- `device_id`：中心生成，永久不变
- `agent_uuid`：手机本地生成，辅助识别

建议 `device_id` 形式：

- `dev_000001`
- `dev_000823`

### 8.2 device_id 设计原则
`device_id` 不能依赖以下信息：

- IP
- Android ID
- ADB 序列号
- 物理位置

原因是这些字段都可能变化，而平台设备主键必须长期稳定。

### 8.3 首次接入流程
首次接入建议按以下顺序：

1. 手机安装 Agent
2. Agent 首次启动，生成本地 `agent_uuid`
3. Agent 上报基础信息
4. 中心创建待绑定设备记录
5. 中心分配 `device_id`
6. 手机保存 `device_id`
7. 运维在后台完成绑定、分组和物理位置录入

### 8.4 物理位置字段
建议使用 `physical_slot` 记录设备上架位置，例如：

- `Z01-R03-S12`

含义：

- `Z01`：区域
- `R03`：机架或排
- `S12`：槽位

原则：

- `device_id` 永久不变
- `physical_slot` 可变
- 两者必须解耦

## 9. 中心服务模块规划
`mobilerpa-center` 中心服务建议拆成以下模块：

- `api`
- `ws`
- `scheduler`
- `device`
- `task`
- `script`
- `filemgr`
- `report`
- `logger`

推荐目录边界：

- `cmd/`：启动入口
- `internal/app`：应用装配
- `internal/config`：配置
- `internal/api`：HTTP 路由和处理器
- `internal/ws`：WebSocket 生命周期与消息分发
- `internal/device`：设备注册、绑定、状态处理
- `internal/task`：任务创建、流转、结果落库
- `internal/script`：脚本版本和清单
- `internal/scheduler`：调度循环、超时、重试、取消
- `internal/filemgr`：文件路径和上传落盘
- `internal/report`：日报与汇总
- `internal/logger`：日志初始化和设备日志分流
- `pkg/protocol`：共享协议模型

## 10. 数据存储设计

### 10.1 数据库存储原则
第一阶段使用 SQLite，遵循以下原则：

- SQLite 只保存事实态、索引态、汇总态
- 大量原始日志不直接落库
- 大文件、截图、dump 走文件系统

### 10.2 核心表
第一阶段至少包含以下 6 张表：

1. `devices`
2. `tasks`
3. `task_events`
4. `script_versions`
5. `uploaded_files`
6. `daily_reports`

### 10.3 内存缓存
不引入 Redis，进程内维护以下短期状态：

- `device_id -> websocket session`
- `device_id -> last heartbeat`
- `message_id -> pending ack`
- `device_id -> current runtime state`

原则：

- 内存只存短期态
- SQLite 存长期事实态
- 服务重启后依赖设备自动重连恢复

## 11. 文件与日志存储
第一阶段全部采用 Linux 本地磁盘。

建议目录：

/data/mobilerpa/

  scripts/
    shoppe/
      2026.06.08.001/
  uploads/
    screenshots/

  logs/
    server/
      app.log
      ws.log
      scheduler.log
      error.log

      dev_0001/
        2026-06-08.log
```

日志分流原则：

- 服务日志单独存放
- 设备日志单独存放
- 不混写

### 12.1 手机端目录建议

  scripts/
    shoppe/
      2026.06.08.001/

  temp/
```

### 12.2 业务脚本目录建议
```text
scripts/
  shoppe/
    src/
      core/
      pages/
      flows/
      utils/
      state_machine.js
    debug/
      debug_main.js
      debug_popup.js
      debug_start.js
    manifest.json
```

### 12.3 脚本版本包
脚本以版本包形式交付，不再以散文件方式管理。每个版本包至少包含：

- `main.js`
- `manifest.json`
- `flows/`
- `pages/`
- `utils/`

### 12.4 版本流转
脚本发布流转固定为：

- `dev`
- `beta`
- `prod`

流程为：

1. 本地调试形成 `dev`
2. 测试设备验证
3. 平台发布为 `beta`
4. 灰度验证通过后升为 `prod`

## 13. 任务执行模型
任务必须采用状态机模型，不建议写成大段顺序脚本。

示例步骤：

- `INIT_APP`
- `CHECK_HOME`
- `CHECK_READY`
- `CHECK_DELAY_TEXT`
- `CLICK_START`
- `WAIT_RESULT`
- `HANDLE_POPUP`
- `VERIFY_LOCKED`
- `COMPLETE`

每推进一步就上报 `task_progress`，这样平台才能准确知道：

- 当前执行到哪一步
- 成功还是失败
- 卡在什么地方
- 是否需要人工介入

## 14. 调试体系
调试体系必须独立于平台，但可与平台联调。

### 14.1 本地快速调试模式
- 不依赖平台
- 通过 ADB/推送工具直接在真机运行调试入口
- 快速查看日志、截图和页面状态

### 14.2 局域网联调模式
- 手机连接中心服务
- 仅对测试设备下发调试任务
- 验证版本切换、Agent 协议和上报链路

### 14.3 平台生产模式
- 正式设备池
- 统一调度
- 统一日志、结果和日报

### 14.4 调试工具演进
前期保留现有批处理和 ADB 方式，后续逐步演进为 Go CLI，例如：

```text
mobidev push --device 7199294c --script shoppe --version dev
mobidev run --device 7199294c --entry debug_popup.js
mobidev tail --device 7199294c
mobidev pull-screenshot --device 7199294c
```

## 15. 管理后台范围
后台优先建设以下页面：

1. 设备总览
2. 任务控制台
3. 脚本版本管理
4. 异常中心
5. 日报中心
6. 待绑定设备页

第一阶段后台重点不是复杂权限系统，而是服务平台主链路的可视化与可操作性。

## 16. 测试体系
测试分为 4 层：

### 16.1 单元测试
- 设备注册逻辑
- 任务分配逻辑
- ACK 与重试逻辑
- 日报汇总逻辑

### 16.2 集成测试
- HTTP 接口测试
- WebSocket 协议测试
- SQLite 读写测试
- 文件上传测试

### 16.3 真机联调测试
- 1 到 3 台测试机联调
- 验证连接、下载、执行、上报、重连

### 16.4 灰度测试
- 建立测试设备组
- `beta` 版本只发给测试组
- 验证通过再发生产

## 17. 部署与基础设施

### 17.1 第一阶段部署
采用单机 Linux 部署。

建议配置：

- MVP 到 100 台：8 核 CPU，16GB 到 32GB 内存，512GB SSD
- 几百台规模：16 核 CPU，32GB 到 64GB 内存，1TB NVMe，UPS，定时备份

部署内容：

- Go 服务二进制
- SQLite 数据文件
- scripts 目录
- uploads 目录
- logs 目录
- 可选 Nginx 反向代理

### 17.2 服务运行方式
- 使用 `systemd` 管理服务
- 每日备份 SQLite 和脚本目录
- 每日归档日志和截图

### 17.3 手机侧基础设施
重点不在 USB，而在：

- 稳定供电
- 稳定 Wi-Fi
- 散热
- 编号体系

建议：

- 每区 30 到 60 台设备
- 企业级 AP
- 统一支架或机架
- 手机编号与 `physical_slot` 对应

ADB 只保留用于：

- 初次安装 Agent
- 本地调试
- 故障排查

不作为生产主链路。

## 18. 风险与边界
第一阶段方案是轻量可落地版，需要明确边界。

### 18.1 适用规模
- 10 到 100 台：非常合适
- 几百台：可继续运行
- 1000 台：可验证业务，但需谨慎控制风险

### 18.2 主要风险
- SQLite 写入竞争
- 单机内存状态易失
- 单点故障
- 本地磁盘增长
- WebSocket 连接管理复杂度上升

### 18.3 应对策略
- 任务事件批量落库
- 设备自动重连和状态补报
- 日志归档与截图清理
- SQLite 定期备份
- 在代码边界上为未来迁移 PostgreSQL 和 Redis 预留抽象

## 19. 实施原则总结
后续所有设计与开发遵循以下原则：

1. 先完成单机可运行、可调试、可部署
2. 先打通主链路，再补外围能力
3. 先保证平台与脚本解耦，再追求开发速度
4. 新功能优先进入新仓库，不再向 legacy 目录叠加
5. 先形成稳定规范，再逐步迁移历史脚本和工具

## 20. 当前推荐的启动顺序
建议从以下交付物先开工：

1. 中心服务 SQLite 表结构与初始化逻辑
2. WebSocket 与 HTTP 协议定义
3. 设备注册、绑定、心跳主链路
4. Agent 基础骨架
5. 现有 `shoppe` 脚本迁移第一版

这份文档作为项目总纲，后续如需调整，优先在本文档更新，再同步修改子系统文档。
