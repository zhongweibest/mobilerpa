# MobileRPA 群控工作流编排方案

## 1. 文档目的

本文档用于定义 MobileRPA 在“单任务下发”之上的一层能力：群控工作流编排。

这里的目标不是描述某一个固定流程，而是给系统补一套通用编排模型，使平台后续能够支持：

1. 任意顺序的步骤组合
2. 串行执行
3. 循环执行
4. 可配置的循环次数
5. 可配置的截止时间
6. 到达条件后自动停止
7. 次日按定义重新开始
8. 多台手机基于同一套模板独立运行

本文档是方案级设计，不直接替代当前 `task` 主链路实现，也不改变已验收功能的行为边界。

## 2. 问题背景

当前系统正在建设的主链路是：

1. 设备注册
2. WebSocket 长连接
3. 单任务创建
4. 单任务下发
5. Agent 收到任务并确认
6. 后续补脚本执行和结果回传

这条主链路解决的是“一个任务如何从中心到设备并完成回传”的问题。

但真实群控业务需要的是：

1. 多个动作之间有顺序关系
2. 某些动作执行一次后结束
3. 某些动作组成循环块
4. 循环块有最大执行次数
5. 循环块有截止时间
6. 到达截止时间后不再继续执行
7. 第二天重新从头开始
8. 同一个定义可能下发给很多台设备，但每台设备独立推进

因此，单纯依赖当前 `tasks` 表和单次 `assign_task` / `task_result` 已经不够，需要在 `task` 之上增加“工作流编排层”。

## 3. 核心结论

### 3.1 `task` 不是编排对象，而是执行载体

后续系统中：

- `task` 继续表示一次发给单台设备的原子执行任务
- `workflow` 表示一个可配置的流程模板
- `workflow_run` 表示某台设备上的一次实际流程运行

也就是说：

`工作流编排` -> `拆成 task` -> `Agent 执行 task` -> `回传结果` -> `编排器决定下一步`

### 3.2 工作流定义和运行实例必须分离

群控场景下，同一个流程模板可能被几十台手机同时使用，但每台手机的执行进度、循环次数、上下文变量和停止条件命中情况都不同。

因此必须严格区分：

1. `workflow_def`
   共享模板，描述“应该怎么跑”
2. `workflow_run`
   设备实例，描述“这台手机当前跑到了哪里”

不能把“多台设备共用一个运行实例”作为设计前提。

### 3.3 每台注册手机都是一个独立工作流运行实例

这是本方案最重要的约束之一。

同一个工作流定义下：

- 设备 A 可以执行到循环第 12 次
- 设备 B 可以执行到循环第 27 次
- 设备 C 可能已经因为达到 40 次停止
- 设备 D 可能因为达到 `23:59` 截止时间停止

它们共享的是“模板”，不是“运行时状态”。

## 4. 系统要支持的能力边界

### 4.1 本方案需要支持

1. 任意步骤顺序
2. 多段串行流程
3. 多个循环块
4. 每个循环块独立定义最大次数
5. 每个循环块独立定义截止时间
6. 整个流程可定义“执行完即停止”
7. 流程运行中保存上下文变量
8. 次日根据策略重新开始
9. 同一模板下多设备独立实例化

### 4.2 本方案暂不强制一次支持

1. 图形化拖拽编排器
2. 任意复杂条件表达式引擎
3. 多设备同步协同节点
4. 跨设备共享上下文变量
5. 并行分支与汇聚的完整 BPMN 级能力

第一版建议优先支持：

1. 串行节点
2. 循环块
3. 停止条件
4. 时间窗口
5. 实例级上下文

## 5. 业务抽象模型

### 5.1 原子任务

原子任务是最终发给 Agent 的执行单元，例如：

1. 打开软件 A 并搜索
2. 将结果复制到剪贴板
3. 打开软件 B 并粘贴提交
4. 打开软件 C 点击某按钮
5. 打开软件 D 点击某按钮
6. 打开软件 E 点击某按钮

原子任务最终仍落入现有 `tasks` 主链路。

### 5.2 工作流模板

工作流模板描述一套“步骤定义”和“步骤关系”，例如：

1. `step_a`
2. `step_b`
3. `loop_cde`
4. `loop_fg`
5. `stop`

模板本身不绑定具体设备运行进度。

### 5.3 工作流运行实例

工作流运行实例描述某台设备当前实际运行状态，例如：

1. 当前设备是 `dev_000123`
2. 当前在 `loop_cde`
3. 当前循环计数是 `12`
4. 当前上下文里保存着从软件 A 搜索出来的内容
5. 当前还未到 `23:59`

这部分状态必须按设备隔离。

## 6. 编排对象设计

### 6.1 工作流节点类型

第一版建议只定义少量节点类型：

1. `task`
   普通原子任务节点
2. `loop`
   循环块节点
3. `stop`
   明确结束节点

后续再扩展：

1. `condition`
2. `delay`
3. `manual_gate`
4. `branch`

### 6.2 循环块能力

循环块需要至少支持：

1. 包含一组有顺序的子节点
2. 自定义最大次数
3. 自定义截止时间
4. 达到任一停止条件时退出循环
5. 循环计数按运行实例记录

例如：

```text
loop_cde:
  steps = [step_c, step_d, step_e]
  max_count = 40
  stop_at = 23:59
```

### 6.3 次日重启策略

第一版建议支持最简单且最清晰的策略：

1. `restart_from_beginning_next_day`

含义是：

1. 当天达到截止时间后停止当前运行
2. 第二天重新创建或重置运行实例
3. 从第一个节点重新开始

后续再扩展：

1. 次日从指定节点恢复
2. 次日延续上次上下文
3. 次日重置部分上下文

## 7. 运行时上下文

### 7.1 为什么必须有上下文

在你的例子里：

1. 软件 A 搜索出的内容要传给软件 B
2. 当前循环次数要在运行中保存
3. 是否已到截止时间要参与决策

这说明工作流不能只靠“当前状态枚举”推进，还必须保存结构化上下文。

### 7.2 上下文的归属

上下文必须归属于 `workflow_run`，不能归属于模板。

也就是说：

1. 设备 A 搜索出来的值，只属于设备 A 这次运行
2. 设备 B 的上下文完全独立

### 7.3 上下文建议内容

第一版建议至少支持：

1. `variables`
   业务变量，如搜索结果、输入文本、临时标记
2. `loop_counters`
   每个循环块的当前次数
3. `time_window_state`
   截止时间命中状态
4. `last_task_id`
   最近一个下发任务
5. `last_step_code`
   最近完成节点

## 8. 数据模型建议

### 8.1 新增核心表

建议新增以下表：

1. `workflow_defs`
   工作流定义
2. `workflow_nodes`
   工作流节点定义
3. `workflow_edges`
   工作流节点跳转关系
4. `workflow_runs`
   某台设备上的工作流运行实例
5. `workflow_node_runs`
   某次运行里节点执行记录
6. `workflow_contexts`
   某次运行的上下文变量

### 8.2 与现有表的关系

现有表继续保留：

1. `tasks`
2. `task_events`

新增关系应为：

1. `tasks.workflow_run_id`
2. `tasks.workflow_node_id`

这样后续任何一个 `task_result` 都能反查：

1. 属于哪个工作流运行实例
2. 属于哪个编排节点
3. 执行结束后应推进到哪里

### 8.3 核心约束

必须明确：

1. 一个 `workflow_run` 只能绑定一台设备
2. 一个 `task` 只能绑定一个 `workflow_run`
3. 一个 `workflow_run` 的上下文不能与别的设备共享

## 9. 与当前任务系统的衔接方式

### 9.1 当前任务系统继续保留

当前已经在建设的 `task` 主链路不应被推翻，因为它正好可以作为工作流编排的执行底座。

现有链路：

1. 创建任务
2. 下发任务
3. Agent 确认
4. Agent 执行
5. 回传结果

这条链路后续由“编排器”来驱动，而不是直接由人工一次次手动创建。

### 9.2 编排器的职责

编排器不负责真正执行脚本，只负责：

1. 决定当前该执行哪个节点
2. 生成一个原子 `task`
3. 监听 `task_result`
4. 根据结果推进到下一个节点
5. 处理循环计数
6. 处理截止时间
7. 处理停止条件

### 9.3 任务状态与流程状态分层

后续必须区分两层状态：

1. `task.status`
   单次执行任务状态
2. `workflow_run.status`
   整个流程实例状态

例如：

1. 某个 `task` 执行成功
2. 但 `workflow_run` 还处于 `running`
3. 因为后面还有下一节点未执行

## 10. 典型例子映射

### 10.1 例子一：`A -> B -> CDE循环 -> 次日重启`

可映射为：

1. `step_a`
2. `step_b`
3. `loop_cde`
4. `stop`

其中：

```text
loop_cde:
  steps = [step_c, step_d, step_e]
  max_count = 40
  stop_at = 23:59
  next_day_policy = restart_from_beginning_next_day
```

### 10.2 例子二：`A -> B -> CDE循环 -> FG循环`

可映射为：

1. `step_a`
2. `step_b`
3. `loop_cde`
4. `loop_fg`
5. `stop`

每个循环块有自己的限制条件：

```text
loop_cde:
  max_count = 40
  stop_at = 23:59

loop_fg:
  max_count = 10
  stop_at = 23:59
```

### 10.3 例子三：`ABDEF` 执行完即停止

可映射为：

1. `step_a`
2. `step_b`
3. `step_d`
4. `step_e`
5. `step_f`
6. `stop`

不需要循环块。

## 11. 群控场景下的关键运行约束

### 11.1 同一模板，多设备独立运行

系统必须支持：

1. 同一个 `workflow_def` 分配给 10 台、30 台、100 台设备
2. 每台设备独立创建 `workflow_run`
3. 每台设备独立保存上下文和循环次数

### 11.2 单设备同一时刻只跑一个主工作流实例

这是本项目当前阶段的硬约束，不是建议项：

1. 同一设备同一时刻只允许一个主工作流实例处于 `running`
2. 中心在创建 `workflow_run` 或推进新一轮主流程前，必须先检查该设备是否已有 `running` 状态的主工作流实例
3. 如果设备已有正在运行的主工作流实例，则新的主工作流请求不能直接并行启动，必须返回“设备已有运行中主流程”的明确结果

原因：

1. 手机端 UI 自动化天然是串行资源
2. 同时在一台手机上跑多个流程几乎必然互相干扰
3. 这能显著降低调度复杂度和排障难度

当前阶段不设计“一台设备同时运行多个工作流”的能力。

后续如业务确实需要，也只能在你确认后，再单独设计以下扩展能力：

1. 队列等待
2. 低优先级挂起
3. 任务抢占

### 11.3 停止条件优先级

第一版建议停止条件优先级如下：

1. 人工停止
2. 设备离线或异常不可执行
3. 达到截止时间
4. 达到最大循环次数
5. 正常结束

## 12. 对现有开发计划的影响

### 12.1 不建议直接塞入 `C2-3`

`C2-3` 当前目标应该仍然聚焦：

1. 单任务真正执行
2. 单任务回传 `task_result`
3. 单任务回传步骤进度

如果在 `C2-3` 同时引入通用工作流编排，范围会一下扩大，容易把当前主链路拖慢。

### 12.2 建议新增独立开发周期

建议在 `S2` 末尾新增一个周期：

1. `C2-5：群控工作流编排基础`

该周期专门负责：

1. 工作流模板模型
2. 工作流运行实例模型
3. 循环与停止条件
4. 工作流与任务系统衔接
5. 单设备单主流程约束

### 12.3 后续页面能力顺延

工作流编排基础完成后，再进入：

1. 工作流定义管理页
2. 工作流运行监控页
3. 单设备当前流程详情
4. 群控批量启停能力

## 13. 推荐实施顺序

建议按以下顺序落地：

1. 先完成 `C2-2`
   任务下发与回执
2. 再完成 `C2-3`
   单任务脚本执行、`task_result`、进度上报
3. 再完成 `C2-4`
   前端任务页最小入口
4. 然后新增 `C2-5`
   群控工作流编排基础
5. 再进入工作流前端、排障和实时订阅增强

这样做的原因是：

1. 工作流编排必须建立在单任务执行主链路稳定之上
2. 否则会把“执行问题”和“编排问题”混在一起，难以排障

## 14. 第一版验收建议

工作流编排第一版验收建议不要一开始就做复杂 UI，而是先从接口和实例运行验证入手。

建议第一版人工验收至少覆盖：

1. 创建工作流模板
2. 为单台设备创建运行实例
3. 串行步骤自动推进
4. 循环块按最大次数停止
5. 截止时间命中后停止
6. 第二天重新开始
7. 同一模板下两台设备互不影响

## 15. 结论

本项目后续必须在现有 `task` 系统之上增加“群控工作流编排层”。

这层能力的关键不是固定某一种流程，而是支持：

1. 可配置步骤顺序
2. 可配置循环块
3. 可配置截止时间
4. 可配置停止条件
5. 每台手机独立实例化运行

结论性约束如下：

1. `task` 是原子执行载体，不是工作流本身
2. `workflow_def` 是共享模板
3. `workflow_run` 是按设备隔离的运行实例
4. 每台注册手机都是一个独立工作流运行实例
5. 群控系统必须把“模板共享”和“实例隔离”作为基础设计前提
