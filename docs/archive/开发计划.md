# MobileRPA 开发计划

> 归档说明：本文档已归档，只作为早期开发顺序讨论记录。当前开发阶段定义以 `docs/开发阶段计划.md` 为准，当前状态维护以 `docs/项目看板.md` 为准。

> 维护说明：本文档保留为早期开发顺序和依赖关系参考。当前执行状态、任务完成情况和下一步优先级以 `docs/项目看板.md` 为准；若两者不一致，优先采用看板。

## 1. 文档目的
本文档用于定义 MobileRPA 的具体开发顺序，明确哪些部分先做、后做，哪些可以并行推进，哪些必须串行完成。

本文档强调的是“执行顺序”和“依赖关系”，默认以第一阶段可落地 MVP 为目标，并兼顾后续第二、第三阶段的演进路径。

## 2. 开发总原则

### 2.1 先主链路，后外围
优先打通“设备注册 -> 长连接 -> 下发任务 -> 执行脚本 -> 上报结果”这条主链路，再补充后台页面、报表、灰度和运维能力。

### 2.2 先协议和边界，后细节实现
设备模型、任务模型、消息协议、脚本目录边界需要先稳定下来，否则前后端和 Agent 会反复返工。

### 2.3 平台线与脚本线并行
平台线和脚本线要并行推进，但必须共享统一的协议和脚本边界约定。

### 2.4 调试链路不能等平台全部完成
本地调试和联调能力必须尽早建设，否则后续 Agent 和脚本迁移会缺乏验证手段。

## 3. 总体阶段划分
开发分为 6 个阶段：

1. 阶段 A：规划冻结与骨架校准
2. 阶段 B：中心主链路 MVP
3. 阶段 C：Agent 主链路 MVP
4. 阶段 D：脚本迁移与调试体系
5. 阶段 E：后台可视化与生产化能力
6. 阶段 F：规模化优化

其中：

- 阶段 A 是后续所有工作的前置条件
- 阶段 B、C、D 存在较强并行关系
- 阶段 E 依赖前面主链路打通
- 阶段 F 属于优化阶段，不应提前挤占主链路资源

## 4. 阶段 A：规划冻结与骨架校准

### 4.1 目标
统一工作区结构、仓库边界、开发规范和文档基线，为后续编码提供稳定约束。

### 4.2 主要任务
1. 确认 `mobilerpa-center` 和 `mobilerpa-agent` 双仓结构
2. 确认 legacy 目录只保留、不新增
3. 完成整体规划文档
4. 完成开发计划文档
5. 校准中心端和 Agent 端基础目录
6. 明确开发规范和文档约定

### 4.3 并行关系
本阶段内部可少量并行，但原则上应尽快完成后冻结。

### 4.4 是否串行
本阶段是强前置阶段。未完成前，不建议大规模展开后续模块开发。

## 5. 阶段 B：中心主链路 MVP

## 5.1 目标
完成中心服务从存储层到协议层到设备接入层的最小闭环。

### 5.2 串行主线
以下任务建议按顺序推进：

1. SQLite 表结构设计
2. 数据库初始化与迁移入口
3. 设备注册接口
4. 设备绑定状态模型
5. 心跳接口或心跳处理逻辑
6. WebSocket 连接入口
7. 消息协议解析与分发
8. 基础任务下发模型
9. 基础任务状态回写

原因：

- 协议处理建立在数据模型之上
- WebSocket 生命周期依赖设备身份模型
- 调度和任务状态依赖连接状态和任务表结构

### 5.3 可并行任务
以下工作可以与串行主线并行：

- 服务配置加载完善
- 日志初始化框架
- 本地文件目录约定
- API 错误码结构
- 协议文档整理
- 单元测试样板搭建

### 5.4 本阶段交付结果
完成后应至少具备：

1. 中心服务可启动
2. SQLite 可初始化
3. 设备可以注册并获得 `device_id`
4. 设备可建立 WebSocket 连接
5. 中心可识别 `hello` 和 `heartbeat`
6. 中心可为单设备分配一个基础任务

## 6. 阶段 C：Agent 主链路 MVP

### 6.1 目标
完成手机端 Agent 的最小可运行骨架，使其能接入中心并执行最基础的脚本任务。

### 6.2 串行主线
建议按以下顺序：

1. `agent.js` 基础入口
2. 本地配置读写
3. `agent_uuid` 生成与持久化
4. 注册调用封装
5. `device_id` 保存
6. WebSocket 连接管理
7. 心跳发送
8. 消息处理分发
9. 脚本版本检查
10. 基础脚本执行入口
11. 结果与进度上报

### 6.3 可并行任务
可以并行推进：

- Agent 日志封装
- 本地目录初始化
- 截图与 dump 工具函数
- 错误处理和重连策略草案
- 基础配置模板

### 6.4 与阶段 B 的关系
阶段 C 与阶段 B 高度并行，但有两个强依赖点：

1. 注册接口必须先稳定到可用
2. WebSocket 消息结构必须先有第一版定义

因此最合理的方式是：

- 阶段 B 先快速产出协议骨架和注册骨架
- 阶段 C 随即接上联调

## 7. 阶段 D：脚本迁移与调试体系

### 7.1 目标
建立新脚本结构，把现有业务逻辑逐步迁移进 `mobilerpa-agent`，同时形成独立调试能力。

### 7.2 串行主线
建议顺序如下：

1. 确认脚本目录结构
2. 建立 `manifest.json` 规范
3. 建立 `src/` 与 `debug/` 边界
4. 建立通用脚本执行入口
5. 迁移 `shoppe` 第一版主流程
6. 迁移页面模块和工具模块
7. 补任务状态机步骤拆分
8. 接入 Agent 调用
9. 加入进度上报钩子

### 7.3 可并行任务
以下工作可以并行：

- 调试入口脚本整理
- ADB 调试文档整理
- 批处理工具梳理
- Go CLI 方案预研
- 脚本清单格式校验工具

### 7.4 与阶段 B/C 的关系
阶段 D 与阶段 C 强耦合，与阶段 B 中度耦合：

- 没有 Agent 骨架，脚本无法走平台链路
- 没有脚本迁移，平台无法验证真实业务执行
- 没有调试链路，脚本迁移成本会快速升高

因此 D 阶段必须尽早开始，而不是等平台完成后再做。

## 8. 阶段 E：后台可视化与生产化能力

### 8.1 目标
在主链路跑通后，补足面向运维和生产使用的可视化管理能力。

### 8.2 推荐顺序
建议按以下顺序：

1. 设备总览页
2. 待绑定设备页
3. 任务控制台
4. 脚本版本管理页
5. 异常中心
6. 日报中心

原因：

- 设备页最先支撑联调和运维
- 任务页支撑实际投放
- 版本页支撑脚本分发闭环
- 异常和日报建立在前述数据稳定基础上

### 8.3 可并行任务
可并行推进：

- Vue 工程基础搭建
- API SDK 封装
- 页面布局与导航
- 通用状态颜色和字段格式化
- 基础权限占位

### 8.4 与前序阶段关系
阶段 E 不应先于阶段 B/C/D 大规模启动，但前端工程脚手架可以更早开始。

建议做法：

- 主链路稳定前，只做前端骨架和设备页
- 主链路稳定后，再扩展任务页和版本页

## 9. 阶段 F：规模化优化

### 9.1 目标
在第一阶段可生产运行后，补充面向几十台到几百台设备的规模化能力。

### 9.2 主要任务
1. 设备分组
2. 灰度发布
3. 任务批量投放
4. ACK 与重试完善
5. 日志归档和清理策略
6. 截图与文件归档策略
7. 调度性能优化
8. SQLite 压力评估
9. PostgreSQL/Redis 抽象预留梳理

### 9.3 是否并行
本阶段大部分能力可并行，但必须建立在前面各阶段完成的基础上。

## 11. 并行与串行关系总览

### 11.1 强串行项
以下内容必须基本串行：

1. 整体规划冻结 -> 协议第一版 -> 中心注册模型
2. 数据表结构 -> 数据访问层 -> 注册与状态处理
3. Agent 本地标识 -> 注册流程 -> `device_id` 保存 -> 长连接
4. 脚本结构确定 -> `manifest` 规范 -> 脚本迁移

### 11.2 可并行项
以下内容适合并行：

1. 中心服务骨架与 Agent 骨架
2. 协议文档整理与代码骨架搭建
3. 脚本迁移与调试工具整理
4. 前端基础工程与中心端 API 开发
5. 日志体系与文件目录管理

### 11.3 典型并行组合
推荐的并行组合如下：

1. 一条线做中心端数据库和注册接口
2. 一条线做 Agent 本地标识、注册调用和心跳
3. 一条线做 `shoppe` 脚本迁移和调试入口
4. 一条线做前端设备页骨架

这四条线共享同一个协议文档和字段约定。

## 12. 第一阶段推荐执行节奏

### 12.1 第一步
先完成以下三项：

1. SQLite 表结构
2. 设备注册接口
3. WebSocket 协议第一版

### 12.2 第二步
紧接着并行推进：

1. Agent 注册与长连接
2. `shoppe` 脚本迁移第一版
3. 前端设备页骨架

### 12.3 第三步
主链路联调：

1. 中心识别设备上线
2. Agent 拉起脚本
3. 平台收到进度和结果
4. 日志和截图能回传

### 12.4 第四步
补齐生产最小能力：

1. 待绑定设备页
2. 基础任务下发
3. 脚本版本管理最小闭环
4. 测试设备组联调

## 13. 当前最推荐的开工项
如果按当前工作区状态继续推进，最推荐的开工顺序是：

1. `repos/mobilerpa-center`：SQLite 表结构、初始化逻辑、设备注册接口
2. `repos/mobilerpa-center`：WebSocket 协议定义与基础消息处理
3. `repos/mobilerpa-agent`：`agent.js` 基础骨架
4. `repos/mobilerpa-agent`：迁移 `shoppe` 第一版脚本结构
5. `repos/mobilerpa-center/web`：设备页骨架

## 14. 验收标准
本计划的第一阶段完成标准是：

1. 新设备安装 Agent 后可注册并拿到 `device_id`
2. 设备可稳定连接中心并周期心跳
3. 中心可向单设备下发一个基础任务
4. Agent 可执行一个迁移后的脚本入口
5. 平台能看到设备状态、任务状态和基础日志

达到以上标准后，再进入脚本版本管理、异常中心和日报中心的扩展阶段。
