# MobileRPA 项目看板

## 1. 看板使用方式

本文档是 MobileRPA 当前执行状态、任务完成情况和下一步优先级的唯一维护入口。

看板按三层维护：

1. 开发阶段：例如 `S1` 中心接入主链路。
2. 开发周期：例如 `C1-1` WebSocket `hello/heartbeat` 链路。
3. 开发任务：例如 `P1-BE-006` 完成 WebSocket `hello` 手工验收脚本。

验收文档选择规则：

1. 每个需要人工验收的开发周期，必须在周期目标后写明“验收文档”。
2. 当前任务验收时，优先使用看板中该周期标注的验收文档。
3. 如果看板中没有标注，使用 `docs/acceptance-checklists/README.md` 的验收入口索引。
4. 如果标注为“待实现后生成”，说明当前功能还没有真实可执行验收文档，不能使用归档草稿替代验收。
5. 如果两处都没有对应文档，需要先补充或确认模块验收文档，不能临时猜测使用哪份文档。

## 2. 状态标记规则

- `[ ]`：未开始
- `[~]`：进行中
- `[?]`：等待用户验收
- `[x]`：已完成
- `[!]`：阻塞中

## 3. 状态维护硬约束

1. 任务、周期、阶段都必须有独立状态。
2. 我可以在实际开始开发某个任务时，把任务状态从 `[ ]` 改为 `[~]`。
3. 当我完成实现、联调或验证准备后，只能把任务状态改为 `[?]`，表示等待用户验收。
4. 只有用户明确确认“验收通过”“确认完成”或表达等价含义后，我才能把任务、周期或阶段状态改为 `[x]`。
5. 不能仅凭代码写完、测试通过、脚本跑通或我自己的判断，就把任务、周期或阶段标记为 `[x]`。
6. 一个开发周期内的必须任务全部 `[x]` 后，还需要用户明确确认该周期验收通过，才能把周期标记为 `[x]`。
7. 一个开发阶段内的必须周期全部 `[x]` 后，还需要用户明确确认该阶段验收通过，才能把阶段标记为 `[x]`。
8. 进入下一个开发周期或开发阶段前，需要先完成当前周期或阶段验收，除非用户明确同意并行推进或调整顺序。
9. 如果我认为计划顺序需要调整，只能先提出建议和理由，必须经过用户同意后才能修改计划或改变推进顺序。
10. 默认文档和代码注释都使用中文。
11. 如果我有疑问、觉得要求不合理或发现隐藏风险，必须先停下来说明判断、建议和可选方案，等待用户确认后再执行。
12. 涉及工程惯例的文件名或入口名，例如 `README.md`、配置文件、启动入口、目录结构，不允许我直接按自己的偏好改动，必须先让用户确认。

## 4. 责任角色

- 后端：中心服务、数据库、HTTP、WebSocket、任务调度、文件归档
- Agent/脚本：AutoJs Agent、业务脚本、调试入口、真机联调
- 前端：Vue 管理后台、页面交互、接口联调
- 验收：人工功能验证、试运行记录、问题回归
