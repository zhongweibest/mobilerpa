# MobileRPA 开发阶段计划

## 1. 文档目的

本文档用于定义 MobileRPA 的开发阶段、开发周期和任务拆分关系。

后续项目推进采用三层结构：

1. 开发阶段：描述一组阶段性业务目标。
2. 开发周期：描述阶段内可以验收的一段开发闭环。
3. 开发任务：描述最终落到代码、文档、联调或人工验收的具体事项。

本文档不再维护工时估算。当前任务状态、周期状态、阶段状态统一在 `docs/项目看板.md` 中维护。

## 2. 状态与验收门禁

### 2.1 状态标记

- `[ ]`：未开始
- `[~]`：进行中
- `[?]`：等待用户验收
- `[x]`：已完成
- `[!]`：阻塞中

### 2.2 完成状态规则

开发任务、开发周期、开发阶段都必须遵守同一条规则：

只有在用户明确确认“验收通过”“确认完成”或表达等价含义后，才能把状态改为 `[x]`。

不能因为代码写完、测试通过、脚本跑通或我自行判断完成，就把任务、周期或阶段标记为 `[x]`。

### 2.3 层级完成规则

1. 一个开发任务完成并通过用户验收后，才可以把该任务标记为 `[x]`。
2. 一个开发周期内所有必须任务都标记为 `[x]`，并且用户明确确认该周期验收通过后，才可以把该周期标记为 `[x]`。
3. 一个开发阶段内所有必须周期都标记为 `[x]`，并且用户明确确认该阶段验收通过后，才可以把该阶段标记为 `[x]`。
4. 进入下一个开发周期或开发阶段前，需要先完成当前周期或阶段的验收，除非用户明确同意并行推进或调整顺序。

### 2.4 验收文档入口规则

每个开发周期都必须能找到对应的人工验收文档。

选择顺序：

1. 优先查看 `docs/项目看板.md` 中该开发周期的“验收文档”字段。
2. 如果看板没有写清楚，再查看 `docs/acceptance-checklists/README.md` 的验收入口索引。
3. 如果看板标注为“待实现后生成”，说明该功能还没有真实可执行验收文档，不能使用归档草稿替代验收。
4. 如果仍然没有对应关系，需要先补充或确认模块验收文档，不能直接开始验收。
5. 新增或调整开发周期、开发任务、模块验收文档时，必须同步维护看板和验收入口索引。
6. 新增会影响总体方案、协议、配置、运行策略或任务编排方式的功能时，必须先同步更新 `docs/整体规划方案.md`、`docs/开发阶段计划.md` 和 `docs/项目看板.md`，再进入实现。

### 2.5 计划调整规则

如果我发现计划顺序不合理、依赖关系有风险或某些任务适合提前/延后，我可以提出调整建议。

但在用户同意前：

1. 不擅自修改开发阶段、开发周期和关键任务顺序。
2. 不擅自把任务移动到新的周期或阶段。
3. 不因为我认为“更合理”就直接进入下一个阶段或功能开发。

### 2.6 协作确认规则

如果我有疑问、觉得某个要求不合理，或者发现某个操作存在隐藏影响，必须先暂停执行并向用户说明：

1. 我理解的当前目标是什么。
2. 我看到的风险或不合理点是什么。
3. 我建议的处理方式是什么。
4. 还有哪些可选方案。

用户确认前，不擅自继续执行会产生结果变化的操作。

涉及工程惯例的命名尤其需要确认，例如 `README.md` 是否保留、是否改名、是否增加中文说明文件，都必须先让用户确认。

### 2.7 执行顺序与并行原则

项目推进优先围绕主链路展开，主链路顺序为：

1. 设备注册
2. WebSocket 长连接
3. 单任务下发
4. 脚本执行
5. 进度、日志、截图和结果回传

执行原则：

1. 先稳定设备模型、任务模型、消息协议和脚本目录边界，再扩展页面和运维能力。
2. 中心端、手机端、脚本端和前端可以并行推进，但必须共享同一套协议和字段约定。
3. 本地调试和真机联调不能等平台全部完成后再做，调试入口应尽早建设。
4. 主链路未跑通前，不提前投入大量资源做报表、复杂权限、完整异常中心和大规模灰度平台化能力。
5. 后台实时能力统一采用“HTTP 全量 + WebSocket 增量”的模式，不鼓励各页面分别建设长期轮询逻辑。

强串行依赖：

1. 整体方案与协议第一版 -> 中心注册模型 -> 设备接入。
2. SQLite 表结构 -> 数据访问层 -> 注册与状态处理。
3. `agent_uuid` 生成 -> 注册流程 -> `device_id` 保存 -> WebSocket 长连接。
4. 脚本目录结构 -> `manifest.json` 规范 -> 脚本迁移 -> 平台调用。

推荐并行组合：

1. 后端推进数据库、注册接口、WebSocket 和任务模型。
2. 手机端推进本地身份、注册调用、长连接和任务消息分发。
3. 脚本端推进 `shoppe` 迁移、本地调试入口和关键步骤上报。
4. 前端推进设备页、任务页骨架和接口占位。
5. 验收角色从第一个开发周期开始准备测试机、接口集合、验收记录和问题回归。

### 2.8 后台实时订阅推进约束

最终系统需要覆盖设备、任务、日志、异常四类实时消息，并建设一个统一的消息列表能力。为了避免后续返工，开发阶段计划增加以下约束：

1. 首屏查询和历史回看统一走 HTTP。
2. 实时变化统一走 WebSocket 增量推送。
3. 页面不能各自独立长期轮询替代实时订阅。
4. 后台实时协议优先按主题拆分，例如 `devices`、`tasks`、`logs`、`alerts`、`feed`。
5. 消息列表只消费标准化事件，不直接消费所有原始日志。
6. 设备实时、任务实时、日志实时、异常实时和消息列表不要求同一周期一次做完，必须分阶段落地。

推荐落地顺序：

1. 先完成设备页最小闭环，再补设备状态实时推送。
2. 任务主链路稳定后，再补任务状态和进度实时推送。
3. 日志与异常链路稳定后，再补日志摘要、异常告警和消息列表。
4. 早期生产试运行阶段，再补重连、去重、补偿和保序等增强能力。

### 2.9 角色协作边界

推荐按以下角色划分责任：

1. 后端角色：中心服务配置、SQLite、HTTP、WebSocket、任务主链路、文件接收和日志归档。
2. 手机端/脚本角色：手机端入口、本地配置、长连接、脚本执行、业务脚本迁移和真机调试工具。
3. 前端角色：后台页面、设备查看、任务下发入口、设备绑定、日志和截图查看。
4. 联调与验收角色：测试设备准备、人工验收步骤执行、问题回归、试运行记录。

团队人数较少时可以兼任，但责任边界和验收责任仍需在看板中保持清晰。

## 3. 开发阶段总览

| 阶段 | 名称 | 阶段目标 | 状态维护位置 |
|---|---|---|---|
| S0 | 项目初始化与文档基线 | 完成仓库、规范、最小中心接口和文档基线 | `docs/项目看板.md` |
| S1 | 中心接入主链路 | 设备可注册、上线、心跳并被中心识别 | `docs/项目看板.md` |
| S2 | 单任务下发与脚本执行 | 中心可向单设备下发任务，Agent 可执行脚本并回传结果 | `docs/项目看板.md` |
| S3 | 最小后台与排障能力 | 人可以通过后台查看设备、任务、日志和截图 | `docs/项目看板.md` |
| S4 | 早期生产试运行 | 支撑 10 到 30 台设备进入早期生产试运行 | `docs/项目看板.md` |
| S5 | 规模化与生产加固 | 向 50 到 100 台、再到 300 台分阶段扩展 | `docs/项目看板.md` |

## 4. S0：项目初始化与文档基线

阶段目标：完成双仓结构、开发规范、中文文档规则、中心服务最小 HTTP 验收链路。

周期目标：

1. 初始化 `mobilerpa-center` 和 `mobilerpa-agent` 两个仓库。
2. 补齐 `.gitignore`。
3. 完成首次提交。
4. 明确中文文档和中文代码注释规则。

主要任务：

- `DONE-001` 规划双仓结构。
- `DONE-002` 初始化中心仓库。
- `DONE-003` 初始化 Agent 仓库。
- `DONE-004` 完成两个仓库首次提交。
- `DONE-005` 完成中心端 `.gitignore`。
- `DONE-006` 完成 Agent 端 `.gitignore`。
- `DONE-010` 统一中文文档和中文代码注释规则。

周期验收方式：

- 用户确认仓库结构可接受。
- 用户确认文档和注释默认中文规则可接受。

### C0-2：中心 HTTP 最小验收链路

- `DONE-008` 完成中心端设备列表接口。
- `DONE-009` 完成真实启动服务和 API 验收脚本。
- `P1-BE-001` 建立中心服务启动入口。
- `P1-BE-002` 建立 SQLite 初始化逻辑。
- `P1-BE-002A` 支持 `.env` 默认配置启动中心服务。
- `P1-BE-003` 实现设备注册接口。

## 5. S1：中心接入主链路

阶段目标：设备可以注册，中心可以识别设备，并能通过 WebSocket 维护基础在线状态。

### C1-1：WebSocket `hello/heartbeat` 链路

周期目标：

1. 模拟客户端可以连接中心 `/ws`。
2. 模拟客户端可以发送 `hello`。
3. 模拟客户端可以发送 `heartbeat`。
4. 中心可以记录设备在线状态。
5. 断开连接后具备基础离线判定。

主要任务：

- `P1-BE-006` 完成 WebSocket `hello` 手工验收脚本。
- `P1-BE-007` 完成 WebSocket `heartbeat` 手工验收脚本。
- `P1-BE-008` 增加设备离线判定策略。
- `P1-QA-002` 完成 WebSocket 上线验收。

周期验收方式：

- 用户通过真实服务和模拟脚本确认 `hello/heartbeat` 链路通过。
- 用户确认设备在线、离线状态变化符合预期。

### C1-1A：心跳超时自动离线增强

周期目标：

1. 中心在 WebSocket 未显式断开时，仍能根据最近心跳时间自动回收在线状态。
2. 心跳间隔、离线超时、扫描间隔进入中心服务启动配置。
3. 人工验收可以覆盖“停止心跳但连接未及时断开”的场景。

主要任务：

- `P1-BE-008A` 实现心跳超时自动离线扫描。
- `P1-BE-008B` 支持 `CENTER_HEARTBEAT_INTERVAL`、`CENTER_DEVICE_OFFLINE_TIMEOUT`、`CENTER_DEVICE_OFFLINE_SCAN_INTERVAL` 启动配置。
- `P1-QA-002A` 完成心跳超时自动离线人工验收文档。

周期验收方式：

- 用户通过真实服务、Apifox、真机或等价模拟环境确认“连接未显式断开但超过离线超时后自动变为 `offline`”。
- 用户确认三个配置项可通过启动参数生效，并且手动环境变量仍可覆盖 `.env` 默认值。

### C1-2：Agent 注册与心跳

周期目标：

1. Agent 可启动。
2. Agent 可读写本地配置。
3. Agent 可生成并持久化 `agent_uuid`。
4. Agent 可调用中心注册接口并保存 `device_id`。
5. Agent 可连接 WebSocket 并发送心跳。
6. Agent 在中心服务短暂下线后可自动重连并恢复心跳。

主要任务：

- `P1-AG-001` 建立 `agent.js` 主入口。
- `P1-AG-002` 实现本地配置读写。
- `P1-AG-003` 实现 `agent_uuid` 生成和持久化。
- `P1-AG-004` 实现中心注册调用。
- `P1-AG-005` 实现 WebSocket 连接。
- `P1-AG-006` 实现心跳发送。
- `P1-AG-006A` 实现 WebSocket 自动重连。
- `P1-AG-007` 增强调试推送脚本，支持对当前 ADB 已连接设备使用 `all` 批量推送 Agent。

周期验收方式：

- 用户在 AutoJs6 或等价真机环境中确认 Agent 可启动、注册、保存身份并上报心跳。
- 用户在中心服务短暂关闭后，确认 Agent 会按 `3s -> 6s -> 12s -> 24s -> 48s -> 60s -> 60s ...` 的节奏无限次尝试重连，并在服务恢复后自动重新上线。
- 用户可通过 `push-agent.bat` 的 `all` 选择，对当前已连接且已授权的多台真机完成一次批量推送。

### C1-3：设备列表前端入口

周期目标：

1. 前端工程结构明确。
2. 设备页骨架可展示。
3. 设备列表接口可接入。
4. 页面可显示设备注册状态和心跳字段。

主要任务：

- `P1-QA-004A` 完成设备删除人工验收。

周期验收方式：

- 用户通过浏览器或前端运行结果确认设备列表页面可看到真实接口数据。
- 用户通过 PowerShell、Apifox 或前端页面确认离线设备可以删除，删除后列表刷新不再显示该设备。

### C1-4：阶段验收与接口集合

周期目标：

1. 补齐 Apifox 接口集合或等价 API 验收说明。

## 6. S2：单任务下发与脚本执行

阶段目标：中心可以向单台设备下发任务，Agent 可以拉起脚本并回传执行结果。

### C2-1：任务模型与中心接口

周期目标：

1. 任务状态枚举明确。
2. 任务创建接口可用。
3. 任务列表接口可用。
4. 任务事件记录可用。
5. 提供任务事件查询接口，便于通过 Apifox 等工具直接验收事件记录。
6. 为后续后台任务实时推送预留统一事件模型。

主要任务：

- `P2-BE-001` 定义任务状态枚举。
- `P2-BE-002` 实现任务创建接口。
- `P2-BE-003` 实现任务列表接口。
- `P2-BE-003A` 实现任务事件查询接口。
- `P2-BE-007` 实现任务事件记录。
- `P2-BE-008` 设计后台任务事件标准化结构，供后续订阅推送和消息列表复用。

周期验收方式：

- 用户通过 API 工具确认任务创建、查询和事件记录可用。
- 用户确认任务事件结构不只适用于落库，也适用于后续实时订阅扩展。

### C2-2：任务下发与回执

周期目标：

1. 中心可向指定设备发送 `assign_task`。
2. Agent 可识别任务消息。
3. Agent 可发送 `task_ack`。
4. 中心可更新任务接收状态。

主要任务：

- `P2-BE-004` 实现单设备任务下发。
- `P2-BE-005` 实现 `task_ack` 处理。
- `P2-AG-001` 实现任务消息分发。
- `P2-AG-002` 实现任务确认回传。
- `P2-QA-001` 完成单设备任务下发验收。

周期验收方式：

- 用户确认中心下发任务后，设备或模拟 Agent 能收到并确认。

当前验收文档入口：

- `repos/mobilerpa-center/docs/server/中心服务手工验收.md` 中的 `8B. C2-2 任务下发与回执手工验收`
- `repos/mobilerpa-agent/docs/手机端手工验收.md` 中与 `assign_task/task_ack` 相关的日志观察步骤

### C2-3：脚本执行与结果回传

周期目标：

1. Agent 可根据任务参数拉起脚本。
2. `shoppe` 第一版入口可被 Agent 调用。
3. 脚本执行结果可回传中心。
4. 中心可显示成功或失败结果。
5. Agent 本地缺少脚本版本时，可从中心服务下载脚本元数据和脚本文件后再执行。

主要任务：

- `P2-AG-003` 实现脚本执行入口。
- `P2-AG-004` 实现执行结果回传。
- `P2-AG-005` 迁移 `shoppe` 第一版入口。
- `P2-BE-009` 实现脚本元数据与下载接口。
- `P2-AG-007` 实现 Agent 缺失脚本时自动下载。
- `P2-AG-006` 增加关键步骤进度上报。
- `P2-BE-006` 实现 `task_result` 处理。
- `P2-QA-002` 完成脚本执行验收。
- `P2-QA-003` 完成结果回传验收。

周期验收方式：

- 用户在真机或接近真实环境中确认脚本被拉起并回传结果。
