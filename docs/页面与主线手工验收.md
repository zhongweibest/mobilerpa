# MobileRPA 页面与主线手工验收

## 1. 目的

本文档作为中心端正式验收入口，按真实页面操作顺序覆盖当前已经落地的后台主线能力。

当前中心端正式运行入口统一为：

1. 设备发现
2. 设备列表与设备绑定
3. 脚本管理
4. 工作流编排
5. 计划任务
6. 计划任务实例
7. 系统配置

如页面结果异常，再查看：

1. `repos/mobilerpa-center/docs/server/中心服务补充检查手册.md`
2. `repos/mobilerpa-agent/docs/手机端手工验收.md`

## 2. 前置条件

1. 中心服务已经启动。
2. 管理后台已经启动，并能访问 `http://127.0.0.1:5173`。
3. 手机准备、AutoJs6 权限、无线调试配对方式，以 `docs/系统使用说明书.md` 为准。
4. 至少有一台已完成无线调试配对的 Android 设备。
5. Agent 发布目录和脚本发布目录已经准备完成。

## 3. 打开后台与基础菜单

浏览器访问：

```text
http://127.0.0.1:5173
```

预期结果：

1. 左侧存在功能菜单区域。
2. 菜单中至少可以看到：`首页看板`、`设备列表`、`设备发现`、`设备绑定`、`计划任务`、`计划任务实例`、`工作流编排`、`脚本管理`、`系统配置`。
3. 首次进入默认打开“首页看板”。
4. 左侧菜单以当前正式功能入口为准，不出现额外的独立运行入口。

## 4. 设备发现与 Agent 下发

浏览器访问：

```text
http://127.0.0.1:5173/discovery
```

验收步骤：

1. 点击“刷新发现结果”。
2. 勾选一台或多台可连接设备。
3. 填写真机可访问的中心服务地址，例如 `http://192.168.0.155:18080`。
4. 根据需要选择“重置设备已有配置”“推送后自动运行 Agent”。
5. 点击批量下发按钮。

预期结果：

1. 页面存在“设备发现”标题、“刷新发现结果”按钮和中心地址输入项。
2. 页面发起 `GET /api/v1/discovery/devices` 请求，并显示发现到的设备列表。
3. 只有 `connectable=true` 的设备允许勾选。
4. 页面发起 `POST /api/v1/discovery/agent-deployments` 请求。
5. 下发结果按设备逐项展示连接、推送、启动、状态和说明。
6. 如页面提供单设备“启动 Agent”“停止 Agent”操作，反馈必须明确到具体设备。

## 5. 设备列表与设备绑定

### 5.1 设备列表

浏览器访问：

```text
http://127.0.0.1:5173/devices
```

预期结果：

1. 页面标题区域显示“设备总览”或等价标题。
2. 页面存在“刷新设备”按钮。
3. 表格中可看到设备名称或 `device_id`、在线状态、绑定状态、最近心跳、物理位置和删除按钮。
4. `status=online` 显示“在线”，`status=offline` 显示“离线”。
5. `bind_status=pending` 显示“待绑定”。
6. `last_heartbeat_at` 为空时显示“暂无”。
7. `physical_slot` 为空时显示“未录入”。

补充检查：

1. 离线设备允许删除。
2. 在线设备删除按钮保持禁用。
3. “查看占用”弹窗能显示 `plan_run_id`、`plan_device_run_id` 和占用说明。

### 5.2 设备绑定

浏览器访问：

```text
http://127.0.0.1:5173/bindings
```

预期结果：

1. 页面可分别创建分区、排号、槽位三个层级节点。
2. 排号创建时必须先选择分区。
3. 槽位创建时必须先选择分区和排号。
4. 槽位列表只展示 `slot` 节点，并能看到 `分区-排号-槽位` 的完整路径。
5. 设备绑定与解绑只能落在槽位级节点。

## 6. 脚本管理

浏览器访问：

```text
http://127.0.0.1:5173/scripts
```

预期结果：

1. 页面能展示脚本列表和版本列表。
2. 页面支持新增脚本版本。
3. 页面支持对已有版本执行更新版本。
4. 页面上的脚本版本标识与后续计划任务中可选版本保持一致。
5. 更新成功后，页面能够给出明确反馈。

说明：

1. 已发布版本不直接在页面内编辑源码正文。
2. 脚本版本更新后的真实执行验证，以计划任务联动结果为准。

## 7. 工作流编排

浏览器访问：

```text
http://127.0.0.1:5173/workflows
```

预期结果：

1. 页面只负责维护工作流定义，不再承担直接运行工作流的主入口职责。
2. 页面支持创建、查看和删除工作流定义。
3. 页面支持顺序节点、单循环节点和停止节点的基础编排闭环。
4. 页面应明确体现“正式运行入口在计划任务”这一约束。

## 8. 计划任务与计划任务实例

### 8.1 计划任务定义

浏览器访问：

```text
http://127.0.0.1:5173/plans
```

验收步骤：

1. 创建一个脚本型 `once` 计划任务。
2. 创建一个工作流型 `once` 计划任务。
3. 分别点击“立即启动”或等价入口。

预期结果：

1. 页面顶部存在“创建计划任务”与“刷新”按钮。
2. 页面中的正式运行对象是脚本型或工作流型计划任务。
3. 创建成功后，列表中能看到目标类型、目标内容、调度类型、状态和默认设备数。
4. 启动成功后，会创建对应的计划任务实例。

### 8.2 计划任务实例

浏览器访问：

```text
http://127.0.0.1:5173/plan-runs
```

预期结果：

1. 页面能够展示计划任务实例列表。
2. 每条记录至少能看到计划任务实例 ID、计划任务名称、目标类型、设备数、状态、开始时间和结束时间。
3. 页面支持查看设备、查看事件、追加设备和停止实例。
4. “计划任务实例”页是唯一正式运行态入口。
5. 工作流型计划任务启动后，页面仍以计划任务实例视角展示状态，而不是退回历史工作流实例页。
6. 工作流型计划任务执行完成后，实例状态会收敛为 `success` 或 `failed`，不会长期停留在 `running`。

### 8.3 事件与设备查看

预期结果：

1. 查看事件时，脚本型计划任务至少能看到计划任务实例级事件和设备运行级事件。
2. 工作流型计划任务至少能看到 `workflow_session_ack`、`workflow_session_progress`、完成类事件或等价事件。
3. 查看设备时，能够看到每台设备当前状态与当前工作流节点。
4. 当事件很多时，滚动条出现在弹窗内部，而不是整个浏览器页面。
5. 追加设备失败提示应明确说明是哪台设备被哪个 `plan_run` / `plan_device_run` 占用。

## 9. 系统配置

浏览器访问：

```text
http://127.0.0.1:5173/settings
```

预期结果：

1. 页面可查看并维护中心端需要的系统配置项。
2. 与设备发现、Agent 下发相关的关键配置项至少包括 `CENTER_ADB_PATH`、`CENTER_AGENT_ROOT_PATH`、`CENTER_TOOLKIT_PATH`、`CENTER_BASE_URL`。
3. 保存后，页面能给出明确成功或失败反馈。

## 10. 与手机端联动的关键观察点

当页面触发了 Agent 下发、脚本执行或工作流执行后，应同步观察：

1. 设备列表中的在线状态、最近心跳和占用状态。
2. 计划任务实例页中的设备运行状态和事件流。
3. 手机端日志中的注册、心跳、脚本下载、脚本执行和停止行为。

如果页面发起操作后没有得到预期结果，排查顺序如下：

1. 先看当前页面的请求结果和错误提示。
2. 再看 `repos/mobilerpa-center/docs/server/中心服务补充检查手册.md`。
3. 再看 `repos/mobilerpa-agent/docs/手机端手工验收.md`。
