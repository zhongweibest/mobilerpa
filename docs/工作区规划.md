# MobileRPA 工作区规划

> 维护说明：本文档只作为工作区结构入口说明。详细整体方案以 `docs/整体规划方案.md` 为准，当前执行状态以 `docs/项目看板.md` 为准。

## 目标
当前工作区将从“单机 AutoJs 脚本目录”升级为“多仓库协作工作区”，支持以下能力：

- 基于 Go 的中心控制服务
- 基于 Vue 的后台管理前端
- AutoJs 手机端 Agent 与业务脚本
- 独立的真机快速调试链路

## 仓库策略
在同一个本地工作区内，保持两个独立仓库：

1. `repos/mobilerpa-center`
   - 采用 monorepo 形式，包含 Go 后端和 Vue 管理前端。
   - 原因：当前阶段后端与前端会围绕同一套协议、设备模型和发布流程频繁联动，放在同一仓库可以降低协作和变更成本。

2. `repos/mobilerpa-agent`
   - 包含 AutoJs Agent、业务脚本、脚本清单以及本地调试工具。
   - 原因：手机端运行时与脚本逻辑强耦合，且其版本节奏应与中心服务相对独立。

## Git 版本管理规划

当前建议采用“三个 Git 仓库”的结构：

1. 工作区根仓库：`D:\dev\code\mobilerpa`
   - 用于管理工作区级文档、规划、看板、归档、通用说明和安全的配置模板。
   - 建议纳入：`docs/`、根目录 `README.md`、根目录 `.gitignore`、可复用的工作区脚本模板。
   - 不建议纳入：`projects/`、`scripts/`、`.tmp/`、本机真实配置、运行数据库、构建产物、设备私有配置。

2. 中心服务仓库：`repos/mobilerpa-center`
   - 继续作为独立 Git 仓库。
   - 管理 Go 后端、Vue 前端、中心服务模块文档、中心服务验收文档和中心服务工具脚本。

3. Agent 仓库：`repos/mobilerpa-agent`
   - 继续作为独立 Git 仓库。
   - 管理 AutoJs Agent、业务脚本迁移后的正式代码、Agent 模块文档、脚本模板和调试工具。

这种结构的好处：

1. 根目录文档和规划有 Git 兜底，后续移动、归档或删除文档都可以安全追溯。
2. 中心服务与 Agent 保持独立提交历史，避免两个运行端互相污染版本节奏。
3. `projects/`、`scripts/` 继续作为 legacy 源材料，不因临时脚本、历史脚本和本机调试文件污染正式仓库。
4. 后续如果需要统一远端管理，可以用根仓库记录子仓库地址和说明，但暂不建议直接把子仓库内容平铺进根仓库。

### 根仓库建议忽略规则

根目录 `.gitignore` 建议至少包含：

```text
.tmp/
projects/
scripts/
config.ini
choose_device_push.bat

.vscode/
!.vscode/*.example.json

repos/mobilerpa-center/
repos/mobilerpa-agent/
```

说明：

1. `.tmp/` 是运行产物和测试数据库，不进 Git。
2. `projects/`、`scripts/` 是旧脚本源材料，先不进 Git。
3. `config.ini` 和 `choose_device_push.bat` 可能包含本机路径、设备序列号或私人配置，先不进 Git。
4. `.vscode/` 当前包含真实设备序列号，建议后续只提交 `.example.json` 模板。
5. `repos/` 下两个子仓库由各自 Git 管理，根仓库默认忽略它们，避免嵌套仓库状态混乱。

### 是否需要现在初始化根仓库

建议下一步初始化根仓库，但在执行前需要用户明确确认。

推荐初始化范围：

1. 初始化 `D:\dev\code\mobilerpa` 为 Git 仓库。
2. 新增根目录 `.gitignore`。
3. 只提交 `docs/` 和根目录安全说明文件。
4. 暂不提交 `projects/`、`scripts/`、`.tmp/`、`.vscode/`、`config.ini`、`choose_device_push.bat`。
5. 两个子仓库继续分别提交，不混入根仓库提交。

## 为什么前端暂时不单独拆仓库
前端单独拆仓库通常适用于以下情况：

- 前端团队独立运作
- 前后端发布节奏明显不同
- API 协议已经相对稳定

当前还不满足这些前提。现阶段后台 UI、WebSocket 协议、设备元数据、脚本发布流程都会一起快速演进，因此中心仓库采用 monorepo 是更低摩擦的选择。

## Legacy 代码处理原则
当前的 `projects/` 和 `scripts/` 等目录仍视为 legacy 源材料。
它们暂时不删除，后续在新结构稳定后，逐步迁移到 `repos/mobilerpa-agent` 中。

## 文档语言约定
- 当前工作区内的项目文档默认使用中文编写。
- 后续新增的架构、开发、协议、部署、运维文档，默认也使用中文。
- 后续新增的代码注释默认使用中文编写，包括方法注释、结构体注释、字段注释和必要的行内注释。
- 如确有必要引入英文内容，应以中文为主、英文为辅。
