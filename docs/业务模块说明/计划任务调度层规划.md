# 计划任务调度层规划

正式运行主线由计划任务调度域和工作流定义域共同组成：`workflow_def` 负责描述多脚本能力之间的关系模板，`plan_def -> plan_run -> plan_device_run` 负责承载实际调度、实例启动、设备运行记录和停止控制。脚本执行层只负责单设备脚本执行、结果回传和必要的执行记录。

## 1. 计划任务调度层的核心定义

计划任务调度层建立在以下共识之上：

1. 脚本版本是能力单元，能力以 `script_name + script_version` 为唯一执行版本标识。
2. 工作流定义是能力模板，描述多个脚本能力之间的关系与推进规则。
3. 单设备脚本执行是能力实例，是某个脚本能力在某台设备上的一次原子执行。
4. 计划任务是调度主体，决定“什么能力在什么时间、以什么节奏、在哪些设备上执行”。
5. 计划任务运行实例是某次调度真正落地后的运行批次，实例下再挂多台设备运行记录。
6. 设备占用最终应归口到计划任务运行域，而不是继续由历史执行域和工作流实例域各自分散维护。

## 2. 与计划任务调度域和工作流定义域的关系

这一层的核心约束如下：

1. `workflow_def` 保留为能力模板，不承载开始时间、截止时间、次日自动启动等调度属性。
2. `plan_def` 用于描述一条正式调度规则，可指向“单脚本+版本”或“工作流定义”。
3. `plan_run` 用于描述计划任务的一次真实启动批次。
4. `plan_device_run` 用于描述某台设备在该批次中的独立运行状态。
5. 运行态统一从“计划任务实例”入口查看，旧的“工作流实例层”不再作为主入口保留。
6. 正式运行入口统一收敛到计划任务与计划任务实例，调试能力不再单独形成对外运行层语义。

## 3. 能力边界

计划任务调度能力支持：

1. 基于单脚本版本创建计划任务。
2. 基于工作流定义创建计划任务。
3. 计划任务定义时选择目标设备集合。
4. 支持一次性执行。
5. 支持按自然天循环执行。
6. 支持每日开始时间。
7. 支持每日截止时间。
8. 支持手动启用和停用计划任务。
9. 支持手动停止运行实例。
10. 支持运行中添加设备。
11. 支持运行中删除设备。
12. 支持单设备停止和整体停止。
13. 支持设备占用冲突返回明确错误。
14. 支持工作流目标在运行期间继续复用自身的节点、循环和停止规则。

时间规则收口如下：

1. `daily_start_time` 和 `daily_deadline_time` 统一定义在计划任务层，不下沉到工作流定义层。
2. “次日自动启动”不再依赖前一天是不是命中截止时间，而是依赖计划任务自身是否处于启用状态且配置了按天循环。
3. 只要某台设备在前一天的该计划任务运行实例中被真正启动过，那么第二天到达新的 `daily_start_time` 时，都要重新加入新实例，从头开始执行。
4. 不论设备前一天是因为命中截止时间停止、人工删除设备、执行完成、工作流自然结束，还是其他受控停止原因，只要它属于前一天这条计划任务的已启动设备集合，次日都应按新实例重新启动。
5. 如果计划任务被手动停用，则不再自动拉起新实例。
6. 不实现复杂日历、周计划、节假日规则、多时段窗口和节点级时间策略。

不强制支持：

1. 图形化拖拽计划日历。
2. 任意复杂条件表达式引擎。
3. 多计划任务并行抢占同一设备。
4. 跨设备共享变量。
5. 在线热替换已运行中的脚本或工作流目标。

## 4. 计划任务目标模型

计划任务中的目标对象必须明确绑定：

1. `target_type`
2. `target_id`
3. 如为脚本目标，则还需要 `script_name + script_version`
4. 如为工作流目标，则需要 `workflow_def_id`

支持两种目标类型：

1. `script`
   直接调度某个 `script_name + script_version`。
2. `workflow`
   调度某个 `workflow_def_id`。

### 脚本版本引用关系

正式主线下，脚本源码不直接进入计划任务定义或工作流定义，而是统一通过“脚本名称 + 脚本版本”引用脚本版本元数据。

关联链路固定为：

1. `plan_defs.target_script_name` + `plan_defs.target_script_version`
   用于脚本型计划任务定位目标脚本版本。
2. `workflow_nodes.script_name` + `workflow_nodes.script_version`
   用于工作流节点定位目标脚本版本。
3. `script_versions(script_name, version)`
   保存该脚本版本的入口文件、清单、存储路径和状态。

这样设计的好处是：

1. 同一个脚本版本可以被多个计划任务和多个工作流节点复用。
2. 计划任务定义和工作流定义不保存脚本源码正文。
3. 回查某个计划任务或工作流节点引用了哪个脚本版本会很直接。
4. 便于做脚本版本灰度、回滚和审计。

这意味着：

1. 工作流作为被计划任务调度的能力模板存在，而不是最顶层调度实体。
2. 单脚本和工作流在调度入口上统一，不再分别维护两套跨天、循环和设备批量启动逻辑。

## 5. 运行实例与上下文隔离

计划任务定义与运行实例必须分离：

1. `plan_def`
   共享调度规则，描述“应该在什么时候、以什么方式调度什么目标”。
2. `plan_run`
   一次真实启动批次，描述“这条计划任务在某一天某个时刻启动了一次”。
3. `plan_device_run`
   某台设备在该批次中的独立运行记录。

计划任务运行时上下文必须归属于 `plan_run` 或 `plan_device_run`，而不是归属于模板。至少保存：

1. `started_devices_snapshot`
2. `added_devices`
3. `removed_devices`
4. `stop_reason`
5. `last_started_date`
6. `last_completed_date`

若目标类型是工作流，则工作流内部上下文继续保留在工作流执行域；若目标类型是脚本，则可只维护计划任务设备运行上下文。

## 6. 设备选择、占用与运行中动态调整

计划任务启动时必须明确选择目标设备。定义请求至少应包含：

1. `target_type`
2. `target_id`
3. `device_ids[]`

同一个计划任务一次启动多台设备时，本质上是：

1. 校验所有设备
2. 创建一个 `plan_run`
3. 在该 `plan_run` 下为每台可运行设备创建一条 `plan_device_run`

必须坚持统一设备占用约束：

1. 同一设备同一时刻只允许属于一个运行中的计划任务设备运行记录。
2. 任何脚本执行下发动作都必须检查该设备是否已被运行中的计划任务占用。
3. 若设备已被占用，则接口必须返回明确错误，并指出占用它的是哪条计划任务或哪次计划任务运行实例。

计划任务启动后，运行设备集合不应被视为冻结，必须允许两类运行中调整操作：

1. 向计划任务实例添加新设备。
2. 从计划任务实例移除某台设备。

其中要明确：

1. 添加设备后，应立即创建新的 `plan_device_run`，并立即拉起目标执行。
2. 删除设备后，应立即停止该设备执行，不影响同一实例下其他设备继续运行。
3. 添加设备时必须复用与启动时相同的占用校验规则。
4. 已被手动从实例中移除的设备，不再属于该实例的活跃设备集合。

## 7. 停用、停止与次日是否继续的规则

这部分严格区分“停止实例”和“停用计划任务”：

1. 停止某个 `plan_device_run`
   只影响该实例中的该台设备，不影响该计划任务之后是否继续自动启动。
2. 停止某个 `plan_run`
   只停止这次运行实例，不直接改变该计划任务之后是否继续自动启动。
3. 停用某个 `plan_def`
   表示从此不再自动生成新的运行实例。

因此，用于“截至第二天不再启动整个工作流或单脚本计划”的正式方法，应是：

1. 手动停用计划任务。

而不是依赖“手动停止了一次运行实例”去隐式推导计划任务不再自动启动。

## 8. 数据模型

为了避免把调度规则、能力模板和原子执行混在一起，按以下表组织：

1. `plan_defs`
2. `plan_targets`
3. `plan_devices`
4. `plan_runs`
5. `plan_device_runs`
6. `plan_events`
7. `plan_contexts`

含义如下：

1. `plan_defs`
   维护计划任务定义，例如目标类型、循环方式、开始时间、截止时间、启停状态。
2. `plan_targets`
   维护计划任务目标对象，统一映射到脚本或工作流。
3. `plan_devices`
   维护计划任务绑定的设备集合。
4. `plan_runs`
   维护一次真实启动批次，例如某个计划任务在今天 09:00 被调度启动了一次。
5. `plan_device_runs`
   维护单台设备在该批次下的独立执行状态，例如运行状态、是否手动移除、停止原因。
6. `plan_events`
   维护计划任务域的标准化事件，供前端实例页、消息列表和问题排查使用。
7. `plan_contexts`
   维护批次级或设备级上下文，例如昨日已启动设备集合、手动增删设备记录、最近自动启动日期。

工作流定义层只保留能力模板相关字段，不再新增时间策略字段。开始时间和截止时间统一放在计划任务层。

## 9. 执行链路与中心/Agent 职责边界

在小规模环境里，中心服务直接启动工作流或直接下发单设备脚本执行也能工作；在 300 到 500 台设备规模下，必须约束中心与 Agent 的职责边界。

执行链路按以下边界组织：

1. 中心服务负责：
   - 管理脚本版本、工作流定义和计划任务定义。
   - 定时扫描计划任务，生成或停止 `plan_run`。
   - 对设备占用冲突、运行状态、停止控制做统一裁决。
   - 聚合计划任务实例事件、单设备运行事件和异常事件。
2. Agent 负责：
   - 接收单脚本执行指令或工作流执行指令。
   - 在本地执行目标脚本，或推进工作流内部节点。
   - 在关键步骤完成时回传标准化事件。
   - 在需要脚本时按脚本名和版本号拉取或校验本地脚本包。

稳态方向不是“页面直接启动工作流实例”，而是：

1. 页面创建或启用计划任务。
2. 中心按计划任务规则生成 `plan_run`。
3. `plan_run` 再决定拉起脚本目标还是工作流目标。
4. Agent 仅回传关键步骤事件、结果事件、停止事件和异常事件。
5. 中心基于这些关键事件更新 `plan_device_runs`、工作流执行域和消息事件域。

## 10. 计划任务与工作流的典型映射

典型场景应映射为：

1. 工作流定义层负责描述 `A -> B -> CDE 循环 -> FG 循环` 这类能力关系。
2. 计划任务层负责描述“每天 09:00 启动该工作流，23:59 截止，在一组设备上运行”。
3. 单脚本场景则映射为“每天 10:00 调度 `open_wechat@v0.1.0` 到一组设备执行”。

其共同原则是：

1. 能力模板和调度规则必须分离。
2. 单脚本和工作流共享同一套调度机制。
3. 多台设备共享同一条计划任务定义，但运行记录完全隔离。

## 11. 500 台设备规模下的压力评估与约束

按 `heartbeat = 30s` 估算，500 台设备在线时，平均心跳流量约为：

1. `500 / 30 ≈ 16.7 条心跳/秒`

仅从心跳角度看，这个量级对 Go 中心服务是可控的。真正的压力点不在“在线设备数量本身”，而在以下几类场景：

1. 每天大量计划任务同时到达开始时间，需要瞬时拉起多台设备执行。
2. 每台设备每个步骤都强依赖中心逐条下发，导致双向命令频率过高。
3. 后台实时订阅把所有设备、所有计划任务事件无差别广播给所有页面。
4. 查询接口继续把计划任务实例、设备运行明细和工作流明细混在同一张大列表里。

因此在 300 到 500 台设备目标下，必须遵守以下约束：

1. 心跳、计划任务事件、任务结果、工作流关键步骤事件全部走标准化消息体，但按主题拆分消费。
2. WebSocket 写入按设备连接串行化，避免并发写同一连接。
3. 计划任务实例页、设备运行页和工作流事件页必须分层查询，不做一张大列表全承载。
4. 前端订阅必须按主题、页面、条件做过滤，不做全量广播直出。
5. 计划任务层只保留关键事件，不默认上报超细粒度实时流日志。

## 12. 验收目标

人工验收至少覆盖：

1. 创建一条“单脚本+版本”计划任务。
2. 创建一条“工作流定义”计划任务。
3. 计划任务可绑定多台设备。
4. 到达每日开始时间后，会创建一个新的 `plan_run`。
5. 同一 `plan_run` 下可看到多台设备运行记录。
6. 运行中添加设备后，新设备立即开始执行。
7. 运行中删除设备后，该设备立即停止执行。
8. 到达每日截止时间后，仍在运行的设备会被停止。
9. 第二天到达新的开始时间后，会为昨日启动过的设备重新创建新的运行批次。
10. 手动停用计划任务后，次日不再自动启动。
11. 若设备已被其他计划任务占用，则返回明确冲突错误。
12. 运行态统一从计划任务实例页查看，而不是继续从旧工作流实例页查看。

## 13. 当前正式边界

当前正式边界如下：

1. 正式运行入口统一为计划任务与计划任务实例。
2. 工作流定义页只负责维护能力模板，不承担运行态入口职责。
3. 工作流型与脚本型计划任务共用 `plan_def -> plan_run -> plan_device_run` 运行链路。
4. 设备占用统一按 `plan_device_runs.status in (pending, running)` 判定。
5. 更细粒度的实时流日志、运行变量编辑、在线流程图编辑不纳入当前最小闭环范围。
