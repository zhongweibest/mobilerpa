# Agent 黑屏保活与任务亮屏方案

## 目标
- 在不新增原生 Android 宿主项目的前提下，仅基于 AutoJs6 Agent 提升黑屏场景下的在线稳定性。
- 让 Agent 在黑屏后尽量保持 CPU 活跃、Wi-Fi 不休眠、WebSocket 不断线。
- 让 Agent 在收到中心下发的计划任务后，先主动点亮屏幕，再执行依赖界面的自动化任务。
- 不实现自动解锁；默认前提是设备已配置为点亮后可直接进入可操作状态，或无需密码解锁。

## 方案边界
- 本方案可以提升黑屏保活成功率，但不能承诺所有 ROM、所有机型都完全一致。
- 本方案属于“进程存活时尽量保活”，不等于“Agent 进程被系统杀死后一定能自动恢复”。
- 纯 AutoJs6 可实现 `WakeLock`、`WifiLock`、连接保活、任务触发亮屏。
- 纯 AutoJs6 不适合承担强保证的“进程死亡后系统级拉起”，这一点与原生前台服务仍有差距。

## 总体设计

### 设计拆分
方案分为两部分：

1. 黑屏保活
- 通过 `PARTIAL_WAKE_LOCK` 保持 CPU 尽量不休眠。
- 通过 `WifiLock` 保持 Wi-Fi 尽量不休眠。
- 通过 WebSocket watchdog 和自动重连维持中心长连接。

2. 任务到达后主动亮屏
- Agent 收到中心下发的计划任务。
- 先检查当前屏幕是否熄灭。
- 如果熄灭，则主动唤醒屏幕。
- 等待屏幕进入可交互状态后，再执行任务。

## 一、黑屏保活方案

### 1. 保 CPU

#### 目标
- 设备黑屏后，Agent 主线程、心跳线程、重连线程仍能继续运行。
- 避免系统在短时间内让 JS 执行完全挂起。

#### 实现方式
- 在 Agent 启动时通过 Android `PowerManager` 申请 `PARTIAL_WAKE_LOCK`。
- `PARTIAL_WAKE_LOCK` 的目标不是点亮屏幕，而是让 CPU 在灭屏时仍尽量保持运行。
- 锁对象由 Agent 全局生命周期统一管理，只允许持有一把有效锁。

#### 建议策略
- Agent 启动成功后立即申请。
- Agent 正常停止时主动释放。
- 如果出现脚本级重启，先释放旧锁，再重新申请。
- 定时检测锁状态；如果锁丢失，则补申请一次。

#### 说明
- `PARTIAL_WAKE_LOCK` 更适合常驻连接、心跳、重连、轻量调度。
- 不建议频繁重复申请；应做成幂等封装。

### 2. 保 Wi-Fi

#### 目标
- 黑屏后尽量避免 Wi-Fi 进入休眠导致长连接断开。
- 降低“手机灭屏几分钟后掉线”的概率。

#### 实现方式
- 通过 Android `WifiManager` 申请 `WifiLock`。
- 推荐使用高性能或全模式 Wi-Fi 锁，具体取决于 AutoJs6 当前可直接访问的 Android API 常量。
- 同样使用全局单例方式管理，只允许存在一把有效 `WifiLock`。

#### 建议策略
- Agent 建连前或建连成功后立即申请。
- Agent 正常停止时释放。
- 连接断开不立即释放；因为断开后仍需要靠 Wi-Fi 完成自动重连。
- 定时检测锁状态；锁失效则自动补申请。

#### 说明
- `WifiLock` 不能保证网络永不抖动，但能显著减少灭屏休眠导致的断网。
- 若系统开启极端省电策略，仍可能被 ROM 干预，需要配合系统设置。

### 3. WebSocket 保活与 watchdog

#### 目标
- 黑屏状态下继续维持中心连接。
- 若连接断开，可自动恢复。

#### 实现方式
- 保留现有自动重连机制。
- 增加 watchdog 概念，定时检查以下状态：
- WebSocket 是否存在。
- WebSocket 是否处于 open 状态。
- 最近一次心跳发送时间是否超时。
- 最近一次收到中心消息的时间是否超时。

#### 处理策略
- 如果连接对象不存在，则立即进入重连流程。
- 如果连接对象存在但长时间没有心跳/消息，则主动关闭旧连接并重连。
- 任意时刻只允许存在一个连接实例、一个心跳调度器、一个重连调度器。

#### 建议超时
- 心跳周期按当前实现保留。
- watchdog 检查周期建议 `15s` 到 `30s`。
- 消息静默超时建议略大于多个心跳周期，例如 `90s` 到 `180s`。

## 二、收到计划任务后主动亮屏

### 1. 目标
- 中心下发计划任务时，即便设备当前黑屏，Agent 也能先亮屏再执行。
- 不做自动解锁，只负责把屏幕点亮。

### 2. 适用范围
- 适用于依赖 UI 自动化、无障碍点击、页面识别、App 前台操作的任务。
- 对纯后台任务，可不必强制亮屏。

### 3. 实现流程
1. Agent 收到计划任务消息。
2. 判断当前屏幕是否处于熄灭状态。
3. 若已熄灭，则调用亮屏流程。
4. 等待屏幕真正唤醒并稳定。
5. 再进入任务执行逻辑。

### 4. 亮屏方式

#### 建议方式
- 通过 `PowerManager` 获取用于唤醒设备的锁或唤醒能力。
- 调用后短暂持有，确保屏幕点亮。
- 点亮成功后，等待 `500ms` 到 `2000ms` 再继续。

#### 注意点
- 不建议长时间持有“亮屏型”锁，只用于一次任务触发时的短暂唤醒。
- 如果当前屏幕已经点亮，则不重复执行唤醒。
- 对连续到达的多个任务，应做防抖，避免频繁点亮。

### 5. 建议策略开关
- 可在 Agent 侧增加配置项，例如：
- `wake_screen_before_task: true`
- `wake_screen_cooldown_seconds: 30`

#### 含义
- `wake_screen_before_task`
  控制收到计划任务后是否自动亮屏。
- `wake_screen_cooldown_seconds`
  控制两次自动亮屏之间的最小间隔，避免短时间频繁唤醒屏幕。

## 三、模块划分建议

### 1. 新增保活模块
建议新增类似模块：
- `agent/lib/power_keeper.js`

职责：
- 管理 `WakeLock`
- 管理 `WifiLock`
- 提供 `acquireAll()`
- 提供 `releaseAll()`
- 提供 `ensureAll()`
- 提供 `wakeScreenIfNeeded()`

### 2. Agent 主入口接入
建议在：
- `agent/agent.js`

中接入：
- Agent 启动后调用 `acquireAll()`
- Agent 停止前调用 `releaseAll()`
- 在总调度器中定时调用 `ensureAll()`

### 3. WebSocket 客户端接入
建议在：
- `agent/lib/ws_client.js`

中补充：
- 记录最近一次收发消息时间
- 增加 watchdog 定时器
- 在收到计划任务时，先调用 `wakeScreenIfNeeded()` 再执行任务

## 四、配置项建议

### Agent 配置建议新增字段
```json
{
  "keep_cpu_awake": true,
  "keep_wifi_awake": true,
  "wake_screen_before_task": true,
  "wake_screen_cooldown_seconds": 30,
  "ws_watchdog_interval_seconds": 20,
  "ws_silence_timeout_seconds": 120
}
```

### 字段说明
- `keep_cpu_awake`
  是否启用 CPU 保活。
- `keep_wifi_awake`
  是否启用 Wi-Fi 保活。
- `wake_screen_before_task`
  是否在收到计划任务时自动亮屏。
- `wake_screen_cooldown_seconds`
  自动亮屏防抖窗口。
- `ws_watchdog_interval_seconds`
  watchdog 巡检周期。
- `ws_silence_timeout_seconds`
  长连接静默超时阈值。

## 五、执行时序建议

### Agent 启动
1. 读取配置。
2. 初始化 `power_keeper`。
3. 申请 `WakeLock`。
4. 申请 `WifiLock`。
5. 建立 WebSocket 连接。
6. 启动心跳。
7. 启动 watchdog。

### 收到计划任务
1. 判断是否启用“任务前亮屏”。
2. 若启用，则判断当前是否灭屏。
3. 若灭屏，则执行亮屏。
4. 等待短暂稳定时间。
5. 启动实际任务。

### Agent 停止
1. 停止任务调度器。
2. 停止心跳。
3. 停止 watchdog。
4. 关闭 WebSocket。
5. 释放 `WifiLock`。
6. 释放 `WakeLock`。

## 六、风险与限制

### 1. ROM 差异
- 不同厂商对后台保活限制不同。
- 某些 ROM 即使持有锁，也可能在极端省电模式下中断脚本。

### 2. 权限与系统设置
- 需要尽量关闭 AutoJs6 的电池优化。
- 建议加入自启动白名单、后台无限制、省电策略白名单。
- 若 Wi-Fi 在系统层被强制断开，`WifiLock` 也不能完全兜底。

### 3. 不保证进程死亡自动恢复
- 如果 AutoJs6 进程被系统真正杀死，纯脚本方案很难像原生前台服务一样稳定自恢复。
- 因此本方案重点是“降低黑屏掉线概率”，不是“替代 Android 宿主服务”。

### 4. 亮屏不等于解锁
- 本方案默认只负责点亮屏幕。
- 如果设备仍有锁屏层且无法直接操作，则任务可能仍无法继续执行。

## 七、建议验收场景

### 黑屏保活验收
- 屏幕熄灭 `10` 分钟后，中心仍显示设备在线。
- 屏幕熄灭 `30` 分钟后，Agent 心跳仍持续上报。
- 黑屏期间断网再恢复，WebSocket 能自动重连。

### 任务亮屏验收
- 设备黑屏时由中心下发计划任务，设备应先亮屏再执行。
- 若设备原本已亮屏，不应重复明显闪屏或反复唤醒。
- 连续短时间下发多个任务，应符合亮屏防抖预期。

### 异常场景验收
- Agent 停止后，锁应正常释放。
- Agent 重启后，不应出现重复锁对象或重复 watchdog。
- 网络异常恢复后，连接应能自动回到稳定状态。

## 八、落地建议

### 第一阶段
- 先实现 `WakeLock + WifiLock + watchdog`。
- 验证黑屏 `10` 到 `30` 分钟的在线稳定性。

### 第二阶段
- 再实现“收到计划任务后主动亮屏”。
- 重点验证 UI 自动化任务在黑屏触发下的实际成功率。

### 第三阶段
- 根据真机表现微调：
- watchdog 周期
- 静默超时
- 亮屏等待时长
- 亮屏防抖时长

## 九、结论
- 只基于 AutoJs6，实现“黑屏后尽量保活 + 收到计划任务后主动亮屏”是可行的。
- 这条路线适合当前项目阶段，改动集中在 Agent 现有代码内，不需要新增 Android 宿主工程。
- 该方案能明显提升黑屏场景下的可用性，但仍应明确其上限是“增强保活”，不是“系统级强保活”。
