# Agent 运行时最小契约

本文定义 Agent 与业务脚本之间的最小协作边界，便于 Agent、脚本和中心端对齐。

## 1. 任务载荷最小字段

Agent 在启动脚本前，传入的上下文至少应包含以下字段：

```json
{
  "task_id": "task_20260607_001",
  "device_id": "android_lab_01",
  "mode": "debug",
  "script_name": "shoppe",
  "script_version": "dev",
  "trace_id": "trace_20260607_001",
  "params": {
    "operator_id": "demo"
  },
  "timeouts": {
    "task_timeout_ms": 600000,
    "step_timeout_ms": 60000
  }
}
```

字段说明：
- `task_id`：任务唯一标识
- `device_id`：执行设备唯一标识
- `mode`：`debug`、`beta`、`prod` 三种模式之一
- `script_name`：脚本目录名
- `script_version`：脚本版本号或环境版本别名
- `trace_id`：跨日志链路跟踪 ID
- `params`：业务参数
- `timeouts`：执行时限

补充约束：

- 同一时刻 Agent 只执行一个主任务链路，不支持并发拉起多个主业务脚本。
- `script_name` 和 `script_version` 是任务与脚本缓存目录的直接映射键。
- Agent 在真正执行前，必须先确认本地存在匹配版本的脚本目录和 `manifest.json`。

## 2. 脚本入口约定

业务脚本入口文件需要导出统一接口：

```javascript
module.exports = {
    run(context) {
        return {
            success: true,
            code: "OK",
            message: "done",
            data: {}
        };
    }
};
```

约束说明：
- `run(context)` 必须是 Agent 和本地调试共用的同一个核心入口
- 返回值必须可序列化
- 不允许把平台连接、设备注册等逻辑写入业务脚本入口

## 3. 脚本执行结果最小字段

脚本执行结束后，Agent 至少需要拿到以下结果结构：

```json
{
  "success": true,
  "code": "OK",
  "message": "task finished",
  "step": "submit_order",
  "data": {
    "order_id": "demo"
  },
  "artifacts": {
    "screenshots": [],
    "dumps": []
  }
}
```

字段说明：
- `success`：任务是否成功
- `code`：机器可识别的结果码
- `message`：供人查看的摘要
- `step`：最后执行到的关键步骤名
- `data`：业务结果
- `artifacts`：截图、dump 等调试证据

## 4. 结果码约定

- `OK`：成功
- `RETRYABLE_NETWORK`：网络波动，可重试
- `RETRYABLE_UI`：页面状态异常，可重试
- `FATAL_SCRIPT`：脚本逻辑错误，不应自动重试
- `FATAL_ENV`：设备或权限问题，不应自动重试
- `CANCELLED`：任务被取消
- `TIMEOUT`：任务超时

## 5. 日志最小规范

每条关键日志包含：

```json
{
  "ts": "2026-06-07T20:00:00+08:00",
  "level": "INFO",
  "task_id": "task_20260607_001",
  "trace_id": "trace_20260607_001",
  "script_name": "shoppe",
  "script_version": "dev",
  "step": "open_home",
  "message": "page opened"
}
```

最低要求：
- 每个关键步骤至少一条开始或结束日志
- 失败时必须记录失败步骤、结果码、摘要
- `debug` 模式允许附带更详细上下文

## 6. 本地缓存布局

```text
agent/
scripts/
  <script_name>/
    versions/
      dev/
      beta/
      prod/
runtime/
  logs/
  screenshots/
  dumps/
```

至少要保证：
- 脚本版本目录和运行产物目录分离
- 日志、截图、dump 可按任务 ID 检索

固定目录如下：

```text
/sdcard/mobilerpa/
  agent/
    agent.js
    runtime/
      config.json
      bootstrap.json
  scripts/
    <script_name>/
      <script_version>/
        manifest.json
        main.js
        ...
  temp/
    downloads/
```

下载与执行约束：

1. `assign_task` 到达后，先根据 `script_name`、`script_version` 检查本地缓存目录。
2. 如果本地缺失该版本，先请求 `manifest` 并准备下载目录文件。
3. 按 `files` 清单逐个下载到目标版本目录或临时下载目录。
4. 按文件级摘要与 `manifest` 元数据校验。
5. 执行入口以 `manifest.json.entry_file` 为准，不写死某个固定文件名。
6. 执行完成后保留脚本缓存，相同版本任务直接复用。

Agent 本地配置补充约束：

1. `runtime/config.json` 属于真机本地运行态文件，不通过推送脚本直接覆盖。
2. 推送脚本每次推送都应刷新 `runtime/bootstrap.json`，用于同步最新的 `center_base_url` 和连接参数。
3. `agent.js` 启动时如果发现缺少 `runtime/config.json`，必须自动读取 `runtime/bootstrap.json` 生成正式配置。
4. 如果 `runtime/config.json` 已存在，优先复用其中保存的 `agent_uuid`、`device_id` 等本地身份配置，但仍要允许 `bootstrap.json` 覆盖 `center_base_url` 和连接参数。

与中心约定的脚本分发协议如下：

### 6.1 元数据查询

请求：

```text
GET /api/v1/script/manifest?script_name=<script_name>&script_version=<script_version>
```

预期最小响应字段：

- `script_name`
- `script_version`
- `entry_file`
- `files`
- `compatible_agent`

### 6.2 包下载

请求：

```text
GET /api/v1/script/download?script_name=<script_name>&script_version=<script_version>&relative_path=<path>
```

Agent 处理步骤：

1. 调用 `manifest` 接口确认元数据。
2. 检查 `compatible_agent` 是否满足 Agent 版本。
3. 根据 `files` 清单逐个下载目标文件。
4. 按文件级摘要和清单元数据校验。
5. 读取本地 `manifest.json` 再做一次字段确认。
6. 使用 `entry_file` 加载并执行。

### 6.3 执行安全边界

- 不允许任务消息直接带本地绝对路径作为执行入口。
- 不允许跳过 `manifest` 校验后直接执行未知脚本文件。
- 不允许同一时刻并发下载并拉起多个主业务脚本。

## 7. 发布前必查项

- Agent 版本与 `compatible_agent` 范围匹配
- 目标脚本 `manifest.json` 字段完整
- 调试入口和生产入口共用同一核心 `src/`
- 至少完成 1 条 Happy Path 和 3 条异常路径验证

## 8. 快速上线检查项

### 8.1 最小执行链路

- 设备上可以启动 Agent 或等价调试入口。
- Agent 与脚本之间的任务载荷、脚本选择和结果回传字段冻结。
- 至少有一个可本地调试、也可被 Agent 调用的脚本样板。
- 真机本地调试流程可复现：推包、启动、看日志、拉截图。
- 开发人员按文档应能在 30 分钟内完成首次真机联调。

### 8.2 稳定执行与失败证据链

- Agent 单任务状态机至少覆盖：待执行、执行中、成功、失败、超时、取消。
- 同一业务脚本可以区分 `dev`、`beta`、`prod` 版本。
- 脚本下载或更新后的本地缓存布局明确。
- 失败时必须能定位到脚本版本、步骤名和失败摘要。
- 失败证据链至少包含日志、截图、页面 dump 中的两类。
- 至少准备 5 个核心异常场景的真机回归用例。

### 8.3 试运行观测与人工兜底

- 试运行前冻结上线检查项：设备在线率、脚本版本、权限、网络、电量、存储空间。
- 任务级指标至少包含：领取成功率、启动成功率、执行成功率、平均耗时、超时率。
- 明确人工介入条件：连续失败、页面变化、账号风控、系统弹窗。
- 失败原因应能归类到脚本、页面、设备、网络中的至少一类。
- 发生异常时应能在 10 分钟内决定继续灰度、暂停或回滚。

### 8.4 灰度发布与回滚

- 支持按设备组逐步放量，避免全量瞬时切换。
- 上线当天按小时检查成功率、超时率和异常分布。
- 快速回滚方案至少包含：停发新任务、回退脚本版本、切换设备组。
- 高频失败问题需要执行热修复、暂停或降级策略。
- 首周结束后沉淀已知限制、人工处理方法、版本发布记录和技术债清单。

### 8.5 跨周期持续项

- 维护真机资源池：设备、电量、网络、ADB 权限持续可用。
- 持续更新高频页面截图基线，用于发现页面变化。
- 明确测试账号、生产账号、验证码或风控处理流程。
- 持续同步调试说明、发布记录和已知问题。

## 9. 明确不在本契约内的内容

以下内容不属于本契约范围：
- 多任务并发调度
- 完整脚本包签名与校验
- 复杂权限编排
- 平台侧租户、账号、配额策略
