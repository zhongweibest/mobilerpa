# Agent 与脚本架构

## 1. 仓库边界

手机端运行能力与业务脚本能力已经拆分为两个独立仓库：

1. `repos/mobilerpa-agent`
   负责手机端 Agent 运行时、连接协议、计划任务实例下发接收、脚本下载、脚本拉起、结果回传以及相关验收文档。
2. `repos/mobilerpa-script`
   负责业务脚本 TypeScript 源码工程、共享 helper、调试入口和最终发布产物。

另外，设备发现、Agent 推送、启动与停止能力统一放在：

3. `repos/mobilerpa-toolkit`

## 2. Agent 仓库结构

- `js-agent/`
  JavaScript 回滚基线。
- `ts-agent/`
  TypeScript 正式维护主线。
- `ts-agent/release/`
  正式发布目录，供 `mobilerpa-toolkit` 推送到手机端。
- `docs/`
  运行架构、运行契约、开发约定与验收文档。

## 3. 脚本仓库结构

- `packages/<script-package>/`
  业务脚本源码工程。
- `packages/shared/`
  公共运行时 helper、类型、取消检查、应用启动等共享能力。
- `publish/<script_name>/<version>/`
  发布产物目录，供中心端上传、下发和真机调试使用。

手机端和 AutoJs6 始终只运行构建后的 JavaScript，不直接运行 TypeScript 源码。

## 4. 运行分层

1. Agent 层
   负责设备注册、稳定 `agent_uuid` 生成、WebSocket 长连接、心跳、断线重连、计划任务实例指令接收、脚本版本检查、脚本下载与执行结果回传。
2. 脚本层
   负责具体自动化执行逻辑，例如页面识别、点击输入、OCR、弹窗处理和业务状态推进。
3. 工具层
   负责设备发现、Agent 推送、Agent 启动、Agent 停止等运维动作。

## 5. 运行模式

1. 本地快速调试模式
   通过脚本版本目录下的 `index_debug.js` 在 AutoJs6 中直接启动，不依赖中心端和 Agent 下发。
2. 平台联调模式
   由中心端通过计划任务实例下发运行指令，Agent 检查脚本版本、下载脚本并拉起执行。
3. 生产模式
   保持与联调模式相同的运行边界，只是在脚本版本、中心地址和运维方式上使用正式配置。

三种模式必须共享同一套脚本核心逻辑，只在入口文件和运行时装配方式上有所区别。

## 6. 发布关系

1. Agent 正式发布源默认使用 `repos/mobilerpa-agent/ts-agent/release/`。
2. Agent 回滚时可切回 `repos/mobilerpa-agent/js-agent/`。
3. 业务脚本通过 `repos/mobilerpa-script` 统一构建后，发布到各自的 `publish/<script_name>/<version>/` 目录。
4. 中心端与手机端始终消费构建产物，不直接消费源码工程。

## 7. Agent 与脚本最小运行契约

Agent 在启动脚本前，传入的上下文至少包含以下字段：

```json
{
  "plan_run_id": "plan_run_20260607_001",
  "plan_device_run_id": "plan_device_run_20260607_001",
  "device_id": "android_lab_01",
  "mode": "debug",
  "script_name": "shoppe",
  "script_version": "dev",
  "trace_id": "trace_20260607_001",
  "params": {
    "operator_id": "demo"
  },
  "timeouts": {
    "task_timeout_ms": 600000,
    "step_timeout_ms": 60000
  }
}
```

补充约束：

1. 同一时刻 Agent 只执行一个计划任务设备运行链路。
2. `script_name` 和 `script_version` 是计划任务设备运行与脚本缓存目录的直接映射键。
3. Agent 在真正执行前，必须先确认本地存在匹配版本的脚本目录和 `manifest.json`。

业务脚本入口统一导出：

```javascript
module.exports = {
    run(context) {
        return {
            success: true,
            code: "OK",
            message: "done",
            data: {}
        };
    }
};
```

执行结果至少返回：

```json
{
  "success": true,
  "code": "OK",
  "message": "task finished",
  "step": "submit_order",
  "data": {
    "order_id": "demo"
  },
  "artifacts": {
    "screenshots": [],
    "dumps": []
  }
}
```

统一结果码包括：

1. `OK`
2. `RETRYABLE_NETWORK`
3. `RETRYABLE_UI`
4. `FATAL_SCRIPT`
5. `FATAL_ENV`
6. `CANCELLED`
7. `TIMEOUT`

业务脚本不承担平台注册、长连接和平台通信逻辑，只负责自动化执行。关键日志至少包含时间、级别、`plan_run_id`、`plan_device_run_id`、`trace_id`、`script_name`、`script_version`、步骤名和消息正文。
