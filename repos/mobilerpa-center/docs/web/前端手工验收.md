# MobileRPA Center Web 手工验收

## 1. 目的

本文档用于手工验收当前管理后台前端的页面结构、设备列表入口、任务中心入口和基础联调能力。

## 2. 前置条件

1. 中心服务已经启动
2. 中心服务接口可访问
3. 当前目录位于 `repos/mobilerpa-center/web`
4. 本机已经安装 Node.js

## 3. 启动方式

如果前端依赖尚未安装，先执行：

```powershell
npm install
```

本地联调默认读取：

```text
web/.env.development
```

如需修改代理目标地址，直接编辑其中的 `VITE_API_PROXY_TARGET`。

启动前端：

```powershell
npm run dev -- --host 0.0.0.0 --port 5173
```

预期结果：

1. Vite 启动成功
2. 浏览器可访问 `http://127.0.0.1:5173`
3. 页面访问 `/api` 时通过 Vite 代理转发到中心服务

## 4. 验收步骤

### 4.1 准备中心服务数据

如果当前没有设备数据，可先注册一台测试设备：

```powershell
$body = @{
  agent_uuid  = "agent-web-001"
  device_name = "Web Device"
  brand       = "Google"
  model       = "Pixel 8"
  android_id  = "android-web-001"
  adb_serial  = "adb-web-001"
} | ConvertTo-Json

Invoke-RestMethod -Uri "http://127.0.0.1:18080/api/v1/device/register" `
  -Method Post `
  -ContentType "application/json" `
  -Body $body
```

### 4.2 验证后台壳层与左侧菜单

浏览器访问：

```text
http://127.0.0.1:5173
```

预期结果：

1. 左侧存在功能菜单区域
2. 菜单中至少可以看到：
   `首页看板`
   `设备列表`
   `设备发现`
   `设备绑定`
   `任务中心`
   `脚本版本`
   `异常中心`
   `日报中心`
   `消息列表`
3. 首次进入默认打开“首页看板”
4. 首页正文提示当前看板内容暂未实现

### 4.3 打开设备页

浏览器访问：

```text
http://127.0.0.1:5173/devices
```

预期结果：

1. 页面标题区域显示“设备总览”或等价标题
2. 页面存在“刷新设备”按钮
3. 页面能看到设备表格或空状态提示

### 4.4 验证设备列表接口接入

如果中心服务已有设备数据，预期页面表格中能看到：

1. 设备名称或 `device_id`
2. 在线状态
3. 绑定状态
4. 最近心跳
5. 当前任务
6. 物理位置
7. 删除按钮

### 4.5 验证状态字段展示

预期展示规则：

1. `status=online` 时显示“在线”标签
2. `status=offline` 时显示“离线”标签
3. `bind_status=pending` 时显示“待绑定”标签
4. `last_heartbeat_at` 为空时显示“暂无”
5. `physical_slot` 为空时显示“未录入”

### 4.6 验证刷新按钮

点击页面右上角“刷新设备”按钮。

预期结果：

1. 按钮进入短暂刷新状态
2. 页面重新请求设备列表接口
3. 表格数据刷新后仍正常显示

### 4.7 验证删除离线设备

先确保列表中至少有一台 `status=offline` 的设备。

点击该设备所在行的“删除”按钮，并在浏览器确认框中点击“确定”。

预期结果：

1. 页面弹出确认提示，只允许删除离线设备
2. 点击确认后，按钮短暂显示“删除中...”
3. 删除成功后，该设备立即从当前列表中消失
4. 再点击“刷新设备”后，被删除设备不会重新出现

### 4.8 验证在线设备不能删除

如果列表中有一台 `status=online` 的设备，预期结果：

1. 该行“删除”按钮为禁用状态
2. 不会因为误点而触发删除请求

### 4.9 验收任务中心页面

浏览器访问：

```text
http://127.0.0.1:5173/tasks
```

预期结果：

1. 左侧菜单中存在“任务中心”
2. 页面顶部存在“刷新任务”按钮
3. 页面中存在“创建任务”表单
4. 页面下方存在任务列表表格

### 4.10 页面创建任务

先确保至少有一台设备已经注册。

在页面中填写：

1. 目标设备：选择一个 `device_id`
2. 脚本名称：`shoppe_sync`
3. 脚本版本：`v0.1.0`
4. 优先级：`1`
5. 任务参数 JSON：

```json
{
  "shop_id": "shop-demo-001",
  "sleep_ms": 1000,
  "should_fail": false
}
```

点击“创建任务”。

预期结果：

1. 页面不报错
2. 任务列表中新增一条 `pending` 任务
3. 该任务显示正确的 `device_id`、脚本名和脚本版本

### 4.11 页面下发任务

在任务列表中找到刚创建的 `pending` 任务，点击“下发任务”。

预期结果：

1. 按钮短暂进入“下发中...”状态
2. 下发成功后任务状态更新
3. 如果 Agent 已连接并执行完成，后续刷新后可以看到 `success` 或 `failed`

### 4.12 页面查看任务事件

点击某条任务后的“查看事件”。

预期结果：

1. 页面下方出现该任务的事件列表
2. 至少可看到：
   `task_created`
   `task_assigned`
3. 如果 Agent 已完成回执，还可以看到：
   `task_ack`
   `task_running`
   `task_progress`
   `task_result`
4. 如果存在 `task_progress` 事件，弹窗中可以直接看到步骤名和中文说明，例如 `CHECK_HOME` 与 `任务执行中：正在检查首页状态`

### 4.13 与脚本自动下载联动的验收入口

如果你要从页面验证“脚本缺失时自动下载”，推荐顺序如下：

1. 先按手机端手工验收文档删除真机上的本地脚本
2. 再回到 `/tasks` 页面创建并下发 `shoppe_sync@v0.1.0`
3. 观察真机日志与任务事件是否显示下载后继续执行

对应手机端验收入口见：

```text
repos/mobilerpa-agent/docs/手机端手工验收.md
```

## 5. 常见问题

### 5.1 页面打开后为空白

先检查前端开发服务是否启动成功，再检查浏览器控制台是否有依赖或编译错误。

### 5.2 左侧菜单没有显示

先确认当前代码已包含新的 `AppShell` 菜单布局，再确认浏览器访问的是最新启动的前端服务。

### 5.3 页面提示加载失败

先确认中心服务是否已经启动，再确认 `web/.env.development` 中的 `VITE_API_PROXY_TARGET` 是否指向正确地址。

### 5.4 设备表格为空

先调用设备注册接口准备一台测试设备，再点击“刷新设备”。

### 5.5 删除失败

先确认目标设备已经是 `offline`。

如果仍失败，优先检查：

1. 中心服务是否已更新到包含 `DELETE /api/v1/devices/{deviceId}` 的版本
2. 浏览器网络面板里该请求是否返回 `409 device_online_cannot_delete`
3. 当前页面是否仍连接到旧的中心服务地址

## 6. 本轮新增页面验收补充

本节用于补充当前版本新增的设备占用查看、人工结束手工任务、工作流实例页与追加设备提示。若前文与本节冲突，以本节为准。

### 6.1 设备列表中的“查看任务”按钮

浏览器访问：
```text
http://127.0.0.1:5173/devices
```

操作步骤：
1. 找到任意一台设备所在行。
2. 点击“查看任务”。

预期结果：
1. 弹出设备占用详情模态窗口。
2. 如果设备空闲，应提示当前未被任务或工作流占用。
3. 如果设备被手工任务占用，应显示 `task_id`、任务状态和占用说明。
4. 如果设备被工作流占用，应显示 `workflow_instance_id` 和占用说明。

### 6.2 设备占用中的“人工结束手工任务”

前提：目标设备当前正被手工任务占用。

操作步骤：
1. 在设备列表点击“查看任务”。
2. 在占用详情弹窗中点击“人工结束手工任务”。
3. 确认弹窗中点击确认。

预期结果：
1. 页面提示结束成功。
2. 再次打开同一设备的占用详情时，应显示设备已释放或不再被该任务占用。
3. 任务中心中对应任务状态应更新为失败，并带有人工结束语义。

### 6.3 任务中心创建/下发任务的占用提示

浏览器访问：
```text
http://127.0.0.1:5173/tasks
```

操作步骤：
1. 选择一台已被其他手工任务或工作流占用的设备。
2. 尝试创建任务，或对该设备关联的待下发任务执行“下发任务”。

预期结果：
1. 页面提示不再是泛化失败信息。
2. 页面应明确提示是哪台设备被哪个手工任务或哪个工作流实例占用。

### 6.4 工作流实例页主视图

浏览器访问：
```text
http://127.0.0.1:5173/workflow-instances
```

预期结果：
1. 页面按工作流实例维度展示列表，而不是按工作流定义维度展示。
2. 默认按最近更新时间倒序排列。
3. 列表底部存在分页器，可切换每页 `10/20/30/50/100`。

### 6.5 工作流实例页查看设备与事件

操作步骤：
1. 在任一实例行点击“查看设备”。
2. 在任一设备运行记录上点击“查看事件”。

预期结果：
1. 设备列表以模态窗口展示。
2. 设备事件也以模态窗口展示。
3. 当事件很多时，滚动条应出现在模态窗口内部，而不是整个浏览器页面。
4. 事件弹窗支持事件类型筛选和手动刷新。

### 6.6 工作流实例页追加设备

操作步骤：
1. 在运行中的实例行点击“追加设备”。
2. 选择一台当前空闲且在线的设备并确认追加。

预期结果：
1. 追加成功时弹出成功提示。
2. 设备列表重新加载后，目标实例下新增该设备运行记录。

### 6.7 工作流实例页追加设备失败提示

操作步骤：
1. 在运行中的实例行点击“追加设备”。
2. 选择一台已被其他手工任务或工作流占用的设备并确认追加。

预期结果：
1. 页面提示应明确说明是哪台设备被哪个任务或哪个工作流实例占用。
2. 不应只显示笼统的“操作失败”。
## 7. 计划任务主线页面验收

### 7.1 工作流编排页职责收口

验收目标：

1. “工作流编排”页只负责工作流定义维护。
2. 正式运行入口统一收口到“计划任务”和“计划任务实例”。

验收点：

1. 页面应明确提示“工作流编排页只负责维护工作流定义”。
2. 页面不再承担直接运行工作流的主入口职责。
3. 页面重点保留的操作应是：
   - 创建工作流定义
   - 查看工作流定义列表
   - 删除工作流定义

### 7.2 计划任务页主视图

验收点：

1. 页面能够展示计划任务定义列表。
2. 每条记录至少能看到：
   - 计划任务名称
   - 目标类型
   - 目标内容
   - 调度类型
   - 状态
   - 默认设备数
3. 页面支持“创建计划任务”和“启动计划任务”。

### 7.3 计划任务实例页主视图

验收点：

1. 页面能够展示计划任务实例列表。
2. 每条记录至少能看到：
   - 计划任务实例 ID
   - 计划任务名称
   - 目标类型
   - 设备数
   - 状态
   - 开始时间
   - 结束时间
3. 页面支持查看设备、查看事件、追加设备和停止实例。

### 7.4 工作流型计划任务的前端验收要求

验收点：

1. 当计划任务定义 `target_type = workflow` 时，页面仍应以“计划任务”视角展示，而不是退回到工作流实例页作为主入口。
2. 当工作流型计划任务启动后，“计划任务实例”页应能正确展示对应实例状态。
3. 工作流型计划任务执行完成后，前端看到的计划任务实例状态应自动收敛为 `success` 或 `failed`，不能长期停留在 `running`。
