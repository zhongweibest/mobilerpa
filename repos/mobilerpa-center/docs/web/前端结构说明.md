# MobileRPA Center Web 最小结构说明

## 1. 当前现状

`web/` 目录当前已经形成第一版可开工工程骨架，包含：

- `Vue 3 + Vite + TypeScript` 基础工程文件
- `src/router/` 路由入口
- `src/layouts/` 后台骨架布局
- `src/api/` 接口请求层
- `src/stores/` Pinia 状态层
- `src/pages/home/` 主页看板占位页
- `src/pages/devices/` 设备列表页入口
- `src/pages/common/` 通用未实现占位页
- `src/styles/` 页面全局样式

为降低并行开发冲突，前端建议先在 `web/` 内独立维护自己的起盘文档，不改动上层总览文档。

## 2. 建议目录

当前第一版最小结构如下：

```text
web/
  README.md
  docs/
    前端结构说明.md
    archive/
      前端快速上线清单.md
  src/
    api/
    layouts/
    pages/
      home/
      common/
      devices/
    router/
    stores/
    styles/
    types/
    utils/
  index.html
  package.json
  tsconfig.json
  vite.config.ts
```

后续按任务推进继续补充：

- `pages/tasks/`
- `pages/workflows/`
- `pages/bindings/`
- `components/`
- `public/`

## 3. 页面范围

4 周快速上线版建议只做以下页面：

- 设备列表页
- 设备绑定页或绑定操作入口
- 任务列表页
- 工作流编排页
- 任务创建页或任务创建弹窗
- 任务详情页
- 日志与截图查看入口

可以先保留但不做完整实现的页面：

- 脚本版本管理
- 异常中心
- 日报中心

## 4. 模块职责

- `api/`: 只放接口请求和 DTO 转换，不放页面状态
- `pages/`: 按业务页面拆分，避免把所有页面堆到一个目录
- `stores/`: 只放跨页面共享状态，如用户信息、筛选缓存、全局字典
- `types/`: 放设备、任务、日志、截图等核心类型
- `utils/`: 放时间格式化、状态映射、查询参数处理等通用逻辑

## 5. 联调约定

为了减少前后端互相阻塞，建议第一个版本就明确：

- 列表字段缺失时以前端降级展示为主，不阻塞页面骨架联调
- 状态枚举统一在 `types/` 和 `utils/` 中集中映射
- 所有时间字段统一约定时区与格式
- 所有空值字段统一约定默认展示文案

## 6. 起盘建议

前端真正开工时，优先顺序建议是：

1. 基础工程和布局
2. 设备列表页
3. 任务列表和创建
4. 任务详情、日志、截图
5. 绑定页

不要在首周同时展开版本管理、异常中心、报表中心，否则很容易把 4 周快速上线版拖成半成品。

## 7. 快速上线边界

四周快速上线版前端优先满足：

1. 能看到设备状态。
2. 能发起和查看基础任务。
3. 能查看日志、结果和截图等排障信息。
4. 能支撑 10 到 30 台设备的小规模试运行。

当前后台壳层补充约定：

1. 左侧功能菜单作为统一导航入口长期保留。
2. 首页默认保留为“主页看板”占位页，先不堆砌假数据。
3. 设备列表通过左侧菜单进入，不再直接占用首页路由。
4. 已规划但未实现的页面允许先展示菜单入口，并在页面内明确提示“未实现”。

当前阶段暂不追求：

1. 完整权限体系。
2. 完整报表中心。
3. 完整异常中心。
4. 大规模灰度发布能力。
5. 复杂交互体验打磨。

## 8. 四周交付节奏

### 第 1 周：起盘与联调骨架

- 初始化 Vue 3、Vite、TypeScript 管理后台工程。
- 建立基础布局、路由、状态管理和接口占位层。
- 冻结设备列表和任务列表的最小字段。
- 完成设备列表页首版骨架并接入真实 `GET /api/v1/devices`。

### 第 2 周：设备页与任务页最小闭环

- 完成设备列表页和设备详情入口。
- 展示在线状态、最后心跳、当前任务、绑定状态、物理槽位等字段。
- 完成任务列表页、单任务创建和下发表单。
- 建立统一错误提示、空状态和加载状态。

### 第 3 周：排障链路与试运行支撑

- 完成任务详情页或任务结果侧栏。
- 接入基础日志查看和截图预览。
- 完成待绑定设备页或绑定操作入口。
- 支持按设备、状态、时间进行基础筛选。

### 第 4 周：上线收口与可操作性补强

- 补齐刷新、筛选保留、最近操作提示和失败重试提示。
- 补齐环境变量、构建命令、部署产物说明和反向代理要求。
- 输出最小操作手册：如何看设备、如何发任务、如何定位失败。
- 明确首版不做的内容并保留占位。

## 9. 对后端的最小依赖

前端最晚应在第一个开发周期内拿到以下稳定内容：

1. 设备列表接口与字段定义。
2. 任务列表接口与字段定义。
3. 创建任务接口入参。
4. 任务状态枚举。
5. 设备在线状态判定规则。
6. 日志与截图查看接口的最小返回结构。
7. 工作流定义、启动、运行实例查询与停止接口。

如果这些字段在第二个开发周期仍频繁变化，前端页面会反复返工，直接影响快速上线节奏。

## 10. 当前工作流与计划任务入口约定

- “工作流编排”菜单只负责维护工作流定义，不再作为正式运行态主入口。
- 正式运行入口统一收口到“计划任务”和“计划任务实例”两个页面。
- 如果需要运行一个工作流，应先在“计划任务”页创建 `target_type=workflow` 的计划任务，再在“计划任务实例”页查看运行状态、事件、追加设备和移除设备。
- `schedule_type=once` 的前端语义固定为“一次性（手工启动）”，不表示指定年月日时分秒自动执行。
- `schedule_type=daily` 的前端语义固定为“按天循环（自动调度）”；页面中的“立即启动”只表示人工立刻拉起一个实例，不改变后续 daily 自动调度行为。
- “工作流实例”页面当前只保留为历史兼容查看页，不再作为正式运行入口。
- 工作流后端接口和计划任务后端接口的手工验收统一收口到 `docs/server/中心服务手工验收.md`。
