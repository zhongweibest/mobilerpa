# 前端开发约定

## 1. 目标

本文档用于约定中心服务前端的本地启动方式、环境配置方式、接口调用规则和日常开发边界，避免每次都依赖手工设置 PowerShell 环境变量。

## 2. 配置原则

1. 前端开发环境默认配置放在 `web/.env.development`。
2. 前端生产构建配置放在 `web/.env.production`。
3. 开发环境默认通过 Vite 代理处理 `/api` 和 `/ws`，代理目标由 `VITE_API_PROXY_TARGET` 指定。
4. 生产环境通过 `import.meta.env.VITE_API_BASE_URL` 读取中心服务地址。
5. 如果生产环境未提供该变量，代码会回退到内置默认地址 `http://127.0.0.1:28080`。
6. 日常本地启动优先修改配置文件，不要求手工设置 PowerShell 环境变量。

## 3. 环境文件约定

### 3.1 `web/.env.development`

用途：

- 本地联调
- 开发阶段默认启动
- 手工验收时的默认代理目标配置

约定：

- 默认写入本机中心服务代理目标 `VITE_API_PROXY_TARGET`
- 修改后前端会在 `npm run dev` 时自动读取
- 如果本机中心服务端口变更，只需要调整这个文件，不需要改代码
- 浏览器访问的是前端自己的 `/api` 和 `/ws`，由 Vite 转发到中心服务

### 3.2 `web/.env.production`

用途：

- 生产构建
- 发布包打包前的正式地址配置

约定：

- 必须填写真实的线上中心服务地址 `VITE_API_BASE_URL`
- 不要把开发环境地址直接作为生产默认值长期保留
- 发布前需要再次确认该文件中的地址是否正确

## 4. 前端请求规则

1. 所有前端请求统一走 `src/api/` 下的封装，不在页面里散写 `fetch`。
2. 请求基地址统一由 `getApiBaseUrl()` 管理。
3. 开发环境默认返回空基地址，直接请求相对路径，由 Vite 代理到中心服务。
4. 生产环境读取到的 `VITE_API_BASE_URL` 需要去掉末尾多余的 `/`，避免拼接出重复斜杠。
5. 当请求失败时，页面只负责展示错误提示，不在页面里拼接后端地址逻辑。

## 5. 验收与调试

1. 本地开发优先使用 `web/.env.development`。
2. 验收时如果前端加载失败，优先检查 `web/.env.development` 里的 `VITE_API_PROXY_TARGET` 是否正确。
3. 如果浏览器出现 `/api` 或 `/ws` 代理失败，优先检查中心服务是否已启动。
4. 生产构建前优先检查 `web/.env.production` 中的 `VITE_API_BASE_URL`。
5. 如果需要临时切换中心服务地址，优先修改环境文件，不建议通过手工注入 PowerShell 环境变量作为长期方案。

## 6. 后续扩展

1. 后续如果增加多环境地址，只新增对应的 `web/.env.*` 文件，不改变页面内的配置读取方式。
2. 如果需要调整代理路径或网关策略，先更新 `vite.config.ts` 和本约定，再更新验收文档。

## 7. 页面风格与组件约定

1. 前端页面默认使用 `Element Plus` 组件，不再以原生 `input`、`select`、`table` 作为长期页面方案。
2. 后台信息架构、配色层次、卡片布局、左侧导航组织方式，统一借鉴 [yudao-ui-admin-vue3](https://gitee.com/yudaocode/yudao-ui-admin-vue3) 的管理后台风格，但不要求逐像素复刻。
3. 页面应优先采用“页面头部 + 摘要卡片 + 主数据卡片 + 详情/弹窗”的后台结构，避免把所有表单和表格直接平铺在一个区域内。
4. 新增或改造页面时，优先复用 `Element Plus` 的 `Dialog`、`Form`、`Table`、`Card`、`Alert`、`Tag`、`Descriptions` 等组件，而不是重复自定义基础交互控件。
5. 需要用户确认的创建、上传、批量执行等操作，默认使用模态窗口承载，不直接把所有输入项常驻暴露在页面主体区域。
6. 同一资源存在多版本、多状态或多维信息时，优先使用分组展示、下拉切换、标签化状态和详情面板，不直接展开成重复行列表。

## 8. 批量操作页面约定

1. 涉及批量设备操作的页面，必须拆成“发现结果区”“多选目标区”“执行参数区”“结果反馈区”四个可见区域。
2. 多设备操作默认使用显式复选框，不使用隐式批量选择。
3. 页面必须展示当前能力边界和前置条件，例如“仅支持中心服务所在机器可通过 `adb` 访问的无线调试设备”。
4. 页面必须按设备逐项展示执行结果，不能只显示一个总成功或总失败提示。
5. 如果页面提供“全部下发”“全部执行”之类的全量动作，必须与“全部选择”区分语义。
6. “全部选择”只表示前端当前列表范围内的选择动作；“全部下发”必须优先调用后端独立批量接口，由后端自行查询目标集合，不依赖前端分页、勾选状态或逐条循环请求。
