# 中心服务开发约定

## 通用约定

- 优先拆成职责清晰的小模块。
- 持久化状态进入 SQLite，短期运行态放在内存。
- 不要把设备运行时逻辑和后台界面逻辑混在一起。

## Go 代码约定

- `cmd/` 只保留启动入口，业务逻辑进入 `internal/`。
- 优先使用显式构造函数，避免包级全局状态。
- 处理器和后台循环优先使用带 `context` 的 API。
- 日志统一通过标准日志入口输出，避免页面逻辑直接拼接运行日志。
- 所有面向设备的协议载荷统一定义在 `pkg/protocol`。
- 与工作流执行主线相关的状态机责任必须明确区分“中心投影状态”和“Agent 真源状态”，不能在没有迁移说明的情况下把单设备工作流节点推进重新堆回中心。

## API 约定

- HTTP 接口统一挂在 `/api/v1` 下。
- WebSocket 消息必须包含 `type`、`request_id`、`device_id`、`timestamp`、`payload`。
- 设备相关操作尽量做到幂等。
- 涉及本机工具编排的接口必须显式区分“查询类接口”和“执行类接口”，例如“设备发现列表”和“批量下发 Agent”不能混成一个接口。
- 涉及调用本机 `adb`、脚本文件或仓库路径的功能，必须把可变路径收敛到启动配置中，不能把本机绝对路径硬编码进业务逻辑。
- 中心服务调用本机推送能力时，必须区分“人工手工脚本”和“中心服务专用非交互入口”，不能为了自动化破坏原有手工工作流。
- 网页批量下发 Agent 只允许走非交互能力，不能依赖命令行选号、人工输入或交互确认。
- 网页触发 Agent 下发时，只负责下发 Agent 本体、运行目录和中心连接配置；业务脚本同步必须走独立的脚本管理或脚本下发链路。
- 对“全部下发脚本”“全部执行任务”这类全量动作，必须提供独立后端接口，由后端自行查询目标设备集合并遍历执行，不能依赖前端分页结果或前端逐条循环请求。
- 工作流执行协议如果进入正式主线，默认优先采用“会话级下发 + Agent 本地流转”，不再默认扩展“逐节点 task + 中心同步推进”的实现。

## 持久化约定

- 不把 SQLite 当作原始高频事件日志文件来使用。
- 原始大量设备日志保存在磁盘文件中。
- 数据库表只保存汇总信息、索引信息、元数据和状态流转结果。
- 局域网设备发现结果属于短期运行态，第一版默认不落库，只通过接口实时返回。
- 中心服务所有核心业务表默认采用数据库自增整数主键。
- 除键值配置表这类天然适合业务键主键的场景外，普通业务表的主键字段统一命名为 `id`，不再混用 `device_id`、`task_id`、`workflow_def_id` 这类“表名加后缀”主键命名。
- 表之间的外键字段统一使用“目标语义 + `_id`”命名，例如 `device_id`、`task_id`、`workflow_def_id`、`plan_run_id`。
- `id` 只承担数据库主键职责；如未来确实需要可读业务编号，只能作为普通业务字段或展示字段存在，不能替代数据库主键。
- 如果未来确实需要保留可读业务编号，只能作为普通业务字段或展示字段存在，不能替代数据库主键职责。
- 工作流、计划任务、设备执行会话等与执行主线有关的表字段，必须优先按“中心投影状态”建模；如果某字段在 Agent 才是真源，文档中必须明确标注，避免双主。

## 配置约定

- 启动配置优先级固定为“程序默认值 -> `.env` -> 进程环境变量”。
- 新增会影响运行策略的配置项时，必须同步更新 `config.go`、`.env.example`、手工验收文档和开发约定。
- 计划任务调度默认按秒级扫描，但扫描协程只负责发现任务并投递异步执行请求，不能在扫描协程中直接串行完成大批量设备启动或停止。
- 计划任务、工作流这类可能同时影响多设备的后台任务，默认应采用“受控并发 worker pool”而不是“无上限 goroutine”模型。
- 与设备发现和网页下发相关的配置至少包含 `CENTER_ADB_PATH`、`CENTER_AGENT_ROOT_PATH`、`CENTER_TOOLKIT_PATH`、`CENTER_BASE_URL`。
- 中心服务执行网页下发时，统一调用 `mobilerpa-toolkit push-center`。
- 如果要支持 Linux 部署，新增能力优先落到独立 Go 工具，不继续扩大对 `.bat` 或 PowerShell 的直接依赖。
- 生产环境中，中心服务的设备发现、Agent 下发、Agent 启动、Agent 停止默认只认 `mobilerpa-toolkit`，不再把 `.bat`、`.ps1` 作为正式依赖。
- 当工作流执行主线以 300 到 500 台设备为目标时，中心侧新增设计优先削减“逐节点网络往返”和“逐节点同步数据库写链路”，优先把单设备本地状态机能力下沉到 Agent。

## 发布约定

- 中心后端与后台前端的改动在同一仓库内提交。
- 在正式引入迁移工具之前，任何表结构调整都需要在 `docs/` 中补充迁移说明。

## 文档约定

- 项目文档默认使用中文编写。
- 代码注释默认使用中文编写，包括方法注释、结构体注释、结构体字段注释和必要的行内注释。
- 代码注释遵循“必要且简洁”的原则，不写无信息量注释。
- 包名、变量名、字段名、接口路径、协议字段和第三方库名称保留其技术标识，不强行翻译。
- 每次编写接口、WebSocket 或功能验收文档时，必须附带可直接复制到 Apifox、Postman 或同类工具的完整请求内容。
- 验收文档不能只给脚本片段或代码调用方式，必须同时给出工具侧完整请求地址、请求体、预期响应和字段替换说明。
- 涉及本机 `adb`、无线调试、局域网设备发现和网页下发 Agent 的功能，验收文档必须写清楚前置条件，例如 `adb` 已安装、手机已完成无线调试配对、中心服务机器与手机位于同一局域网。
- 当职责边界发生变化，例如“工作流状态机真源从中心迁移到 Agent”，必须同步更新架构文档、开发约定、Agent 职责文档与相关验收文档，不能只改代码不改文档。

## 协作确认约定

- 如果我对需求有疑问，或者觉得某个要求不合理，必须先停下来说明疑问、风险、建议方案和可选方案，等待用户确认后再继续。
- 涉及工程惯例、工具默认识别或平台展示的命名，例如 `README.md`、配置文件名、启动入口和目录结构，不能由我直接做最终决定。
- 计划顺序、开发周期、任务状态和验收结论的调整，必须先征得用户确认。
- 用户确认前，只允许进行阅读、检查、梳理和提出建议，不擅自执行会改变结果的操作。
