# 计划任务 500 台设备压测与验收步骤

## 1. 目标

本验收文档用于验证以下内容：

1. 计划任务 `daily` 自动启动在 500 台设备规模下是否仍可稳定触发。
2. 中心服务“扫描协程轻量化 + 异步 worker + 受控并发下发”改造后，是否不会因为大量设备同时启动而阻塞调度主循环。
3. 启动波峰期间，计划任务实例创建、设备运行记录推进、设备确认、运行结果回传是否保持可观测。

## 2. 本次验收关注点

本次重点不是要求 500 台真机必须全部真实执行，而是分成三层来验收：

1. 调度层验收：到点后计划任务实例是否能在秒级内被创建。
2. 下发层验收：中心服务是否会持续稳定地下发大量设备任务，而不是卡死在扫描线程里。
3. 执行层验收：在有限真机或模拟连接条件下，是否能看到设备侧正常收到任务、确认任务、回传结果。

## 3. 前置条件

### 3.1 服务配置

建议先在 `.env` 中设置：

```env
CENTER_PLAN_SCAN_INTERVAL=1s
CENTER_PLAN_START_WORKERS=2
CENTER_PLAN_START_FANOUT=20
```

如需更激进的压测，可临时改为：

```env
CENTER_PLAN_START_WORKERS=4
CENTER_PLAN_START_FANOUT=50
```

不建议一开始就把并发度开得过高，否则 SQLite 写入和 WebSocket 下发会一起放大波峰。

### 3.2 验收对象

建议准备以下任一环境：

1. 理想方案：500 台真实在线设备。
2. 过渡方案：少量真机 + 大量模拟 WebSocket 连接。
3. 最小方案：10 到 30 台真机，重点观察调度与下发节奏，而不是绝对总量。

### 3.3 建议脚本

建议使用一个轻量脚本版本做压测，例如：

1. `open_qq@v0.1.0`
2. `open_wechat@v0.1.0`
3. 一个只做简单日志输出并快速返回成功的压测脚本

压测阶段不建议直接使用业务逻辑复杂、执行时间长、依赖无障碍交互链很长的脚本。

## 4. 压测方式

### 4.1 计划任务定义建议

创建一个 `daily` 类型计划任务：

1. 目标类型：脚本
2. 脚本名：任选一个轻量脚本
3. 脚本版本：固定版本
4. 调度类型：按天循环
5. 每天开始时间：设置为当前时间后 1 到 2 分钟
6. 每天结束时间：设置为开始后 10 到 20 分钟
7. 默认设备：尽可能选择全部目标设备

### 4.2 核心验证指标

需要重点记录以下时间点：

1. `daily_start_time`
2. 计划任务实例 `plan_run_id.created_at`
3. 第一批 `task_created`
4. 第一批 `task_assigned`
5. 第一批 `task_ack`
6. 最后一批 `task_assigned`
7. 最后一批 `task_ack`

## 5. 验收步骤

### 5.1 验证实例创建延迟

操作：

1. 启动中心服务。
2. 在计划任务页面创建一个 `daily` 计划任务。
3. 将开始时间设置为接下来 1 到 2 分钟的整秒时间。
4. 等待自动触发。

预期：

1. 计划任务实例应在 `daily_start_time` 后约 `0s ~ 1s` 内进入创建流程。
2. 即使设备很多，计划任务实例本身也不应晚很多秒才出现。

验收方式：

1. 在“计划任务实例”页面查看最新实例创建时间。
2. 对比实例时间与 `daily_start_time`。

### 5.2 验证扫描协程未被阻塞

操作：

1. 同时创建 2 到 3 个开始时间接近的 `daily` 计划任务。
2. 观察服务日志。

预期：

1. 扫描线程应持续每秒工作，不应因为前一个计划任务正在启动 500 台设备而完全停摆。
2. 日志中应看到计划任务被异步投递和异步启动，而不是长时间无输出。

建议观察日志关键字：

1. `auto started plan run asynchronously`
2. `dispatch assign_task start`
3. `dispatch assign_task done`

### 5.3 验证大量设备启动与下发节奏

操作：

1. 到点后观察计划任务实例对应的设备运行记录。
2. 同时结合计划任务事件与中心日志观察下发节奏。

预期：

1. 设备运行记录会持续进入启动流程。
2. 对于脚本型计划任务，底层任务会持续被创建并下发。
3. 不是“创建一个、下发一个、卡很久、再继续”的完全串行体验。

### 5.4 验证设备确认与结果回传

操作：

1. 在设备侧保持 Agent 在线。
2. 到点后查看计划任务事件与设备运行事件。
3. 查看 WebSocket 日志与运行结果。

预期：

1. 设备会收到 `assign_task`
2. 设备会回传 `task_ack`
3. 设备执行完成后会回传 `task_result`

### 5.5 验证截止时间自动停止

操作：

1. 将 `daily_deadline_time` 设置为距离开始后 10 到 20 分钟。
2. 等待截止时间到达。

预期：

1. 计划任务实例应自动停止。
2. `plan_run.status` 应变为 `stopped`、`success` 或 `failed` 中的最终态。
3. 未完成设备应被停止并写入对应事件。

## 6. 建议观察接口

### 6.1 查询计划任务定义

```http
GET /api/v1/plans
```

### 6.2 查询计划任务实例列表

```http
GET /api/v1/plans?view=runs
```

### 6.3 查询单个计划任务实例

将 `{plan_def_id}` 与 `{plan_run_id}` 替换成真实值：

```http
GET /api/v1/plans/{plan_def_id}/runs/{plan_run_id}
```

### 6.4 查询计划任务事件

```http
GET /api/v1/plans/{plan_def_id}/runs/{plan_run_id}/events
```

### 6.5 补充观察底层执行

说明：

1. 当前压测主入口统一为计划任务实例与计划任务事件。
2. 如需观察底层脚本执行，请优先查看中心日志、设备日志与计划任务事件流。

## 7. 推荐日志观察点

后端日志建议关注：

1. `auto started plan run asynchronously`
2. `dispatch assign_task start`
3. `dispatch assign_task done`
4. `task_progress received`
5. `device ... became online via hello`
6. `stopped plan run asynchronously by deadline`

## 8. 通过标准

满足以下条件可认为本轮通过：

1. 到点后实例创建延迟稳定在秒级。
2. 启动 500 台设备时，中心服务仍可继续扫描后续计划任务，不出现明显“整段卡死”。
3. 大量设备任务下发过程中，无大面积 `write assign_task ... i/o timeout`。
4. 设备可以持续回传 `task_ack` 与 `task_result`。
5. 到达截止时间后，计划任务实例能自动停止。

## 9. 风险提示

当前实现已经从“扫描线程直接串行启动”升级为“扫描轻量化 + 异步 worker + 受控并发下发”，但仍需注意：

1. SQLite 在极高并发写入场景下不是最终生产形态。
2. 500 台设备真实同时运行时，WebSocket 写入与任务事件落库仍会形成波峰。
3. 如果未来计划任务与工作流规模进一步提升，后续仍建议继续演进：
   - 任务批量写入
   - 事件归档
   - 更强的存储引擎
   - 内部分发队列持久化
