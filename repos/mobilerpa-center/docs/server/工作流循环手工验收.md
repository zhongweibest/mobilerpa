# 工作流循环手工验收

## 1. 验收目标

本文件用于验收当前阶段已经落地的工作流循环主线能力，重点确认以下事项：

1. 能创建带循环节点的工作流定义。
2. `max_iterations` 真正表示循环体脚本最多执行多少次。
3. 每完成一轮循环，中心会写入 `workflow_loop_completed` 事件。
4. 达到循环次数后，工作流会自动走到 `stop` 节点并结束。

## 2. 当前循环语义约定

为了避免后续编排和验收时产生误解，当前实现统一按下面口径理解：

1. `max_iterations = N` 的含义是：循环体脚本最多执行 `N` 次。
2. `workflow_loop_completed.extra.counter` 的含义是：当前已经进入循环体执行了第几轮。
3. 循环退出检查本身不额外记一次 `workflow_loop_completed`。
4. 所以后续你在看日志、事件和数据库时，不会再看到“配置 2 次但计数到了 3”的情况。

## 3. 前置准备

1. 中心服务已启动。
2. 至少有 1 台在线设备，例如 `dev_000001`。
3. 设备已具备任务执行条件。
4. 脚本 `open_qq@v0.1.0` 与 `open_wechat@v0.1.0` 已可用。
5. 你已经可以通过 Apifox 或真机链路回传 `task_result`。

## 4. 创建带循环节点的工作流定义

请求方法：

```text
POST
```

请求地址：

```text
http://127.0.0.1:18080/api/v1/workflows
```

请求体示例：

```json
{
  "workflow_name": "循环验收工作流",
  "description": "A -> loop -> B -> loop -> stop",
  "status": "active",
  "nodes": [
    {
      "node_id": "node_a",
      "node_type": "script",
      "node_name": "步骤A",
      "script_name": "open_qq",
      "script_version": "v0.1.0"
    },
    {
      "node_id": "node_loop",
      "node_type": "loop",
      "node_name": "循环节点",
      "max_iterations": 2
    },
    {
      "node_id": "node_b",
      "node_type": "script",
      "node_name": "循环体步骤",
      "script_name": "open_wechat",
      "script_version": "v0.1.0"
    },
    {
      "node_id": "node_stop",
      "node_type": "stop",
      "node_name": "结束"
    }
  ],
  "edges": [
    {
      "from_node_id": "node_a",
      "to_node_id": "node_loop",
      "edge_type": "next"
    },
    {
      "from_node_id": "node_loop",
      "to_node_id": "node_b",
      "edge_type": "loop_body"
    },
    {
      "from_node_id": "node_b",
      "to_node_id": "node_loop",
      "edge_type": "next"
    },
    {
      "from_node_id": "node_loop",
      "to_node_id": "node_stop",
      "edge_type": "loop_exit"
    }
  ]
}
```

验收点：

1. 返回 `status = ok`。
2. 返回结果中存在真实的 `workflow_def_id`。
3. `node_loop.max_iterations = 2`。
4. 边关系中同时存在 `loop_body` 和 `loop_exit`。

## 5. 启动该工作流

请求方法：

```text
POST
```

请求地址：

```text
http://127.0.0.1:18080/api/v1/workflows/{workflow_def_id}/start
```

请求体示例：

```json
{
  "device_ids": [
    "dev_000001"
  ]
}
```

验收点：

1. 返回 `status = ok`。
2. 返回结果中存在真实的 `workflow_instance_id`。
3. `device_runs[0].status` 为 `pending` 或 `running`。
4. 第一个下发任务对应 `node_a`。

## 6. 人工推动第一步完成，让工作流进入循环体

先查询该工作流实例对应的设备运行记录与当前任务：

```text
GET http://127.0.0.1:18080/api/v1/workflows?view=instances
```

找到刚才这条实例的 `workflow_run_id` 后，再去任务列表中找到该 `workflow_run_id` 对应的当前任务 `task_id`。

然后用你已经在前文验证过的方式，让该任务回传成功结果。你可以继续使用 Apifox WebSocket 模拟 `task_result`，也可以让真机实际执行完成后自动回传。

本步骤验收点：

1. `node_a` 成功后，工作流不会结束。
2. 下一步会进入 `node_b`。
3. 查询工作流事件时，能看到：
   - `workflow_step_succeeded`，`node_id = node_a`
   - `workflow_loop_completed`，`node_id = node_loop`
   - `workflow_step_started`，`node_id = node_b`

## 7. 连续完成两次循环体任务

因为这里 `max_iterations = 2`，所以你需要让循环体步骤 `node_b` 成功 2 次。

每成功一次后，都重新查询当前任务列表，找到同一个 `workflow_run_id` 下新产生的下一个 `task_id`，再继续回传成功。

验收点：

1. `node_b` 实际被执行 2 次，不多不少。
2. 工作流事件中一共出现 2 条 `workflow_loop_completed`。
3. 这 2 条事件的 `extra.counter` 分别为：
   - 第一轮：`1`
   - 第二轮：`2`

## 8. 验证达到循环上限后自动停止

当第二次循环体任务成功后，再查询工作流实例和事件：

```text
GET http://127.0.0.1:18080/api/v1/workflows?view=instances
GET http://127.0.0.1:18080/api/v1/workflows/{workflow_def_id}/runs/{workflow_run_id}/events
```

验收点：

1. 该设备运行记录最终 `status = success`。
2. `current_node_id = node_stop`。
3. 工作流事件里存在：
   - `workflow_run_completed`
   - `node_id = node_stop`
4. 不应再出现第三次 `node_b` 执行。

## 9. 本阶段边界

本文件只覆盖当前已经落地的最小循环闭环，不覆盖以下高级能力：

1. 多个循环节点嵌套。
2. 条件分支。
3. 并行分支。
4. 表达式驱动的动态跳转。
5. 图形化拖拽编排器。
