# 中心服务架构

## 运行时职责
中心服务需要提供以下能力：
- HTTP API：用于设备注册、文件上传、脚本下载和后台管理操作
- WebSocket：用于中心与设备之间的双向实时通信
- 调度器：为当前空闲设备分配任务
- SQLite：保存持久化业务状态
- 本地文件存储：保存上传文件、脚本包和日志

## 模块边界
- `internal/app`：应用启动装配
- `internal/config`：配置加载
- `internal/api`：HTTP 处理器与路由注册
- `internal/ws`：WebSocket 生命周期与消息分发
- `internal/device`：设备注册、绑定状态、心跳处理
- `internal/task`：任务生命周期管理
- `internal/script`：脚本版本与清单管理
- `internal/workflow`：工作流定义、运行实例与节点推进
- `internal/scheduler`：基于内存状态的调度循环
- `internal/filemgr`：本地磁盘路径和上传文件持久化
- `internal/report`：日报与汇总生成
- `internal/logger`：服务日志初始化和设备日志文件分流
- `pkg/protocol`：共享消息协议与载荷定义

> 模块边界说明：`internal/workflow` 收敛为“工作流定义与定义快照构建”，不再承担运行实例与节点推进（详见 `工作流实例层移除（已确认方案）`）。实际代码中还存在的 `internal/plan`、`internal/dispatch`、`internal/discovery`、`internal/settings` 等模块，待本节落地后一并回归到此处的模块清单。

## 工作流实例层移除（已确认方案）

本节为工作流执行域的权威目标模型，取代下方“当前工作流与计划任务落地方式”“设备占用与执行域边界”“当前运行态接口约定”“工作流定义与计划任务的职责边界”“任务中心收口策略”“统一设备占用判定方案”中涉及 `workflow_instances`/`workflow_runs` 的既有描述。schema 变更细节见 `工作流实例层移除迁移说明.md`。

### 目标执行层级

```text
plan_def        能力引用：script@version 或 workflow_def
  └─ plan_run   一次调度的“实例”，承载设备集合管理与运行态控制
       └─ plan_device_run   每台设备的独立运行记录
```

工作流型与脚本型计划任务共用同一条运行链路，不再有并行的 `workflow_instances`/`workflow_runs`。

### 保留与删除

1. 保留：`workflow_defs`、`workflow_nodes`、`workflow_edges`（能力模板定义）；前端“工作流编排”页继续维护定义。
2. 删除：`workflow_instances`、`workflow_runs`、`workflow_contexts`、`workflow_events` 四张表，以及对应的实例/运行/事件代码。
3. 删除：`tasks` 表上的 `workflow_instance_id`、`workflow_run_id`、`workflow_node_id` 列，以及旧的“中心逐节点 task 推进”链路（`startRunFromNode`、`advanceRunAfterNode`、`ensureNoOtherActiveWorkflow` 等）。工作流执行统一走 `workflow_session_*` 会话模型，节点状态机由 Agent 本地推进。
4. 不做数据迁移（开发阶段、无历史包袱）：表结构变更直接落地，历史运行数据随删表一并清除。

### 运行态字段承载

1. `plan_runs.target_ref_id`：工作流型存放 `workflow_def_id`（字符串形态），脚本型仍存放 `script@version`。
2. `plan_device_runs` 新增工作流投影列：`current_node_id`（当前节点）、`last_event_type`（最近事件类型）、`last_event_seq`（事件序号）、`last_event_at`（最近事件时间）。脚本型运行这些列留空。
3. 工作流节点事件并入 `plan_events`，按 `plan_run_id` + `plan_device_run_id` 索引；不再单独维护 `workflow_events`。
4. 设备占用统一为：`plan_device_runs.status` 为 `pending`/`running`（涵盖脚本型与工作流型）。工作流实例层与手工任务移除后，不再有其他占用域，占用判定收敛为 `plan_device_runs` 单域。

### 会话下发与控制归属

1. `start_workflow_session` 载荷按 `plan_run_id` + `plan_device_run_id` 定位设备运行记录；Agent 侧 `workflow_session_runner` 的 checkpoint 同步改用这两个键。
2. 停止、追加、移除设备等运行态控制统一在计划任务实例上进行；前端“工作流实例”页删除，查看某次工作流运行涉及哪些设备与节点事件，统一进入对应计划任务实例页。

### 与已实现临时改动的关系

`统一设备占用判定方案` 中“脚本型计划任务补查 `workflow_runs`”的临时实现，在本节落地后移除：合并后工作流执行态已在 `plan_device_runs` 上，现有 `plan_device_runs` 占用检查天然覆盖脚本型与工作流型，无需再单独查 `workflow_runs`。

### 手工任务移除（一并执行）

本节同期移除“手工任务”能力，只保留脚本型计划任务的执行底座：

1. 删除手工任务入口：`POST /api/v1/tasks`（手工创建）、`PATCH /api/v1/tasks`（手工下发）、`POST /api/v1/tasks/{id}/terminate`、`POST /api/v1/devices/{id}/manual-tasks/{task_id}`，以及 `task.StopManualTask` 等服务端实现。
2. 删除 `task_source_type='manual'` 概念：清理 `plan.inspectDeviceBusy`、`workflow.inspectDeviceAvailability` 中的手工任务占用分支；`task_source_type` 仅保留 `plan_script`（`workflow` 随旧逐节点链路一并消失）。
3. 保留执行底座：`internal/task` 服务（由计划任务内部 `Create` 调用）、`tasks`/`task_events` 表、`assign_task`/`task_ack`/`task_progress`/`task_result` 协议、`dispatch.AssignTask`——脚本型计划任务仍经此执行。
4. 前端任务中心相关残留一并清理（当前前端已无任务页，仅需确认无 API 引用）。

- 多节点部署
- Redis 或消息队列
- 对象存储
- 独立鉴权服务

## 当前工作流与计划任务落地方式
- 脚本是能力定义，任务是该能力在某台设备上的一次运行实例。
- 工作流定义保存为脚本节点、循环节点、停止节点及其连线关系。
- 循环节点当前支持最小闭环编排：`script -> loop -> script(loop_body) -> loop -> stop(loop_exit)`。
- 循环节点的 `max_iterations` 语义固定为“循环体脚本最多执行多少次”。
- 当 `max_iterations = N` 时：
  - 循环体脚本实际执行次数为 `N`
  - `workflow_loop_completed` 事件实际写入次数也为 `N`
  - 循环上下文中保存的循环计数最终值也为 `N`
- 循环节点不会把“退出检查”额外记成一次循环完成，避免出现人为理解上的 `N+1` 偏差。
- 计划任务是统一调度层，负责承载脚本型与工作流型两种运行入口。
- 正式运行态主对象是“计划任务实例”，而不是工作流定义本身。
- 工作流定义页只负责维护定义；工作流型计划任务负责选择设备、定义每日开始时间、每日截止时间和后续自动调度语义。
- 当前系统同一时刻只允许一个主工作流处于运行态，不做多个主工作流并行。
- 同一设备同一时刻只允许属于一个计划任务实例，或一个手工任务，不能同时被多个执行域占用。
- 工作流节点如果修改了脚本名或脚本版本，不对已运行实例热更新，必须人工停止后重新启动；重新启动后设备按最新脚本定义重新拉取执行。
- 计划任务支持 `daily` 自动调度：到达 `daily_start_time` 后自动创建当天实例，到达 `daily_deadline_time` 后自动停止仍在运行的实例。
- 计划任务调度默认按 `1s` 轮询一次，仅负责识别“到点可启动/应停止”的计划任务，不在扫描协程内串行执行大批量设备启动。
- 计划任务命中后会先投递到中心服务内部异步启动队列，再由独立 worker 执行实例创建、设备运行记录初始化、任务创建与任务下发。
- 脚本型计划任务在启动 worker 内采用“先批量创建任务，再用受控并发 fan-out 下发”的策略，避免单个计划任务在 300 到 500 台设备规模下阻塞扫描线程。
- 计划任务启动并发度通过配置控制，默认不使用无限 goroutine 扩散模型，防止同时命中多个计划任务时把数据库和 WebSocket 写链路打爆。
- 计划任务 `once` 的语义是“手工启动一次”，不会按指定年月日时分秒自动启动；如果需要执行，必须由用户在“计划任务”页手工触发启动。
- 计划任务 `daily` 的语义是“按天自动调度”，同时允许用户在页面上人工立即启动一个实例，不影响后续 daily 自动调度。
- 如果中心服务在当天截止时间之后启动，不会为当天补建新的 daily 实例。
- 运行中的计划任务实例允许追加设备和移除设备；该设备集合会同步回计划任务定义，用于次日自动启动时继续生效。
- 当前保留工作流直接启动接口，仅作为兼容入口；后续默认应通过计划任务页发起运行。

## 工作流执行状态机演进方向
- 当前 500 台设备计划任务启动链路已经按“轻量扫描协程 + 异步 worker + 受控并发 fan-out”落地，不再把大批量设备启动压在扫描线程内。
- 后续真正需要继续轻量化的重点，不是重复改造计划任务 worker/pool，而是把“工作流逐节点推进状态机”从中心迁移到 Agent。
- 目标形态是：
  - 中心负责计划任务调度、设备占用、工作流会话下发、关键事件存储与前端查询。
  - Agent 负责单设备工作流状态机、本地节点流转、本地循环计数和本地 checkpoint 恢复。
- 详细方案见：
  - `docs/server/工作流状态机下沉到Agent架构方案.md`

## 设备占用与执行域边界
- 手工任务域、计划任务域与工作流推进链共用同一套设备占用规则。
- 手工任务占用条件：`task_source_type=manual` 且任务状态为 `assigned` 或 `running`。
- 计划任务占用条件：`plan_device_runs.status` 为 `pending` 或 `running`。
- 工作流占用条件：工作流设备运行记录状态为 `pending` 或 `running`，并由其所属计划任务实例统一承接主运行态。
- 如果设备已经执行过某个工作流实例，则不能被再次追加到同一个实例，返回 `workflow_instance_history`。
- 中心服务提供设备占用详情查询接口，前端可以直接展示设备被哪个手工任务或哪个工作流实例占用。
- 中心服务提供人工结束手工任务入口，仅允许结束仍处于 `assigned/running` 的手工任务；结束后立即释放设备占用。

## 统一设备占用判定方案

> 本节为过渡期方案。`工作流实例层移除（已确认方案）` 落地后，工作流执行态并入 `plan_device_runs`，占用判定回归 `plan_device_runs + 手工任务` 两域，本节中“脚本型补查 `workflow_runs`”的临时实现随之移除。

本节记录设备占用判定的统一目标与落地范围。其中“脚本型计划任务补查 `workflow_runs`”为已确认要实现的第一步，其余为方向性约束。

### 背景与当前实现差距

`设备占用与执行域边界` 已经声明“手工任务域、计划任务域与工作流推进链共用同一套设备占用规则”，但当前实现并不统一：

- `internal/plan/service.go` 的 `inspectDeviceBusy` 按 `target_type` 走不同分支。
- `internal/workflow/service.go` 的 `inspectDeviceAvailability`（被 `workflow.GetDeviceBusyDetail` 和计划任务的工作流分支复用）只查 `workflow_runs` 和手工任务，不查 `plan_device_runs`。

实际占用判定矩阵如下：

| 本次启动对象 | 查 `plan_device_runs` | 查 `workflow_runs` | 查手工任务 |
|---|---|---|---|
| 脚本型计划任务（`target_type=script`） | 是 | **否** | 是 |
| 工作流型计划任务（`target_type=workflow`） | 是 | 是 | 是 |
| 手工任务创建（`POST /api/v1/tasks`） | **否** | 是 | 是 |

带来的实际后果：

1. 一台设备上若存在未结束的工作流运行，脚本型计划任务仍能被启动，违反“同一设备同一时刻只能属于一个执行域”。
2. 残留的 `workflow_runs.status=pending/running` 会卡住工作流型计划任务（报 `plan_device_busy`），却不会卡住脚本型计划任务，表现为“同一台设备、不同目标类型的计划任务结果不一致”。

### 本期落地范围（已确认）

1. **脚本型计划任务补查 `workflow_runs` 占用**：让 `inspectDeviceBusy` 在 `target_type=script` 时也检查目标设备是否存在 `workflow_runs.status=pending/running`，命中即返回 `plan_device_busy`。这是本期要实现的内容。
2. **不补齐手工任务创建的占用检查**：`POST /api/v1/tasks` 手工任务创建后续会被整体移除（见 `任务中心收口策略`），因此不再为其补齐 `plan_device_runs` 占用检查，避免为即将下线的链路增加维护面。

实现后，脚本型计划任务在目标设备存在未结束工作流运行时，将由“可启动”变为“被占用拒绝”。这是符合统一规则的正确行为，但属于已验收路径的行为变更，需重新人工验收“脚本型计划任务与工作流并存”场景。

### 运行态控制归属（已确认）

1. 停止、追加、移除设备等运行态控制统一在**计划任务实例**上进行，不在工作流实例上进行。
2. 工作流实例只读：只用于查看哪些设备运行过、以及查看相关设备事件。
3. 因此前端停止入口应位于计划任务实例页；工作流实例页不提供停止按钮（当前前端 `WorkflowsPage` 也确实没有，方向正确）。

### 孤立工作流实例的根因与修复方向

孤立工作流实例（`plan_run_id=0` 的实例）属于必须从根因消除的缺陷，不通过超时回收或兜底策略掩盖。根因有两点：

1. **创建过程非原子**。`workflow.Start` → `addRuns` 把“建实例、建 run、写 `workflow_run_started` 事件、派发会话”拆成多条独立 SQL/exec，没有事务包裹。任意一步在 `createInstance`/`createRun` 之后失败（例如 `workflow_events` 插入失败、会话派发失败），都会留下“实例与 run 已建、但无事件、无 task、未派发”的残留，且 `plan_run_id` 仍为 0。`startWorkflowPlanRun` 进一步放大了这一点：它先调 `workflows.Start`（此时 `plan_run_id=0`），再在另一个事务里建 `plan_runs` 行并回填 `plan_run_id`；若 `Start` 内部中途失败，整个计划任务启动在落 `plan_runs` 行之前就返回错误，于是只剩下一个不归属任何计划任务实例的孤立工作流实例。
2. **直接启动兼容入口本身就会产生无主实例**。`POST /api/v1/workflows/{workflow_def_id}/start` 走 `addRuns`，而 `createInstance` 从不写 `plan_run_id`，必然得到 `plan_run_id=0` 的实例，与“工作流实例应归属计划任务实例、控制入口在计划任务实例”的模型冲突。

实测数据印证（2026-06-26）：`workflow_instances.id=7`（`workflow_def_id=2`、`plan_run_id=0`、`status=running`）下挂 `workflow_runs.id=7`（`device_id=2`、`status=pending`、`current_node_id=node_1`、`current_task_id=0`），且该实例在 `workflow_events` 中零事件、`tasks` 中零记录——正是“`addRuns` 在 `createRun` 之后、事件写入/会话派发之前失败”留下的非原子残留，并长期占用设备 2，导致计划任务“试试”启动时报 `plan_device_busy`。

修复方向（消除根因，非兜底）：

1. 让工作流实例与 run 的创建成为原子操作：`createInstance`、`createRun`、首条事件、会话派发要么整体成功，要么整体回滚；`startWorkflowPlanRun` 应先建立 `plan_runs` 归属再创建工作流实例（或在同一事务内绑定 `plan_run_id`），杜绝“实例已建、归属未绑”的中间态。
2. 下线或收紧直接启动兼容入口：工作流统一经由计划任务发起，确保任何工作流实例自创建起就归属某个计划任务实例，与“运行态控制归属”一致。
3. 在上述根因修复落地前，已产生的残留（如 `workflow_instances.id=7`）作为一次性数据修正处理，不作为长期策略。

### 落地前提

1. “脚本型计划任务补查 `workflow_runs`”已确认可实现；其余根因修复项（原子化、下线直接启动入口）需排期确认后再进入实现。
2. 实现需同步更新 `中心服务开发约定.md` 的占用相关条目、`docs/整体规划方案.md` 第 14 章相关说明，以及对应模块手工验收文档。
3. 不改变 `任务中心收口策略` 中关于 `task` 执行底座暂不删除的既有结论。

## 当前运行态接口约定

> 本节列出的“工作流实例/运行”相关接口在 `工作流实例层移除（已确认方案）` 落地后删除；运行态控制统一走计划任务实例接口（`/api/v1/plans/...`）。

- 计划任务定义列表：`GET /api/v1/plans`
- 计划任务定义创建：`POST /api/v1/plans`
- 计划任务实例列表：`GET /api/v1/plans?view=runs`
- 启动计划任务：`POST /api/v1/plans/{plan_def_id}/start`
- 查询计划任务实例：`GET /api/v1/plans/{plan_def_id}/runs/{plan_run_id}`
- 查询计划任务事件：`GET /api/v1/plans/{plan_def_id}/runs/{plan_run_id}/events`
- 追加设备到计划任务实例：`POST /api/v1/plans/{plan_def_id}/runs/{plan_run_id}/devices`
- 从计划任务实例移除设备：`DELETE /api/v1/plans/{plan_def_id}/runs/{plan_run_id}/devices/{device_id}`
- 停止整个计划任务实例：`POST /api/v1/plans/{plan_def_id}/runs/{plan_run_id}/stop`
- 工作流定义启动：`POST /api/v1/workflows/{workflow_def_id}/start`
- 查询工作流实例视图：`GET /api/v1/workflows?view=instances`
- 追加设备到工作流实例：`POST /api/v1/workflows/{workflow_def_id}/instances/{workflow_instance_id}/devices`
- 停止整个工作流实例：`POST /api/v1/workflows/{workflow_def_id}/instances/{workflow_instance_id}/stop`
- 停止工作流实例中的单台设备：`POST /api/v1/workflows/{workflow_def_id}/instances/{workflow_instance_id}/devices/{device_id}/stop`
- 查询设备占用详情：`GET /api/v1/devices/{device_id}/occupancy`
- 设备域人工结束手工任务：`POST /api/v1/devices/{device_id}/manual-tasks/{task_id}`
- 历史兼容人工结束手工任务：`POST /api/v1/tasks/{task_id}/terminate`

## 当前工作流循环主线范围
- 本阶段先支持“顺序节点 + 单循环节点 + 停止节点”的基础编排闭环。
- 对应的手工验收步骤见：
  - `docs/server/工作流循环手工验收.md`
- 当前前端编排页已经扩展为：
  - 多个顺序段
  - 多个循环段
  - 顺序段与循环段的串行组合
- 例如下面这类结构现在已经属于当前支持范围：
  - `A -> B -> CDE循环(3次) -> FG循环(5次) -> H -> stop`
- 当前循环段语义：
  - 循环段本身是一个 `loop` 节点
  - 循环段内部可以挂多个脚本步骤
  - `max_iterations` 表示整段循环体最多执行多少轮，而不是单个步骤各自的次数
- 本阶段前端创建页支持：
  - 顺序段增删
  - 循环段增删
  - 段顺序上下移动
  - 顺序段内多步骤编辑
  - 循环段内多步骤编辑
  - 循环段名称
  - 循环段最大次数
- 本阶段暂不支持：
  - 多个循环节点嵌套
  - 图形化拖拽编排
  - 条件分支
  - 并行分支
  - 基于表达式的动态跳转

## 工作流定义与计划任务的职责边界
- 工作流定义只负责描述“执行结构”，例如顺序关系、循环关系、停止节点。
- 计划任务负责描述“何时运行”和“运行到什么时候”，例如：
  - 每日开始时间
  - 每日截止时间
  - 次日是否继续拉起
  - 设备集合
- 因此 `daily_start_time`、`daily_deadline_time`、`restart_next_day` 这类调度字段不应继续作为工作流定义前端的主输入项。
- 当前后端仍保留部分工作流定义级截止时间兼容逻辑，主要用于已存在链路和历史数据过渡；后续主线应以计划任务调度为准。

## 任务中心收口策略
- 计划任务已经成为正式运行入口，后续产品层面不再继续扩展“任务中心”页面。
- 但当前不能直接删除底层 `task` 执行域，因为脚本型计划任务、设备占用判断和 Agent 的 `assign_task / task_ack / task_progress / task_result` 协议仍然复用 `tasks` 与 `task_events`。
- 因此任务中心的收口策略必须分阶段进行：
  - 第一阶段：下线前端“任务中心”入口，不再鼓励通过手工任务发起正式运行。
  - 第二阶段：工作流正式执行链路切换到 `workflow_session_*`，应用装配层不再把“工作流逐节点 task 推进”作为默认主链。
  - 第三阶段：只保留手工任务的占用释放与历史兼容能力，逐步停止新增 `task_source_type=manual`。
  - 第四阶段：等脚本型计划任务也完成统一执行会话建模后，再评估是否还能继续保留 `task` 作为内部执行底座，或进一步迁移到新的 execution/session 模型。
- 结论是：当前可以下线“任务中心页面”，工作流正式执行默认走 `workflow_session_*`，但仍不能直接删掉 `internal/task`、`/api/v1/tasks` 背后的底层执行链。
