# 中心服务架构

本文档描述中心端中的“中心服务后端”部分。中心端由中心服务后端和管理后台前端共同组成；管理后台是中心端的 Web 管理入口，不是独立于中心服务之外的另一套业务中心。

## 运行时职责
中心服务需要提供以下能力：
- HTTP API：用于设备注册、文件上传、脚本下载以及管理后台调用的管理接口
- WebSocket：用于中心与设备之间的双向实时通信
- 调度器：负责计划任务调度、运行控制和任务下发
- SQLite：保存持久化业务状态
- 本地文件存储：保存上传文件、脚本包和日志

## 模块边界
- `internal/app`：应用启动装配
- `internal/config`：配置加载
- `internal/api`：HTTP 处理器与路由注册
- `internal/ws`：WebSocket 生命周期与消息分发
- `internal/device`：设备注册、绑定状态、心跳处理
- `internal/task`：任务生命周期管理
- `internal/script`：脚本版本与清单管理
- `internal/workflow`：工作流定义与定义快照构建
- `internal/plan`：计划任务定义、运行实例、设备运行记录与调度控制
- `internal/dispatch`：任务与工作流会话下发
- `internal/discovery`：设备发现、Agent 下发与运维动作
- `internal/settings`：系统配置读写
- `internal/scheduler`：基于内存状态的调度循环
- `internal/filemgr`：本地磁盘路径和上传文件持久化
- `internal/report`：日报与汇总生成
- `internal/logger`：服务日志初始化和设备日志文件分流
- `pkg/protocol`：共享消息协议与载荷定义

> 模块边界说明：`internal/workflow` 收敛为“工作流定义与定义快照构建”，不再承担运行实例与节点推进（详见 `工作流实例层移除（已确认方案）`）。

## 工作流实例层移除（已确认方案）

本节为工作流执行域的权威目标模型，取代下方“当前工作流与计划任务落地方式”“设备占用与执行域边界”“当前运行态接口约定”“工作流定义与计划任务的职责边界”“任务中心收口策略”“统一设备占用判定方案”中涉及 `workflow_instances`/`workflow_runs` 的既有描述。schema 变更细节见 `工作流实例层移除迁移说明.md`。

### 目标执行层级

```text
plan_def        能力引用：script@version 或 workflow_def
  └─ plan_run   一次调度的“实例”，承载设备集合管理与运行态控制
       └─ plan_device_run   每台设备的独立运行记录
```

工作流型与脚本型计划任务共用同一条运行链路，不再有并行的 `workflow_instances`/`workflow_runs`。

### 保留与删除

1. 保留：`workflow_defs`、`workflow_nodes`、`workflow_edges`（能力模板定义）；前端“工作流编排”页继续维护定义。
2. 删除：`workflow_instances`、`workflow_runs`、`workflow_contexts`、`workflow_events` 四张表，以及对应的实例/运行/事件代码。
3. 删除：`tasks` 表上的 `workflow_instance_id`、`workflow_run_id` 列，以及旧的“中心逐节点 task 推进”链路（`startRunFromNode`、`advanceRunAfterNode`、`ensureNoOtherActiveWorkflow` 等）。`workflow_node_id` 仅保留为 workflow session 下派生 task 的节点归属字段。工作流执行统一走 `workflow_session_*` 会话模型，节点状态机由 Agent 本地推进。
4. 不做数据迁移（开发阶段、无历史包袱）：表结构变更直接落地，历史运行数据随删表一并清除。

### 运行态字段承载

1. `plan_runs.target_ref_id`：工作流型存放 `workflow_def_id`（字符串形态），脚本型仍存放 `script@version`。
2. `plan_device_runs` 新增工作流投影列：`current_node_id`（当前节点）、`last_event_type`（最近事件类型）、`last_event_seq`（事件序号）、`last_event_at`（最近事件时间）。脚本型运行这些列留空。
3. 工作流节点事件并入 `plan_events`，按 `plan_run_id` + `plan_device_run_id` 索引；不再单独维护 `workflow_events`。
4. 设备占用统一为：`plan_device_runs.status` 为 `pending`/`running`（涵盖脚本型与工作流型）。工作流实例层与手工任务移除后，不再有其他占用域，占用判定收敛为 `plan_device_runs` 单域。

### 会话下发与控制归属

1. `start_workflow_session` 载荷按 `plan_run_id` + `plan_device_run_id` 定位设备运行记录；Agent 侧 `workflow_session_runner` 的 checkpoint 同步改用这两个键。
2. 停止、追加、移除设备等运行态控制统一在计划任务实例上进行；前端“工作流实例”页删除，查看某次工作流运行涉及哪些设备与节点事件，统一进入对应计划任务实例页。

### 与已实现临时改动的关系

`统一设备占用判定方案` 中“脚本型计划任务补查 `workflow_runs`”的临时实现，在本节落地后移除：合并后工作流执行态已在 `plan_device_runs` 上，现有 `plan_device_runs` 占用检查天然覆盖脚本型与工作流型，无需再单独查 `workflow_runs`。

### 手工任务移除（一并执行）

本节同期移除“手工任务”能力，只保留脚本型计划任务的执行底座：

1. 删除手工任务入口：`POST /api/v1/tasks`（手工创建）、`PATCH /api/v1/tasks`（手工下发）、`POST /api/v1/tasks/{id}/terminate`、`POST /api/v1/devices/{id}/manual-tasks/{task_id}`，以及 `task.StopManualTask` 等服务端实现。
2. 删除 `task_source_type='manual'` 概念：清理 `plan.inspectDeviceBusy`、`workflow.inspectDeviceAvailability` 中的手工任务占用分支；`task_source_type` 仅保留 `plan_script`（`workflow` 随旧逐节点链路一并消失）。
3. 保留执行底座：`internal/task` 服务（由计划任务内部 `Create` 调用）、`tasks`/`task_events` 表、`assign_task`/`task_ack`/`task_progress`/`task_result` 协议、`dispatch.AssignTask`——脚本型计划任务仍经此执行。
4. 前端任务中心相关残留一并清理（当前前端已无任务页，仅需确认无 API 引用）。

## 存储选型
- 数据库：SQLite
- 缓存：仅使用进程内内存
- 文件存储：仅使用本地磁盘
- 部署方式：单台 Linux 主机

## 明确不做的事情
- 多节点部署
- Redis 或消息队列
- 对象存储
- 独立鉴权服务

## 工作流与计划任务落地方式
- 脚本是能力定义，任务是该能力在某台设备上的一次运行实例。
- 工作流定义保存为脚本节点、循环节点、停止节点及其连线关系。
- 循环节点支持最小闭环编排：`script -> loop -> script(loop_body) -> loop -> stop(loop_exit)`。
- 循环节点的 `max_iterations` 语义固定为“循环体脚本最多执行多少次”。
- 当 `max_iterations = N` 时：
  - 循环体脚本实际执行次数为 `N`
  - `workflow_loop_completed` 事件实际写入次数也为 `N`
  - 循环上下文中保存的循环计数最终值也为 `N`
- 循环节点不会把“退出检查”额外记成一次循环完成，避免出现人为理解上的 `N+1` 偏差。
- 计划任务是统一调度层，负责承载脚本型与工作流型两种运行入口。
- 正式运行态主对象是“计划任务实例”，而不是工作流定义本身。
- 工作流定义页只负责维护定义；工作流型计划任务负责选择设备、定义每日开始时间、每日截止时间和后续自动调度语义。
- 同一设备同一时刻只允许属于一个计划任务实例，不能同时被多个执行域占用。
- 工作流节点如果修改了脚本名或脚本版本，不对已运行实例热更新，必须人工停止后重新启动；重新启动后设备按最新脚本定义重新拉取执行。
- 计划任务支持 `daily` 自动调度：到达 `daily_start_time` 后自动创建当天实例，到达 `daily_deadline_time` 后自动停止仍在运行的实例。
- 计划任务调度默认按 `1s` 轮询一次，仅负责识别“到点可启动/应停止”的计划任务，不在扫描协程内串行执行大批量设备启动。
- 计划任务命中后会先投递到中心服务内部异步启动队列，再由独立 worker 执行实例创建、设备运行记录初始化、任务创建与任务下发。
- 脚本型计划任务在启动 worker 内采用“先批量创建任务，再用受控并发 fan-out 下发”的策略，避免单个计划任务在 300 到 500 台设备规模下阻塞扫描线程。
- 计划任务启动并发度通过配置控制，默认不使用无限 goroutine 扩散模型，防止同时命中多个计划任务时把数据库和 WebSocket 写链路打爆。
- 计划任务 `once` 的语义是“手工启动一次”，不会按指定年月日时分秒自动启动；如果需要执行，必须由用户在“计划任务”页手工触发启动。
- 计划任务 `daily` 的语义是“按天自动调度”，同时允许用户在页面上人工立即启动一个实例，不影响后续 daily 自动调度。
- 如果中心服务在当天截止时间之后启动，不会为当天补建新的 daily 实例。
- 运行中的计划任务实例允许追加设备和移除设备；该设备集合会同步回计划任务定义，用于次日自动启动时继续生效。
- 工作流直接启动接口不作为正式运行入口；正式运行入口统一通过计划任务页发起。

## 工作流执行状态机演进方向
- 500 台设备计划任务启动链路按“轻量扫描协程 + 异步 worker + 受控并发 fan-out”落地，不再把大批量设备启动压在扫描线程内。
- 轻量化重点放在“工作流逐节点推进状态机由 Agent 本地承载”，而不是重复改造计划任务 worker/pool。
- 目标形态是：
  - 中心负责计划任务调度、设备占用、工作流会话下发、关键事件存储与前端查询。
  - Agent 负责单设备工作流状态机、本地节点流转、本地循环计数和本地 checkpoint 恢复。
- 详细方案见：
  - `docs/server/工作流状态机下沉到Agent架构方案.md`

## 设备占用与执行域边界
- 正式模型下，设备占用只由计划任务运行域裁决。
- 占用条件：`plan_device_runs.status` 为 `pending` 或 `running`。
- 工作流型与脚本型计划任务共用同一条占用链路，不再存在独立 `workflow_runs` 占用域。
- 设备占用详情查询接口返回计划任务实例占用信息，前端按 `plan_run_id` / `plan_device_run_id` 展示来源。

## 统一设备占用判定方案
本节已落地为单执行域模型：

1. `inspectDeviceBusy` 统一只查询 `plan_device_runs.status in (pending, running)`。
2. 工作流型与脚本型计划任务占用裁决完全一致。
3. 不再补查 `workflow_runs`，因为该运行表已经移除。
4. 设备占用冲突返回 `plan_device_busy`，并携带对应的 `plan_run_id` / `plan_device_run_id` 明细。

## 运行态接口约定

- 计划任务定义列表：`GET /api/v1/plans`
- 计划任务定义创建：`POST /api/v1/plans`
- 计划任务实例列表：`GET /api/v1/plans?view=runs`
- 启动计划任务：`POST /api/v1/plans/{plan_def_id}/start`
- 查询计划任务实例：`GET /api/v1/plans/{plan_def_id}/runs/{plan_run_id}`
- 查询计划任务事件：`GET /api/v1/plans/{plan_def_id}/runs/{plan_run_id}/events`
- 追加设备到计划任务实例：`POST /api/v1/plans/{plan_def_id}/runs/{plan_run_id}/devices`
- 从计划任务实例移除设备：`DELETE /api/v1/plans/{plan_def_id}/runs/{plan_run_id}/devices/{device_id}`
- 停止整个计划任务实例：`POST /api/v1/plans/{plan_def_id}/runs/{plan_run_id}/stop`
- 查询设备占用详情：`GET /api/v1/devices/{device_id}/occupancy`

## 工作流循环主线范围
- 支持“顺序节点 + 单循环节点 + 停止节点”的基础编排闭环。
- 对应的手工验收步骤见：
  - `docs/server/工作流循环手工验收.md`
- 前端编排页支持：
  - 多个顺序段
  - 多个循环段
  - 顺序段与循环段的串行组合
- 例如下面这类结构属于支持范围：
  - `A -> B -> CDE循环(3次) -> FG循环(5次) -> H -> stop`
- 循环段语义：
  - 循环段本身是一个 `loop` 节点
  - 循环段内部可以挂多个脚本步骤
  - `max_iterations` 表示整段循环体最多执行多少轮，而不是单个步骤各自的次数
- 前端创建页支持：
  - 顺序段增删
  - 循环段增删
  - 段顺序上下移动
  - 顺序段内多步骤编辑
  - 循环段内多步骤编辑
  - 循环段名称
  - 循环段最大次数
- 不支持：
  - 多个循环节点嵌套
  - 图形化拖拽编排
  - 条件分支
  - 并行分支
  - 基于表达式的动态跳转

## 工作流定义与计划任务的职责边界
- 工作流定义只负责描述“执行结构”，例如顺序关系、循环关系、停止节点。
- 计划任务负责描述“何时运行”和“运行到什么时候”，例如：
  - 每日开始时间
  - 每日截止时间
  - 次日是否继续拉起
  - 设备集合
- 因此 `daily_start_time`、`daily_deadline_time`、`restart_next_day` 这类调度字段不应继续作为工作流定义前端的主输入项。
- 工作流定义级截止时间兼容逻辑不作为正式调度主线；正式主线以计划任务调度为准。

## 任务中心收口策略
- 计划任务已经成为正式运行入口，产品层面不再继续扩展“任务中心”页面。
- 底层 `task` 执行域保留，原因是脚本型计划任务与 workflow session 中的脚本节点执行复用 `assign_task / task_ack / task_progress / task_result` 协议。
- 收口状态如下：
  - 前端“任务中心”入口已下线。
  - 公开 `/api/v1/tasks` 与手工任务终止相关 HTTP 入口已移除。
  - `task_source_type='manual'` 不再作为正式语义存在；`task_source_type` 仅保留内部 `plan_script` 元数据用途。
- 结论是：工作流正式执行统一走 `workflow_session_*`，正式运行态统一落在 `plan_def -> plan_run -> plan_device_run`；`internal/task` 继续保留为脚本执行底座，但不再对外承担独立运行入口语义。
