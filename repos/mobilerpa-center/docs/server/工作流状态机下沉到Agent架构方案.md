# 工作流状态机下沉到 Agent 架构方案

## 1. 背景

当前系统已经完成两类能力：

1. 计划任务调度层已经采用“轻量扫描协程 + 异步 worker + 受控并发 fan-out”的方式运行，适合承载 300 到 500 台设备的一次性批量启动。
2. Agent 已经具备脚本下载、脚本执行、关键进度上报、任务结果回传、断线重连、单实例运行锁等本地执行能力。

但当前工作流运行链路仍然以“中心逐节点推进”为主：

1. 中心创建工作流节点对应的任务。
2. 中心向设备下发 `assign_task`。
3. 设备回传 `task_result`。
4. 中心收到结果后再计算下一个节点。
5. 中心继续向同一设备下发下一个节点任务。

这种模式在少量设备下可以工作，但在 500 台设备接入场景下存在明显放大效应：

1. 每一个工作流节点切换都要经过中心一次同步决策和一次同步下发。
2. 工作流节点越多，中心的 WebSocket 写入、数据库更新、事件落库和占用检查压力越高。
3. 中心在处理 `task_result` 的同一条链路中继续同步下发下一节点，容易引入连接态与时序抖动问题。
4. 中心服务被迫承担“调度器 + 全局状态机 + 实时下发器”三重职责，不利于后续扩展到 500 台以上设备。

因此，后续工作流主线应改为：

- 中心负责“调度、会话下发、占用管理、观测与存储”
- Agent 负责“单设备工作流状态机、本地节点流转、本地循环计数和本地恢复”

## 2. 设计目标

本方案的目标如下：

1. 保持当前 500 台计划任务启动 worker/pool 方案不变，不重复改造已完成的批量启动基础设施。
2. 把工作流逐节点推进从中心迁移到 Agent，降低中心在工作流执行过程中的网络吞吐和同步决策压力。
3. 让中心服务从“逐节点驱动器”转变为“工作流会话调度器 + 状态投影存储层”。
4. 保持计划任务仍是正式运行入口，工作流定义仍只负责描述执行结构。
5. 保持设备占用规则不变，即同一设备同一时刻只能属于一个执行域。
6. 为后续 500 台设备长期在线、长工作流、多轮循环运行提供更稳的基础。

## 3. 保持不变的前提

下沉方案不改变以下基础约束：

1. 计划任务仍然是正式运行态主对象。
2. 工作流定义只负责执行结构，不负责每日开始时间、截止时间等调度字段。
3. 计划任务的秒级轻量扫描、异步 worker、受控并发 fan-out 已经是正式基础设施，继续保留。
4. 中心与设备之间仍然通过 WebSocket 建立长连接。
5. 脚本版本管理仍由中心维护，Agent 继续按脚本名与版本号从中心拉取脚本。
6. 设备占用、计划任务占用、手工任务占用规则保持现状。

## 4. 当前问题拆解

当前工作流逐节点推进模式的主要问题有：

### 4.1 中心网络吞吐放大

以 500 台设备、每台每天运行多个工作流节点为例，当前链路每个节点至少会带来：

1. 一次 `assign_task`
2. 一次 `task_ack`
3. 若干 `task_progress` 或 `workflow_step_progress`
4. 一次 `task_result`
5. 一次工作流节点推进
6. 下一节点的新任务创建与新任务下发

这样会把“节点数”直接线性映射成中心消息量与数据库写入量。

### 4.2 中心成为单设备状态机真源

当前每一台设备的工作流执行顺序、循环次数、下一节点选择都在中心决定。这会导致：

1. 中心必须保存并实时更新大量单设备局部状态。
2. 中心必须在每次节点完成时立即参与决策。
3. 任何网络抖动或消息处理抖动都会影响下一节点下发。

### 4.3 同步推进链路过长

当前典型同步链路是：

1. Agent 发送 `task_result`
2. 中心更新任务状态
3. 中心追加任务事件
4. 中心读取工作流运行记录
5. 中心写入工作流步骤成功事件
6. 中心计算下一节点
7. 中心创建下一节点任务
8. 中心向设备写出下一条 `assign_task`

这条链路过长，不适合在 500 台设备下高频重复。

## 5. 目标架构

目标架构采用“中心调度工作流会话，Agent 本地驱动状态机”的模式。

### 5.1 中心职责

中心只负责以下职责：

1. 维护工作流定义与版本。
2. 维护脚本版本、脚本清单和脚本下载接口。
3. 通过计划任务决定何时启动某个工作流。
4. 为每台设备创建一个工作流会话。
5. 向设备下发“启动工作流会话”的单条命令。
6. 接收 Agent 回传的关键事件与最终结果。
7. 保存工作流会话投影状态、设备占用状态和计划任务运行状态。
8. 响应前端的查看、停止、追加设备、移除设备、删除实例等操作。

中心不再负责：

1. 每个节点完成后的下一节点实时决策。
2. 循环节点的本轮计数与本地跳转。
3. 每次节点切换时重新为设备逐条创建并下发单任务。

### 5.2 Agent 职责

Agent 负责以下职责：

1. 接收中心下发的工作流会话。
2. 拉取或校验所需脚本版本。
3. 在本地执行顺序节点、循环节点、停止节点。
4. 本地维护当前节点、循环计数、会话状态。
5. 在节点开始、节点成功、节点失败、循环完成、会话结束时回传关键事件。
6. 在中心短时不可用或自身重启后，基于本地 checkpoint 恢复会话执行。

## 6. 工作流会话模型

后续正式运行对象建议从“中心逐节点任务链”转为“工作流会话”。

### 6.1 中心侧对象

中心侧引入以下概念：

1. `workflow_session`
   - 表示某台设备上的一次完整工作流执行会话
   - 由计划任务实例或兼容入口启动
2. `workflow_session_event`
   - 表示 Agent 回传的关键状态事件
3. `workflow_session_snapshot`
   - 表示中心视角下的当前投影状态

### 6.2 Agent 侧对象

Agent 本地维护：

1. `session_id`
2. `workflow_def_id`
3. `workflow_version`
4. `current_node_id`
5. `loop_counters`
6. `status`
7. `last_event_seq`
8. `updated_at`

## 7. 通信协议调整

当前工作流执行主链路建议从 `assign_task` 切换为新的工作流会话协议。

### 7.1 新增消息类型

建议新增：

1. `start_workflow_session`
2. `workflow_session_ack`
3. `workflow_session_event`
4. `workflow_session_result`
5. `stop_workflow_session`

### 7.2 `start_workflow_session` 载荷建议

建议包含：

1. `workflow_session_id`
2. `plan_run_id`
3. `plan_device_run_id`
4. `workflow_def_id`
5. `workflow_version`
6. `device_id`
7. `entry_node_id`
8. `definition_snapshot`
9. `script_manifest`
10. `runtime_policy`

其中：

1. `definition_snapshot` 用于冻结本次执行结构，避免运行中再受中心定义修改影响。
2. `script_manifest` 用于告诉 Agent 本会话需要哪些脚本和版本。
3. `runtime_policy` 可保留未来扩展，例如失败重试、最大执行时长、事件上报级别等。

### 7.3 `workflow_session_event` 上报粒度

生产默认仅要求上报关键事件：

1. `workflow_session_started`
2. `workflow_node_started`
3. `workflow_node_succeeded`
4. `workflow_node_failed`
5. `workflow_loop_completed`
6. `workflow_session_completed`
7. `workflow_session_failed`
8. `workflow_session_stopped`

细粒度日志或百分比进度不作为默认强制要求，避免 500 台设备持续冲击中心。

### 7.4 心跳摘要

建议在已有心跳基础上带上轻量摘要：

1. `workflow_session_id`
2. `session_status`
3. `current_node_id`
4. `last_event_seq`
5. `local_updated_at`

这样即使单条事件丢失，中心也可以通过心跳摘要做投影纠偏。

## 8. Agent 本地状态机范围

### 8.1 第一阶段支持

本方案第一阶段建议 Agent 本地只支持：

1. `script`
2. `loop`
3. `stop`
4. `next`

这与当前中心已支持的工作流主线范围保持一致。

### 8.2 本地执行步骤

单台设备上的一次会话执行流程为：

1. Agent 收到 `start_workflow_session`
2. Agent 回 `workflow_session_ack`
3. Agent 加载 `definition_snapshot`
4. Agent 校验本地脚本版本，不足则向中心下载
5. Agent 从 `entry_node_id` 开始执行
6. 当前节点成功后，Agent 本地计算下一个节点
7. 遇到循环节点时，Agent 本地维护计数器
8. 进入 `stop` 节点或无后续节点时，Agent 本地结束并回传最终结果

### 8.3 本地 checkpoint

Agent 需要把以下内容持久化到本地运行目录：

1. 当前会话 ID
2. 当前节点 ID
3. 循环计数器
4. 当前状态
5. 最后已发送事件序号
6. 最后更新时间

这样可以支持：

1. Agent 崩溃恢复
2. 中心断线后重连
3. 长工作流恢复

## 9. 中心侧状态投影模型

下沉后，中心不再是工作流状态机真源，而是工作流投影状态存储层。

### 9.1 中心保存什么

中心主要保存：

1. 当前会话是否存在
2. 当前节点是什么
3. 当前设备运行状态
4. 最后一条关键事件
5. 最终执行结果

### 9.2 中心不再负责什么

中心不再根据单节点执行结果主动决定：

1. 下一节点是哪一个
2. 是否继续循环
3. 循环次数是否达到退出条件

这些都由 Agent 本地根据 `definition_snapshot` 决定。

## 10. 与计划任务的关系

计划任务仍然是正式运行入口。

### 10.1 启动

当计划任务命中时：

1. 中心创建 `plan_run`
2. 中心为每台设备创建 `plan_device_run`
3. 中心为每台设备创建一个 `workflow_session`
4. 中心通过 worker pool 并发下发 `start_workflow_session`

### 10.2 追加设备

运行中的计划任务实例追加设备时：

1. 中心先做占用检查
2. 中心为新增设备创建新的 `plan_device_run`
3. 中心为新增设备创建新的 `workflow_session`
4. 中心向该设备发送 `start_workflow_session`

### 10.3 移除设备或停止设备

计划任务实例移除设备时：

1. 若设备尚未开始执行或已结束，则中心直接移除投影状态
2. 若设备仍在执行，则中心向该设备发送 `stop_workflow_session`
3. Agent 本地优雅停止并回传最终状态

## 11. 对 500 台设备承载的意义

该方案对 500 台设备场景的意义在于：

1. 中心只在会话开始时下发一次主命令，而不是每个节点都下发一次任务。
2. 节点流转不再把中心的网络写入量与“节点数”线性绑定。
3. 中心数据库写入集中在关键事件，而不是每个节点切换都伴随同步决策和同步下发。
4. Agent 离设备更近，更适合承载本地顺序、循环和局部恢复状态。
5. 中心 worker pool 可以继续专注于“批量启动会话”，而不是“高频逐节点推进”。

## 12. 与现有链路的兼容策略

### 12.1 保留手工任务域

手工任务继续走现有链路：

1. 中心创建任务
2. 中心下发 `assign_task`
3. Agent 回 `task_ack` / `task_result`

手工任务不需要立即迁移到工作流会话模型。

### 12.2 工作流兼容入口

当前 `POST /api/v1/workflows/{workflow_def_id}/start` 可以临时保留，但内部建议逐步改为：

1. 创建兼容工作流实例
2. 为设备生成工作流会话
3. 下发 `start_workflow_session`

### 12.3 旧工作流事件域

旧的 `workflow_step_started`、`workflow_step_succeeded`、`workflow_loop_completed` 等事件类型可以保留，但其语义需要从：

- 中心推进产生的事件

调整为：

- Agent 本地状态机执行后回传的事件

## 13. 迁移步骤

建议按以下顺序迁移：

### 13.1 第一阶段：协议与本地执行器

1. 中心新增 `start_workflow_session` / `stop_workflow_session` 协议。
2. Agent 新增工作流会话执行器。
3. Agent 支持 `definition_snapshot` 本地顺序执行。

### 13.2 第二阶段：循环节点下沉

1. Agent 本地支持循环计数器。
2. Agent 本地支持 `loop_body` / `loop_exit`。
3. 中心不再维护循环上下文真源。

### 13.3 第三阶段：计划任务接入

1. 工作流型计划任务从“启动工作流实例 + 中心逐节点推进”改为“启动工作流会话”。
2. 计划任务实例页面改为基于工作流会话展示运行状态。

### 13.4 第四阶段：兼容链路收口

1. 旧的“中心逐节点工作流推进”链路标记为兼容链路。
2. 新工作流默认全部使用 Agent 本地状态机。
3. 旧链路在完成迁移验证后再考虑删除。

## 14. 风险与注意事项

### 14.1 事件一致性

中心不再是状态机真源后，需要接受“中心是投影、Agent 是执行真源”这一变化。因此需要：

1. 会话事件有序号
2. 心跳带摘要
3. 中心允许基于最新摘要纠偏

### 14.2 Agent 崩溃恢复

如果 Agent 本地不做 checkpoint，则状态机下沉后恢复能力会变差。因此本地 checkpoint 是必须项，不是可选优化。

### 14.3 定义快照冻结

工作流启动后必须冻结 `definition_snapshot`，不能让运行中的设备继续读取中心实时定义，否则会导致不同设备在同一会话中执行不同结构。

### 14.4 事件风暴

即使状态机下沉，若 Agent 默认持续发送细粒度进度，也仍可能造成中心事件风暴。因此默认必须坚持“关键事件优先、细粒度日志按需开启”的原则。

## 15. 本方案与当前批量启动基础设施的关系

本方案不会推翻当前已经完成的 500 台计划任务 worker/pool 设计。

当前基础设施继续负责：

1. 秒级扫描计划任务
2. 异步启动队列
3. 启动 worker
4. 受控并发下发

变化只在于：

- 现在 worker 下发的是逐节点任务入口
- 未来 worker 下发的是工作流会话入口

也就是说，当前“轻量中心”的基础设施已经具备，后续只需要把“工作流逐节点状态机”从中心迁移到 Agent。
