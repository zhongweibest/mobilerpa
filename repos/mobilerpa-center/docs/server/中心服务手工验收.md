# MobileRPA Center 手工启动与接口验收

## 1. 目的
本文档用于在不依赖 `go test` 的情况下，手工启动 `mobilerpa-center` 服务，并通过接口调用完成最小功能验收。

如果你希望直接用脚本自动完成这轮最小验收，也可以使用：

```powershell
cd D:\dev\code\mobilerpa\repos\mobilerpa-center
powershell -ExecutionPolicy Bypass -File .\tools\api-smoke-test.ps1
```

当前适用的验收范围：

1. 健康检查
2. 设备注册
3. 设备列表查询
4. 单设备详情查询
5. 离线设备删除
6. WebSocket `hello/heartbeat`
7. 心跳超时自动离线

当前正式运行入口约定如下：

1. 正式运行主线统一收口到“计划任务 + 工作流”。
2. 工作流正式执行默认走 `workflow_session_*` 协议族，由中心下发工作流会话、由 Agent 本地推进状态机。
3. 工作流实例层已经移除，正式运行态统一为 `plan_def -> plan_run -> plan_device_run`；公开 `/api/v1/tasks` 不再作为手工验收入口保留。
4. 设备绑定正式采用 `location_nodes` 单表树模型；设备表中的 `slot_zone / slot_row / slot_position` 是当前绑定结果投影，`physical_slot` 仅为展示值。

## 2. 前置条件
1. 已安装 Go
2. 当前目录为 `repos/mobilerpa-center/server`
3. 已执行过依赖拉取

## 3. 启动方式

### 3.1 使用环境变量启动

这种方式最明确，也最适合临时覆盖本机配置。在 PowerShell 中执行：

```powershell
$env:CENTER_HTTP_ADDR="127.0.0.1:18080"
$env:CENTER_DB_PATH="D:\dev\code\mobilerpa\.tmp\mobilerpa-center-manual.db"
$env:CENTER_CORS_ALLOWED_ORIGINS="http://localhost:5173,http://127.0.0.1:5173"
$env:CENTER_HEARTBEAT_INTERVAL="30s"
$env:CENTER_DEVICE_OFFLINE_TIMEOUT="90s"
$env:CENTER_DEVICE_OFFLINE_SCAN_INTERVAL="15s"
go run .\cmd\center
```

如果你不想用固定端口，也可以把 `18080` 改成其他端口。

### 3.2 使用 `.env` 默认配置启动

如果希望每次启动时默认使用固定配置，可以在 `repos/mobilerpa-center` 目录复制模板：

```powershell
Copy-Item .\.env.example .\.env
```

然后按本机实际情况修改 `.env`：

```text
CENTER_HTTP_ADDR=127.0.0.1:18080
CENTER_DB_PATH=D:\dev\code\mobilerpa\.tmp\mobilerpa-center-manual.db
CENTER_CORS_ALLOWED_ORIGINS=http://localhost:5173,http://127.0.0.1:5173
CENTER_HEARTBEAT_INTERVAL=30s
CENTER_DEVICE_OFFLINE_TIMEOUT=90s
CENTER_DEVICE_OFFLINE_SCAN_INTERVAL=15s
```

再进入 `server` 目录启动：

```powershell
cd D:\dev\code\mobilerpa\repos\mobilerpa-center\server
go run .\cmd\center
```

配置优先级如下：

1. PowerShell 中手动设置的环境变量优先级最高。
2. `.env` 文件中的配置作为本地默认值。
3. 如果两者都没有设置，程序使用内置默认值：`CENTER_HTTP_ADDR=:8080`、`CENTER_DB_PATH=./data/mobilerpa.db`、`CENTER_CORS_ALLOWED_ORIGINS=http://localhost:5173,http://127.0.0.1:5173`、`CENTER_HEARTBEAT_INTERVAL=30s`、`CENTER_DEVICE_OFFLINE_TIMEOUT=90s`、`CENTER_DEVICE_OFFLINE_SCAN_INTERVAL=15s`。

`.env` 是本机配置文件，已加入 `.gitignore`，不要提交到仓库。

## 4. 验收步骤

### 4.1 健康检查
新开一个 PowerShell 窗口，执行：

```powershell
Invoke-RestMethod -Uri "http://127.0.0.1:18080/healthz"
```

预期结果：

```json
{
  "status": "ok",
  "service": "mobilerpa-center"
}
```

### 4.2 设备注册
执行：

```powershell
$body = @{
  agent_uuid  = "agent-test-001"
  device_name = "Pixel Test"
  brand       = "Google"
  model       = "Pixel 8"
  android_id  = "android-test-001"
  adb_serial  = "adb-test-001"
} | ConvertTo-Json

Invoke-RestMethod -Uri "http://127.0.0.1:18080/api/v1/device/register" `
  -Method Post `
  -ContentType "application/json" `
  -Body $body
```

预期结果：

1. 返回 `status = ok`
2. 返回 `device_id`
3. 第一台设备通常是 `dev_000001`

示例：

```json
{
  "status": "ok",
  "data": {
    "device_id": "dev_000001",
    "bind_status": "pending",
    "status": "registered"
  }
```

## 5. 验收要点
本轮手工验收重点确认：

1. 服务是否能成功启动
2. 服务是否能访问 SQLite
3. 注册接口是否能写入设备
4. 列表接口是否能正确读回设备
5. 单设备查询接口是否能按 `device_id` 正确读回设备
6. 离线设备删除接口是否只删除目标离线设备
7. 在线设备是否会被正确拒绝删除
8. WebSocket `hello` 是否返回 `ack.status = ok`
9. WebSocket `heartbeat` 是否能更新设备在线状态
10. WebSocket 断开后设备是否能回到离线状态
11. WebSocket 未显式断开时，超过离线超时后设备是否会自动变为 `offline`

## 8. Apifox WebSocket 手工验收

### 7.1 连接地址

在 Apifox 中新建 WebSocket 请求，连接地址填写：

```text
ws://127.0.0.1:18080/ws
```

如果你的服务不是 `18080` 端口，请替换为实际端口。

### 7.2 发送 `hello`

发送类型选择文本消息，消息内容必须是完整 JSON，不要带 PowerShell 的 `@{}` 语法。

把下面的 `device_id` 替换为设备注册接口返回的 `data.device_id`：

```json
{
  "type": "hello",
  "request_id": "apifox-hello-001",
  "device_id": "dev_000001",
  "timestamp": 1780935112,
  "payload": {
    "agent_uuid": "agent-test-001"
  }
}
```

预期响应：

```json
{
  "type": "ack",
  "request_id": "apifox-hello-001",
  "device_id": "",
  "timestamp": 1780935112,
  "payload": {
    "message_type": "hello",
    "status": "ok"
  }
}
```

### 7.3 发送 `heartbeat`

继续在同一个 WebSocket 连接中发送：

```json
{
  "type": "heartbeat",
  "request_id": "apifox-heartbeat-001",
  "device_id": "dev_000001",
  "timestamp": 1780935113,
  "payload": {
    "battery": 90,
    "network": "wifi"
  }
}
```

预期响应：

```json
{
  "type": "ack",
  "request_id": "apifox-heartbeat-001",
  "device_id": "",
  "timestamp": 1780935113,
  "payload": {
    "message_type": "heartbeat",
    "status": "ok"
  }
}
```

### 7.4 查询在线状态

发送 `heartbeat` 后，调用设备列表接口：

```text
GET http://127.0.0.1:18080/api/v1/devices
```

预期结果：

1. 对应设备的 `status` 为 `online`
2. 对应设备的 `last_heartbeat_at` 不为空

### 7.5 验证离线状态

在 Apifox 中断开 WebSocket 连接，然后再次调用：

```text
GET http://127.0.0.1:18080/api/v1/devices
```

预期结果：

1. 对应设备的 `status` 变为 `offline`
2. `last_heartbeat_at` 保留最近一次心跳时间

### 7.6 常见错误

如果返回下面内容：

```json
{
  "type": "ack",
  "request_id": "",
  "device_id": "",
  "timestamp": 1780935112,
  "payload": {
    "message_type": "invalid_message",
    "status": "invalid_json"
  }
}
```

说明中心服务没有收到合法 JSON。常见原因：

1. 在 Apifox 中发送了 PowerShell 格式的 `@{}`，而不是 JSON。
2. JSON 字段没有使用双引号。
3. 复制内容前后带了多余字符。
4. 发送的是对象预览或表单内容，而不是 WebSocket 文本消息。

请直接复制本文档中的 JSON 示例再发送。

## 9. 心跳超时自动离线验收

### 8.1 推荐测试参数

为了缩短人工等待时间，建议在本轮验收时临时使用更短参数启动中心服务：

```powershell
$env:CENTER_HTTP_ADDR="127.0.0.1:18080"
$env:CENTER_DB_PATH="D:\dev\code\mobilerpa\.tmp\mobilerpa-center-heartbeat-timeout.db"
$env:CENTER_HEARTBEAT_INTERVAL="5s"
$env:CENTER_DEVICE_OFFLINE_TIMEOUT="15s"
$env:CENTER_DEVICE_OFFLINE_SCAN_INTERVAL="3s"
go run .\cmd\center
```

如果你要验默认策略，也可以使用：

```powershell
$env:CENTER_HEARTBEAT_INTERVAL="30s"
$env:CENTER_DEVICE_OFFLINE_TIMEOUT="90s"
$env:CENTER_DEVICE_OFFLINE_SCAN_INTERVAL="15s"
```

### 8.2 方式一：Apifox 只发送一次 `heartbeat`

这是最容易稳定复现“心跳超时但连接未正常继续发包”的方式。

步骤：

1. 启动中心服务，建议使用上一节中的缩短参数。
2. 先按 `7.2`、`7.3` 的步骤在 Apifox 建立 WebSocket 连接并发送一次 `hello`、一次 `heartbeat`。
3. 保持这个 WebSocket 标签页打开，不要手动点“断开”，也不要继续发送第二次 `heartbeat`。
4. 等待超过 `CENTER_DEVICE_OFFLINE_TIMEOUT`，建议等待 `20` 秒以上。
5. 调用设备列表接口：

```powershell
Invoke-RestMethod -Uri "http://127.0.0.1:18080/api/v1/devices" | ConvertTo-Json -Depth 8
```

预期结果：

1. 对应设备的 `status` 变为 `offline`
2. `last_heartbeat_at` 保留第一次 `heartbeat` 的时间
3. 即使 Apifox 标签页还开着，只要没有继续发送心跳，设备也会被自动判定离线

### 8.3 方式二：真机启动后人为停止心跳

如果你要更接近真实环境，可以用真机制造“连接存在但心跳停止”的场景。

可选做法：

1. 在 AutoJs6 中先让 Agent 正常启动并连上中心。
2. 观察设备状态已经变为 `online`。
3. 然后强制暂停脚本执行、手动停止定时心跳逻辑，或让脚本卡在不会继续发心跳的调试分支。
4. 保持手机网络暂时不主动断开，等待超过 `CENTER_DEVICE_OFFLINE_TIMEOUT`。
5. 再查询设备列表接口确认状态。

预期结果：

1. 对应设备最终自动变为 `offline`
2. 不需要等到服务端先收到明确的 WebSocket 断开事件

说明：

如果当前真机端还不方便精确暂停心跳，优先使用“方式一：Apifox 只发送一次 `heartbeat`”完成中心侧验收即可。这种方式更稳定，也更容易重复验证。

### 8.4 单设备查询确认离线

如果你想只看某一台设备，可以直接查询：

```powershell
Invoke-RestMethod -Uri "http://127.0.0.1:18080/api/v1/devices/dev_000001" | ConvertTo-Json -Depth 8
```

预期字段：

```json
{
  "status": "offline",
  "last_heartbeat_at": "2026-06-10T08:53:52Z"
}
```

其中 `last_heartbeat_at` 以你的真实最后一次心跳时间为准，不要求和示例完全一致。

## 10. 历史兼容任务接口说明

本节原先用于验收公开 `/api/v1/tasks` 手工任务接口。当前版本这组公开接口已从正式路由中移除，不再作为手工验收入口保留。

当前约定如下：
1. 脚本执行底座仍保留 `assign_task` / `task_ack` / `task_progress` / `task_result` 协议与 `internal/task` 服务。
2. 正式人工验收统一走计划任务实例主线，不再通过 HTTP 直接创建、查询或下发手工任务。
3. 如需排查底层任务执行，可查看计划任务事件、设备日志与中心日志，而不是访问公开 `/api/v1/tasks`。

## 11. `C2-2` 任务下发与回执手工验收

### 11.1 验收目标

本节用于确认以下闭环已经打通：

1. 中心可把指定任务下发给当前在线设备。
2. 设备或模拟 Agent 能收到 `assign_task`。
3. 设备或模拟 Agent 能回传 `task_ack`。
4. 中心能把任务状态更新为 `assigned`，并写入 `task_assigned`、`task_ack` 事件。

### 11.2 前置条件

1. 中心服务已经启动。
2. 目标设备已经完成注册，并且当前 WebSocket 在线。
3. 你已经通过计划任务主线触发出一个待执行脚本任务。

### 11.3 通过 Apifox 或等价 WebSocket 工具模拟在线设备

连接地址：

## 16. `C2-6` 工作流每日截止时间与次日自动重启手工验收

### 16.1 验收目标

本节用于验收两项时间规则能力：

1. 工作流命中每日截止时间后，中心自动停止该工作流实例下仍在运行的设备。

## 17. 计划任务主线手工验收

### 17.1 验收目标与主线说明

本节用于验收当前版本的正式运行主线：

1. 工作流编排页只负责维护工作流定义。
2. 正式运行统一通过计划任务定义和计划任务实例完成。
3. 计划任务支持两种目标类型：
   - `script`：直接按脚本名和版本号运行。
   - `workflow`：按工作流定义运行，由工作流节点继续派生具体任务。
4. 计划任务实例是正式运行态主对象。

建议按下面顺序验收：

1. 先验收“脚本型计划任务”。
2. 再验收“工作流型计划任务”。
3. 最后验收“daily 自动调度”。

### 17.2 启动参数确认

启动中心服务时，建议显式带上计划任务扫描周期，便于观察 daily 自动调度：

```powershell
$env:CENTER_HTTP_ADDR="127.0.0.1:18080"

$env:CENTER_WORKFLOW_SCAN_INTERVAL="5s"
$env:CENTER_PLAN_SCAN_INTERVAL="5s"
go run .\cmd\center
```

说明：

1. `CENTER_PLAN_SCAN_INTERVAL` 用于计划任务 daily 自动启动和自动截止停止扫描。
2. 人工验收阶段建议临时设置成 `5s`。

### 17.6 Daily 自动调度验收

#### 17.6.1 创建 daily 脚本型计划任务

请求体示例：

```json
{
  "plan_name": "daily 脚本计划任务验收",
  "description": "用于验收计划任务自动启动和自动停止",
  "target_type": "script",
  "target_script_name": "open_qq",
  "target_script_version": "v0.1.0",
  "schedule_type": "daily",
  "daily_start_time": "23:58:30",
  "daily_deadline_time": "23:59:10",
  "status": "enabled",
  "device_ids": [
    "dev_000001"
  ]
}
```

说明：

1. `daily_start_time` 建议填成未来 30 到 60 秒内的时间
2. `daily_deadline_time` 建议填成开始时间之后 30 到 60 秒内的时间

查询地址：

```text
http://127.0.0.1:18080/api/v1/plans?view=runs
```

验收点：

1. 到达 `daily_start_time` 后，列表中自动新增当天的 `plan_run_id`
2. 新实例的 `run_date` 等于当天日期
3. 如果中心服务是在截止时间之后才启动，则不应为当天补建实例

#### 17.6.3 验证自动截止停止

验收点：

1. 到达 `daily_deadline_time` 后，对应 `plan_run_id.status = stopped`
2. 对应 `device_runs[*].status` 也应收敛到 `stopped`

### 17.7 当前主线结论

本阶段后续默认按下面规则理解：

1. 工作流定义页只负责定义维护，不再承担直接运行入口职责。
2. 正式运行统一使用计划任务定义和计划任务实例。
3. 脚本型计划任务和工作流型计划任务都应以本节验收结果为准。
