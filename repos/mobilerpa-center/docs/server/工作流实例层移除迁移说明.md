# 工作流实例层移除迁移说明

本文配合 `中心服务架构.md` 的 `工作流实例层移除（已确认方案）` 与 `docs/整体规划方案.md` 的 `14.15` 节，记录本次 schema 与代码结构变更。当前为开发阶段、无历史包袱，**不做数据迁移**，表结构与代码直接落地，历史运行数据随删表一并清除。

## 1. 表结构变更

### 1.1 保留（不动）

- `workflow_defs`：工作流定义。
- `workflow_nodes`：工作流节点（script/loop/stop）。
- `workflow_edges`：工作流边（next/loop_body/loop_exit）。

这三张表继续作为“能力模板”的载体，由“工作流编排”页维护。

### 1.2 删除

- `workflow_instances`
- `workflow_runs`
- `workflow_contexts`
- `workflow_events`

工作流运行态不再有独立实例/运行表，执行投影落到 `plan_device_runs`，事件落到 `plan_events`。

### 1.3 改造

#### `plan_runs`

- `target_ref_id` 语义调整：工作流型存放 `workflow_def_id`（字符串形态），不再指向 `workflow_instance_id`；脚本型仍为 `script@version`。字段类型保持 `TEXT`。

#### `plan_device_runs`

新增工作流投影列（脚本型运行留空）：

| 列 | 类型 | 说明 |
|---|---|---|
| `current_node_id` | `TEXT NOT NULL DEFAULT ''` | 当前工作流节点 |
| `last_event_type` | `TEXT NOT NULL DEFAULT ''` | 最近一次工作流事件类型 |
| `last_event_seq` | `INTEGER NOT NULL DEFAULT 0` | 最近事件序号 |
| `last_event_at` | `TEXT NOT NULL DEFAULT ''` | 最近事件时间 |

#### `tasks`

删除工作流关联列：

- `workflow_instance_id`
- `workflow_run_id`
- `workflow_node_id`

保留 `plan_run_id`、`plan_device_run_id` 用于脚本型计划任务执行底座；`task_source_type` 仅保留 `plan_script`（`manual` 与 `workflow` 一并移除）。

## 2. 代码结构变更

### 2.1 `internal/workflow`

收敛为“工作流定义 + 定义快照构建”：

- 保留：定义、节点、边的 CRUD；`buildSessionPayload` 等价的“由 `workflow_def_id` 构造 `definition_snapshot` + `script_manifest`”的纯函数。
- 删除：`workflow_instances`/`workflow_runs` 生命周期（`Start`、`AddDevices`、`StopDefinition`、`StopRunByDevice`、`createInstance`、`createRun`、`addRuns`、`startRunFromNode`、`advanceRunAfterNode`、`advanceRunAfterLoopNode`、`ensureNoOtherActiveWorkflow`、`inspectDeviceAvailability` 等）、`workflow_events`/`workflow_contexts` 读写。

### 2.2 `internal/plan`

吸收工作流运行态：

- 工作流型计划任务启动时，在 `plan_run` + `plan_device_runs` 创建完成后，直接按设备派发 `start_workflow_session`；派发失败时实例归属已绑定，可重试或停止，不再产生 `plan_run_id=0` 孤儿。
- 接收 `workflow_session_event`/`workflow_session_result`，更新对应 `plan_device_runs` 投影列并追加 `plan_events`。
- 停止、追加、移除设备复用既有 `plan_run`/`plan_device_runs` 控制。

### 2.3 `internal/dispatch`

- `StartWorkflowSession`/`StopWorkflowSession` 载荷定位键改为 `plan_run_id` + `plan_device_run_id`。
- `tasks` 表的 `workflow_*` 列绑定逻辑随列删除一并移除。

### 2.4 `internal/ws`

- 工作流会话相关消息（`workflow_session_ack`/`workflow_session_event`/`workflow_session_result`）按 `plan_run_id` + `plan_device_run_id` 路由到 `plan` 服务。

### 2.5 `internal/api`

- 删除工作流实例/运行相关路由：`POST /api/v1/workflows/{id}/start`、`GET /api/v1/workflows?view=instances`、`/instances/...`（追加设备、停止、删除实例、停止单设备）、`GET /api/v1/workflows/{id}/events` 等。
- 工作流定义路由（`GET/POST /api/v1/workflows`、`GET/DELETE /api/v1/workflows/{id}`）保留。

## 3. 协议与 Agent

### 3.1 `pkg/protocol`

`StartWorkflowSessionPayload` / `StopWorkflowSessionPayload` / `WorkflowSessionEventPayload` / `WorkflowSessionResultPayload` 的定位字段调整为：

- 移除：`workflow_instance_id`、`workflow_run_id`（实例层概念）。
- 保留/新增：`plan_run_id`、`plan_device_run_id`、`workflow_def_id`、`device_id`。
- `definition_snapshot`、`script_manifest`、`runtime_policy` 结构不变。

### 3.2 Agent `workflow_session_runner`

- checkpoint 与事件回传按 `plan_run_id` + `plan_device_run_id` 落盘与上报。
- 不再保留 `workflow_instance_id`/`workflow_run_id` 的本地状态。

## 4. 前端

- 删除“工作流实例”页面与菜单。
- 计划任务实例页扩展：展示每台设备的当前工作流节点、最近事件，并在实例事件流中合并工作流节点事件。
- “工作流编排”页继续用于定义维护。

## 5. 手工任务移除（与本次清理一并执行）

只保留脚本型计划任务的执行底座，移除手工任务能力：

1. 删除 HTTP 入口：`POST /api/v1/tasks`（手工创建）、`PATCH /api/v1/tasks`（手工下发）、`POST /api/v1/tasks/{id}/terminate`、`POST /api/v1/devices/{id}/manual-tasks/{task_id}`。`GET /api/v1/tasks`、`GET /api/v1/tasks/{id}/events`、`DELETE /api/v1/tasks/{id}` 若无内部消费也一并下线。
2. 删除 `task.StopManualTask` 及 `internal/task` 中仅服务于手工任务的方法。
3. 删除占用检查中的手工任务分支：`plan.inspectDeviceBusy` 的脚本型手工任务检查、`workflow.inspectDeviceAvailability` 的手工任务检查。合并后占用判定为 `plan_device_runs` 单域。
4. `task_source_type` 仅保留 `plan_script`；`tasks`/`task_events` 表、`assign_task`/`task_ack`/`task_progress`/`task_result` 协议、`dispatch.AssignTask` 保留。
5. 前端确认无任务中心 API 引用（当前 `web/src/api` 已无 `tasks`）。

## 6. 验收要点

迁移落地后人工验收至少覆盖：

1. 创建 `target_type=workflow` 计划任务并启动，设备收到 `start_workflow_session` 并按节点推进。
2. 计划任务实例页可见每台设备当前节点与节点事件。
3. 运行中追加/移除设备、停止实例均生效，且不再出现 `plan_run_id=0` 的孤立记录。
4. 脚本型与工作流型计划任务并存于同一设备时，占用裁决一致（统一为 `plan_device_runs + 手工任务`）。
5. 旧“工作流实例”相关接口与页面已不存在或返回明确下线提示。
