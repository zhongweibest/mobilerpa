# Agent 职责与工作流状态机边界说明

## 1. 文档目的

本文件用于明确：

1. 中心服务与 Agent 在正式运行态中的职责边界。
2. 工作流状态机下沉到 Agent 后，哪些状态由 Agent 负责，哪些状态由中心负责。
3. 后续开发时，如何避免重新回到“中心逐节点推进工作流”的实现路径。

## 2. Agent 的正式职责

Agent 在系统中的正式职责应包括：

1. 设备注册与稳定身份上报。
2. WebSocket 长连接维持、心跳、断线重连。
3. 接收中心下发的执行命令。
4. 拉取脚本版本与运行依赖。
5. 在手机本地执行脚本。
6. 上报关键执行进度与执行结果。
7. 在工作流主线中承载单设备工作流状态机。

Agent 不再只是“脚本执行器”，而是“单设备执行域控制器”。

## 3. 中心服务的正式职责

中心服务在工作流状态机下沉后，正式职责应包括：

1. 维护设备、脚本、工作流、计划任务等静态定义。
2. 通过计划任务调度决定何时启动某个工作流或脚本执行。
3. 管理设备占用与跨执行域冲突。
4. 为设备创建运行会话。
5. 向设备下发启动会话、停止会话等控制指令。
6. 接收 Agent 回传的关键事件、结果与摘要心跳。
7. 把 Agent 回传的执行状态存储为中心侧投影状态。
8. 提供页面查询、实例控制、设备追加、设备移除等后台操作能力。

中心服务不再负责逐节点同步推进。

## 4. 状态机真源约定

### 4.1 真源定义

“状态机真源”指的是：

1. 当前节点是谁
2. 下一节点是什么
3. 循环计数到了多少
4. 当前设备上的工作流是继续、失败、停止还是完成

这些判断的最终真源应在 Agent 本地，而不是中心数据库。

### 4.2 中心侧状态定义

中心侧保存的是“状态投影”，不是“状态机真源”。

中心只应保存：

1. 当前设备最后一次上报的节点
2. 当前设备最后一次上报的状态
3. 当前会话最后一次成功持久化的关键事件
4. 计划任务实例与工作流会话之间的关联关系

## 5. 工作流状态机下沉后的职责划分

### 5.1 Agent 负责

Agent 负责：

1. 解析 `definition_snapshot`
2. 在本地决定 `next` 边跳转
3. 在本地维护循环节点计数器
4. 在本地控制顺序节点、循环节点、停止节点推进
5. 在本地决定某设备的工作流会话何时成功结束
6. 在本地决定某设备的工作流会话何时失败结束
7. 在本地持久化 checkpoint 以支持恢复

### 5.2 中心负责

中心负责：

1. 生成 `workflow_session`
2. 把 `definition_snapshot` 与脚本清单一次性下发到设备
3. 接收 `workflow_session_event`
4. 接收 `workflow_session_result`
5. 更新页面可见的实例状态、设备状态、计划任务状态
6. 在用户请求停止时向 Agent 发出停止命令

## 6. 当前链路与未来链路的区别

### 6.1 当前链路

当前工作流执行链路是：

1. 中心根据当前节点创建一个 task
2. 中心向设备下发 `assign_task`
3. Agent 仅执行这个 task
4. Agent 回 `task_result`
5. 中心根据结果决定下一个节点
6. 中心继续下发下一个 task

### 6.2 未来链路

未来工作流执行链路应改为：

1. 中心向设备下发一个 `workflow_session`
2. Agent 本地加载整个工作流定义快照
3. Agent 在本地逐节点推进
4. Agent 在关键状态点回传事件
5. 中心只记录、展示和聚合

## 7. 为什么不应把状态机继续放在中心

如果继续把状态机留在中心，会出现以下问题：

1. 每个节点完成都需要中心同步参与一次决策。
2. 节点越多，中心的消息吞吐与数据库写入压力越大。
3. 中心容易在 `task_result -> 下一节点 assign_task` 这条同步链路里出现时序抖动。
4. 当计划任务同时控制 300 到 500 台设备时，中心会被逐节点放大后的吞吐拖住。

## 8. 与设备占用规则的关系

工作流状态机下沉到 Agent 后，不改变当前设备占用规则：

1. 同一设备同一时刻只能属于一个执行域。
2. 手工任务、计划任务、工作流会话之间继续共用同一套占用判断。
3. 追加设备、移除设备、停止设备仍由中心统一判断是否允许。

也就是说：

- “是否允许开始”由中心决定
- “开始之后怎么一步步跑”由 Agent 决定

## 9. 与计划任务的关系

工作流状态机下沉后，计划任务仍是正式运行入口。

计划任务负责：

1. 何时开始
2. 何时截止
3. 设备集合是什么
4. 是否允许追加或移除设备

Agent 不负责这些调度层问题，只负责被中心选中后的单设备执行。

## 10. 开发时必须遵守的边界

后续开发中，默认必须遵守以下规则：

1. 不再新增“中心根据单个节点成功结果同步创建下一节点 task”的主线实现。
2. 不再把循环计数真源放回中心数据库。
3. 不再把单设备工作流当前节点判断放回中心作为唯一真源。
4. 中心新增的工作流相关字段，应优先理解为“投影字段”而不是“真源字段”。
5. Agent 新增的工作流执行能力，优先围绕本地会话、本地 checkpoint、本地节点流转建设。

## 11. 例外情况

以下情况允许中心继续保留兼容逻辑，但必须明确标注为过渡方案：

1. 历史工作流实例兼容查询
2. 已存在的旧接口兼容返回
3. 手工任务域继续使用单 task 执行模型
4. 状态迁移期的双写或双读适配

这些例外都不应再成为新主线。
